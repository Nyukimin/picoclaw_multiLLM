# 実装仕様: サブエージェントシステム v1.0

**作成日**: 2026-03-06
**ステータス**: 設計完了 → 実装開始
**ベース**: v3.0（Clean Architecture）+ 上流SubagentManager/RunToolLoop移植
**依存**: 実装仕様_v3.md, TOOL_CONTRACT.md

---

## 1. 概要

### 1.1 目的

PicoClawのサブエージェントシステムは、Agent（主にShiro/Worker）がLLMのtool calling機能を使って
自律的にツールを選択・実行する**ReActループ**を提供する。

上流（旧picoclaw）には `SubagentManager` + `RunToolLoop` が存在していたが、
Clean Architecture移植時にスタブ化されていた。本仕様はこれを正式に移植・実装する。

### 1.2 設計原則

- **段階的構築**: 各Phaseが独立してコンパイル・テスト可能
- **既存インターフェース維持**: `LLMProvider` は変更しない（`ToolCallingProvider` で拡張）
- **ハードコード禁止**: MaxIterations, Model等すべてconfig.yamlから取得
- **循環依存回避**: application層はdomain層のみに依存
- **再帰呼び出し防止**: subagentツール自身はツール一覧から除外

### 1.3 用語

| 用語 | 定義 |
|------|------|
| **ReActループ** | LLM推論 → ツール実行 → 結果フィードバック → 再推論の反復パターン |
| **Tool Calling** | LLMが構造化されたツール呼び出しを返す機能（Ollama /api/chat の tools パラメータ） |
| **ToolCallingProvider** | Tool Calling対応のLLMプロバイダーインターフェース |
| **RunToolLoop** | ReActループの実装（application層） |
| **SubagentManager** | サブエージェントタスクの管理・実行（application層） |

## 3. Phase 1: ToolCallingProvider + Ollama /api/chat

### 3.1 背景

現在の `LLMProvider` は `Generate(ctx, req) (resp, error)` のみ（テキスト補完、Ollama /api/generate）。
Tool calling には Ollama `/api/chat` エンドポイント + `tools` パラメータが必要。

### 3.2 domain/llm 型追加

**ファイル**: `internal/domain/llm/provider.go`

既存の `LLMProvider` インターフェースは**変更しない**。新しい型とインターフェースを追加する。

```go
// --- Tool Calling 対応型 ---

// ChatMessage はツール呼び出し対応メッセージ
type ChatMessage struct {
    Role       string     // "system", "user", "assistant", "tool"
    Content    string
    ToolCalls  []ToolCall // role="assistant" 時のツール呼び出し
    ToolCallID string     // role="tool" 時の対応ID
}

// ToolCall はLLMが返すツール呼び出し
type ToolCall struct {
    ID       string
    Function ToolCallFunction
}

// ToolCallFunction はツール呼び出しの関数情報
type ToolCallFunction struct {
    Name      string
    Arguments map[string]any
}

// ToolDefinition はLLMに渡すツール定義
type ToolDefinition struct {
    Type     string          // "function"
    Function ToolFunctionDef
}

// ToolFunctionDef はツール関数の定義
type ToolFunctionDef struct {
    Name        string
    Description string
    Parameters  map[string]any // JSON Schema
}

// ChatRequest はtool calling対応チャットリクエスト
type ChatRequest struct {
    Model       string
    Messages    []ChatMessage
    Tools       []ToolDefinition
    Temperature float64
}

// ChatResponse はtool calling対応チャットレスポンス
type ChatResponse struct {
    Message      ChatMessage
    Done         bool
    FinishReason string // "stop" or "tool_calls"
}

// ToolCallingProvider はtool calling対応のLLMプロバイダー
// LLMProvider を拡張し、Chat メソッドを追加する
type ToolCallingProvider interface {
    LLMProvider
    Chat(ctx context.Context, req ChatRequest) (ChatResponse, error)
}
```

### 3.3 Ollama /api/chat 実装

**ファイル**: `internal/infrastructure/llm/ollama/provider.go`

OllamaProvider に `Chat()` メソッドを追加。これにより `ToolCallingProvider` を実装する。

**Ollama /api/chat リクエスト形式**:
```json
{
  "model": "picoclaw-v1",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "web_search",
        "description": "Web検索を実行",
        "parameters": {
          "type": "object",
          "properties": {
            "query": {"type": "string", "description": "検索クエリ"}
          },
          "required": ["query"]
        }
      }
    }
  ],
  "stream": false,
  "keep_alive": -1
}
```

**Ollama /api/chat レスポンス形式**:
```json
{
  "model": "picoclaw-v1",
  "message": {
    "role": "assistant",
    "content": "",
    "tool_calls": [
      {
        "function": {
          "name": "web_search",
          "arguments": {"query": "PicoClaw"}
        }
      }
    ]
  },
  "done": true
}
```

**実装ポイント**:
- `stream: false` で同期呼び出し
- `keep_alive: -1` で常駐維持
- tool_calls がある場合 `FinishReason = "tool_calls"`
- tool_calls がない場合 `FinishReason = "stop"`
- Ollama の tool_call には ID がない場合がある → 自動生成（`fmt.Sprintf("call_%d", i)`）

### 4.3 RunToolLoop

**ファイル**: `internal/application/toolloop/loop.go`（新規）

```go
package toolloop

import (
    "context"
    "fmt"
    "github.com/Nyukimin/picoclaw_multiLLM/internal/domain/llm"
    "github.com/Nyukimin/picoclaw_multiLLM/internal/domain/tool"
)

// Config はツールループの設定
type Config struct {
    MaxIterations int // 最大反復回数（0の場合デフォルト10）
}

// Run はReActループを実行する
//
// フロー:
//   1. provider.Chat(messages + tools) を呼び出す
//   2. レスポンスに tool_calls がある場合:
//      a. 各 tool_call を toolRunner.ExecuteV2() で実行
//      b. 実行結果を role="tool" メッセージとして messages に追加
//      c. 1. に戻る
//   3. tool_calls がない場合（通常テキスト応答）:
//      → レスポンスの Content を返す
//   4. MaxIterations を超えた場合:
//      → 最後のレスポンスの Content を返す（途中打ち切り）
//
// ツール実行エラーの場合:
//   エラーメッセージを role="tool" として LLM にフィードバックする
//   （LLM に再判断させるため、ループは中断しない）
func Run(ctx context.Context, provider llm.ToolCallingProvider,
    toolRunner tool.RunnerV2, toolDefs []llm.ToolDefinition,
    messages []llm.ChatMessage, cfg Config) (string, error)
```

### 5.1 ドメインエンティティ

package agent

// SubagentTask はサブエージェントに渡すタスク
type SubagentTask struct {
    AgentName    string // サブエージェント名（ログ・識別用）

    SystemPrompt string // システムプロンプト（空の場合はデフォルト使用）

type SubagentResult struct {

}

### 5.2 SubagentManager

**ファイル**: `internal/application/subagent/manager.go`（新規）

```go
package subagent

// Manager はサブエージェントタスクの実行を管理する
type Manager struct {
    provider   llm.ToolCallingProvider
    toolRunner tool.RunnerV2
    toolDefs   []llm.ToolDefinition
    loopConfig toolloop.Config
}

// NewManager は新しい Manager を作成する
func NewManager(
    provider llm.ToolCallingProvider,
    toolRunner tool.RunnerV2,
    toolDefs []llm.ToolDefinition,
    loopConfig toolloop.Config,
) *Manager

// RunSync はサブエージェントタスクを同期実行する
//
// 1. SystemPrompt + Instruction から ChatMessage を構築
// 2. toolloop.Run() に委譲
// 3. SubagentResult を返す
func (m *Manager) RunSync(ctx context.Context, task agent.SubagentTask) (agent.SubagentResult, error)
```

### 5.3 SubagentFunc 更新

**ファイル**: `internal/infrastructure/tools/subagent.go`

既存の `SubagentFunc` 型は維持（後方互換）。Manager から SubagentFunc を生成するファクトリを追加。

```go
// NewSubagentFuncFromManager は Manager.RunSync を呼び出す SubagentFunc を返す
// agent引数はログ識別用として SubagentTask.AgentName にマッピングされる
func NewSubagentFuncFromManager(mgr *subagent.Manager) SubagentFunc {
    return func(ctx context.Context, message string) (string, error) {
        result, err := mgr.RunSync(ctx, agent.SubagentTask{
            AgentName:   "worker",  // ツール経由の場合は "worker" 固定
            Instruction: message,
        })
        if err != nil {
            return "", err
        }
        return result.Output, nil
    }
}
```

### 6.3 main.go 配線

**ファイル**: `cmd/picoclaw/main.go`

```go
// サブエージェント配線
var subagentMap map[string]tools.SubagentFunc
if cfg.Subagent.Enabled {
    // 1. OllamaProvider は既に ToolCallingProvider を実装
    // 2. ToolRunner から ToolDefinitions() を取得（subagent自身は除外済み）
    toolDefs := toolRunner.ToolDefinitions()

    // 3. Manager 作成
    mgr := subagentPkg.NewManager(
        ollamaProvider,  // ToolCallingProvider として
        toolRunner,      // RunnerV2 として
        toolDefs,
        toolloop.Config{MaxIterations: cfg.Subagent.MaxIterations},
    )

    // 4. SubagentFunc 生成
    subagentMap = map[string]tools.SubagentFunc{
        "worker": tools.NewSubagentFuncFromManager(mgr),
    }
}

// ToolRunner 作成時に Subagents を渡す
toolRunner := tools.NewToolRunner(tools.ToolRunnerConfig{
    // ... 既存設定 ...
    Subagents: subagentMap,
})
```

**注意**: ToolRunner と SubagentManager が相互参照する問題（ToolRunner.Subagents に Manager を渡すが、Manager は ToolRunner を使う）。
解決策: ToolRunner を2段階で構築する。
1. まず subagent なしで ToolRunner を作成
2. Manager を作成（ToolRunner を注入）
3. ToolRunner に subagent を後から登録（`RegisterSubagent()` メソッド追加）

## 8. エラーハンドリング

### 8.1 ツール実行エラー

ツール実行が失敗した場合、エラーメッセージを `role="tool"` としてLLMにフィードバックする。
LLMが再判断し、別のアプローチを試みるか、エラーを報告する。

```go
result, err := toolRunner.ExecuteV2(ctx, toolCall.Function.Name, toolCall.Function.Arguments)
if err != nil {
    // エラーをLLMにフィードバック（ループは中断しない）
    messages = append(messages, llm.ChatMessage{
        Role:       "tool",
        Content:    fmt.Sprintf("Error: %v", err),
        ToolCallID: toolCall.ID,
    })
    continue
}
```

### 8.2 MaxIterations超過

上限に達した場合、最後のLLMレスポンスのContentを返す。エラーにはしない。
（無限ループ防止が目的であり、途中結果は有用な場合がある）

### 8.3 コンテキストキャンセル

`ctx.Done()` を各ループ先頭でチェック。キャンセル時は `context.Canceled` エラーを返す。

### 8.4 未知のツール呼び出し

LLMが未登録のツール名を返した場合:
- `"unknown tool: {name}"` をフィードバック
- ループは継続（LLMに修正の機会を与える）

## 9. セキュリティ考慮

- **shellコマンド**: 既存の `AllowedShellCommands` + `withTimeout` で制約
- **file_write**: 既存の `ProtectedPatterns` で機密ファイル保護
- **再帰呼び出し**: subagent ツール自身を ToolDefinitions から除外
- **ループ上限**: config.yaml の `max_iterations` で制約（デフォルト10）
- **タイムアウト**: コンテキストベースのタイムアウト伝播
