# 実装仕様: ルーティングシステム v6.0

**作成日**: 2026-03-06
**ステータス**: 設計（Phase 1: 現行仕様の明文化、Phase 2: Coderフォールバック拡張）

---

## 1. 概要

RenCrowのルーティングシステムは、ユーザーメッセージを適切なエージェントに振り分ける。
本仕様は現行実装の明文化と、Coderフォールバックチェーンの新規設計を含む。

### 1.1 設計原則

- **レイテンシ優先**: LLM分類器よりルール辞書を優先（即時判定）
- **安全側フォールバック**: 判定不能時はCHATへ（破壊的操作を回避）
- **コスト最適化**: 安価なCoder1を優先利用、高価なCoderは必要時のみ

## 2. ルーティングカテゴリ

| Route | 用途 | 処理先 |
|-------|------|--------|
| `CHAT` | 会話・意思決定 | Mio (Ollama) |
| `PLAN` | 計画策定 | Mio (※将来専用エージェント化) |
| `ANALYZE` | 分析 | Mio (※将来専用エージェント化) |
| `OPS` | 運用操作（実行・デプロイ等） | Shiro (Worker) |
| `RESEARCH` | 深い調査タスク | Mio (※将来専用エージェント化) |
| `CODE` | コーディング（汎用） | Coderフォールバックチェーン |
| `CODE1` | 明示指定: Coder1 | Coder1 (DeepSeek) |
| `CODE2` | 明示指定: Coder2 | Coder2 (OpenAI) |
| `CODE3` | 明示指定: Coder3 / Chrome操作 | Coder3 (Claude) |

定義: `internal/domain/routing/route.go`

## 3. ルーティング決定フロー（4段階優先順位）

```
ユーザーメッセージ
  │
  ├─[優先度1] チャットコマンド (/status, /stop, /compact, /context, /new)
  │  → HandleChatCommand で即座に処理、ルーティング不要
  │
  ├─[優先度2] 明示ルーティングコマンド
  │  → /code, /code1, /code2, /code3, /plan, /chat, /ops, /analyze, /research
  │  → confidence: 1.0
  │
  ├─[優先度3] ルール辞書（キーワードマッチ）
  │  → confidence: 0.85
  │
  ├─[優先度4] (スキップ中) LLM分類器
  │  → レイテンシ優先で現在は使用しない
  │
  └─[フォールバック] CHAT
     → confidence: 0.7
```

実装: `internal/domain/agent/mio.go` → `DecideAction()`

### 3.1 チャットコマンド（ルーティング前に処理）

| コマンド | 機能 |
|----------|------|
| `/status` | セッション状態表示 |
| `/stop` | リクエスト停止 |
| `/compact` | スレッドフラッシュ |
| `/context` | RecallPack内容表示 |
| `/new` | セッションリセット |

チャットコマンドは `HandleChatCommand()` で処理され、`Handled=true` の場合ルーティングには進まない。

実装: `internal/domain/agent/chat_commands.go`

### 3.2 明示ルーティングコマンド

長いコマンドから順にマッチ（`/code3` → `/code2` → `/code1` → `/code`）。

```
/code3 → CODE3    /plan     → PLAN
/code2 → CODE2    /chat     → CHAT
/code1 → CODE1    /ops      → OPS
/code  → CODE     /analyze  → ANALYZE
                   /research → RESEARCH
```

実装: `internal/domain/agent/mio.go` → `parseExplicitCommand()`

### 3.3 ルール辞書

キーワードベースの即時判定。confidence固定 0.85。

| Route | キーワード |
|-------|-----------|
| `CODE` | 実装して, 修正して, リファクタリング, テストを追加, コードを書, バグを直, 関数を作 |
| `PLAN` | 計画, プラン, 設計して, アーキテクチャ, 方針を決 |
| `ANALYZE` | 分析して, 調査して, 解析して, 診断して, レビューして |
| `OPS` | 実行して, 起動して, デプロイ, ビルドして, 停止して, 再起動 |
| `RESEARCH` | リサーチ, 情報を集, ドキュメントを探 |

**注意**: 「調べて」「検索して」はChatのWeb検索で即答するためRESEARCHには含めない。

実装: `internal/infrastructure/routing/rule_dictionary.go`

## 4. タスク実行ディスパッチ（現行）

ルーティング決定後、`MessageOrchestrator.executeTask()` がルートに応じてエージェントに委譲する。

```
CHAT     → mio.Chat()
PLAN     → mio.Chat()        ※将来専用化
ANALYZE  → mio.Chat()        ※将来専用化
RESEARCH → mio.Chat()        ※将来専用化
OPS      → shiro.Execute()
CODE     → coder1.Generate()
CODE1    → coder1.Generate()
CODE2    → coder2.Generate()
CODE3    → coder3 (Proposal+Worker即時実行 or Generate())
```

実装: `internal/application/orchestrator/message_orchestrator.go` → `executeTask()`

### 4.1 CODE3 の特殊処理

Coder3 が `CoderAgentWithProposal` インターフェースを実装している場合:
1. `GenerateProposal()` で Plan + Patch を生成
2. `WorkerExecutionService.ExecuteProposal()` で即時実行
3. 結果をフォーマットして返却

フォールバック: Proposal非対応の場合は `Generate()` で通常応答。

## 5. Coderフォールバックチェーン（v6 新規設計）

### 5.1 Coder の役割

| Coder | LLM | 主用途 | コスト | 備考 |
|-------|-----|--------|--------|------|
| Coder1 (Aka) | DeepSeek | デフォルト・汎用コーディング | 安価 | 最優先で利用 |
| Coder2 (Ao) | OpenAI | 重めのコーディング | 中程度 | Coder1ビジー時のフォールバック |
| Coder3 (Claude) | Anthropic Claude | Chrome操作・ブラウザ自動化 | 高価 | Chrome専用 + 最終フォールバック |

**重要**: 全Coderは汎用。コーディング・調べ物・分析いずれも可能。主用途は優先利用の判断基準であり、排他的な制約ではない。

### 5.2 フォールバックチェーン

`CODE` ルートのディスパッチロジック:

```
CODE ルート
  │
  ├─ Chrome操作が必要？
  │   └─ Yes → Coder3（常にCoder3専用）
  │
  └─ No → Coderフォールバックチェーン
      ├─ Coder1 利用可能？ → Coder1
      ├─ Coder2 利用可能？ → Coder2
      └─ Coder3 利用可能？ → Coder3
```

明示指定（CODE1, CODE2, CODE3）はフォールバックチェーンをバイパスし、指定Coderに直接ルーティング。

### 5.3 Chrome操作の判定

Chrome操作の判定はルール辞書で行う（v6拡張）:

| キーワード | Route |
|-----------|-------|
| Chrome, ブラウザ, ブラウザで, 画面操作, スクレイピング, ページを開, Webを操作 | `CODE3` |

Chrome操作キーワードがマッチした場合、ルール辞書が `CODE3` を返すため、`executeTask()` で Coder3 に直接ルーティングされる。

### 5.4 ビジー状態の管理

Coderのビジー状態を管理するため、Orchestratorにステート管理を追加する。

```go
// CoderStatus はCoderの利用状態
type CoderStatus struct {
    mu       sync.Mutex
    busy     map[string]bool // "coder1", "coder2", "coder3"
}

func (s *CoderStatus) Acquire(name string) bool  // ビジーでなければ占有してtrue
func (s *CoderStatus) Release(name string)        // 解放
func (s *CoderStatus) IsBusy(name string) bool    // 状態確認
```

### 5.5 CODE ルートの新しいディスパッチ（擬似コード）

```go
func (o *MessageOrchestrator) executeCodeRoute(ctx context.Context, t task.Task) (string, error) {
    // フォールバックチェーン: coder1 → coder2 → coder3
    coders := []struct {
        name  string
        coder CoderAgent
    }{
        {"coder1", o.coder1},
        {"coder2", o.coder2},
        {"coder3", o.coder3},
    }

    for _, c := range coders {
        if c.coder == nil {
            continue
        }
        if !o.coderStatus.Acquire(c.name) {
            continue // ビジー → 次へ
        }
        defer o.coderStatus.Release(c.name)
        return c.coder.Generate(ctx, t, "You are a code generation assistant.")
    }

    return "", fmt.Errorf("all coders busy or unavailable")
}
```

## 8. 全体フロー図

```
LINE/Slack
  │
  ▼
MessageOrchestrator.ProcessMessage()
  │
  ├─ HandleChatCommand()  ── /status, /new 等 → 即応答
  │
  ├─ DecideAction()  ── 4段階ルーティング決定
  │   ├─ parseExplicitCommand()  ── /code3 等
  │   ├─ RuleDictionary.Match()  ── キーワード判定
  │   └─ フォールバック CHAT
  │
  └─ executeTask()  ── ルートに応じたディスパッチ
      │
      ├─ CHAT/PLAN/ANALYZE/RESEARCH → Mio (Ollama)
      ├─ OPS                        → Shiro (Worker)
      ├─ CODE (v6)                  → Coderフォールバックチェーン
      │   ├─ Chrome操作? → Coder3
      │   └─ 通常 → Coder1 → Coder2 → Coder3
      ├─ CODE1                      → Coder1 (DeepSeek) 直接
      ├─ CODE2                      → Coder2 (OpenAI) 直接
      └─ CODE3                      → Coder3 (Claude) 直接
```
