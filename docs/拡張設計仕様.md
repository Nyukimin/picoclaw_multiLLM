## 0. 拡張の基本哲学

**以下の 4 原則は絶対に崩さない。**

| # | 原則 | 理由 |
|---|------|------|
| 1 | **Core（共通入口）は安定させる。Domain（カテゴリ固有）だけ増やす。** | Core を変えると全カテゴリに波及する |
| 2 | **Relations（事実）と Similarities（近さ）は分離し続ける。** | 事実と推定を混ぜると説明不能になる |
| 3 | **Similarities は常に TopK。全対全はやらない。** | O(N^2) は件数増で確実に破綻する |
| 4 | **外部 ID は参照で、内部 entity_id は不変の唯一キー。** | 外部 ID は変わりうる。内部 ID で結合を安定させる |

この 4 原則を守る限り、カテゴリが増えても破綻しない。

## 1. カテゴリ追加チェックリスト

新カテゴリ（例: music / game / book / anime / manga）を追加するとき、実装担当がやることは毎回同じ。

### A. Core 拡張（最小）

- `category` enum に追加（`music` / `game` / ...）
- `core_entities` は基本そのまま（`title`, `aliases`, `genres`, `keywords`, `dates`, `description_short` ...）

### B. Domain 追加（カテゴリ固有）

- `domain_music` / `domain_game` / `domain_book` ... のテーブルを追加
- そのカテゴリでしか意味がない項目は Domain に閉じ込める

| カテゴリ | Domain 固有項目の例 |
|---------|-------------------|
| music | ISRC, BPM, label, duration |
| game | platform, store_id, engine |
| book | ISBN, publisher, volumes |
| anime | episodes, studio, season |
| manga | chapters, magazine, serialization_status |

### C. 外部 ID 追加

- `external_ids` に `source` を追加（`musicbrainz` / `steam` / `isbn` ...）
- `UNIQUE(source, external_id)` のルールは継続

### D. ETL アダプタ追加（Extract / Transform）

- 「カテゴリ x 情報源」ごとにアダプタを作る（プラグイン扱い）
- upsert の要は `external_ids` を使った `entity_id` 解決
  - 存在すれば更新
  - なければ新規

### E. Embedding テンプレ追加（任意だが推奨）

- 新カテゴリ用に「テンプレ文字列」を固定する
- `embeddings` は `model` 別に複数持てる設計にする（セクション 3 参照）

### F. 距離レイヤ（metric）追加（必要なら）

- そのカテゴリの距離が必要なら `metric` を追加する
  - 例: `music_audio_like` / `game_genre_like`
- **ただし最初は 2〜3 metric まで。** 増やしすぎると UI も運用も崩れる

> **このチェックリストを守れば、「カテゴリ追加」はルーチン作業になる。**

## 2. 情報源の追加ルール

将来は同じカテゴリに複数の情報源が入る（例: 映画で TMDb + IMDb + Wikidata）。
このときに必要なのは「統合ルール」。よく事故るので、最初から方針を固定する。

### A. 正規化の中心は entity_id

- まず `external_ids` を見て「同一 entity」を判定する
- 判定できない場合は「候補」として保留し、人手 or 後処理で統合する
- **自動誤統合が最悪のケース。** 保留は許容、誤統合は許容しない

### B. 由来（provenance）を残す

どのフィールドがどのソース由来か、最低限でも記録できるようにする。

```sql
field_sources (
  entity_id   TEXT NOT NULL,
  field_name  TEXT NOT NULL,
  source      TEXT NOT NULL,
  fetched_at  TIMESTAMP NOT NULL,
  confidence  REAL DEFAULT 1.0
)
```

後から「なんでこうなった？」が説明できる。

### C. コンフリクト解決ポリシー

- **優先順位をカテゴリごとに決める**
  - 例: 公開日 = TMDb 優先、人物別名 = Wikidata 優先
- 無理に一個に確定せず、候補を残して UI 側で選べる設計も可
  - ただし初期はやりすぎない

## 3. Embedding モデルの変更・追加

モデルの世代更新は必ず来る。壊れないために、最初から「複数 embedding 共存」を前提にする。

### A. embeddings は model をキーに複数保存する

```sql
embeddings (
  entity_id   TEXT NOT NULL,
  model       TEXT NOT NULL,        -- e.g. "e5-large-v1", "bge-m3"
  dim         INTEGER NOT NULL,
  vector      BLOB NOT NULL,
  updated_at  TIMESTAMP NOT NULL,
  PRIMARY KEY (entity_id, model)
)
```

### B. similarity の metric 名に embedding 基盤を含める

命名規約:

```
{category}_{distance_type}__{model_version}
```

例:
- `movie_semantic__e5large_v1`
- `movie_semantic__bge_m3_v1`

こうすると新旧を並走できる（切替・AB テストが可能）。

### C. 再計算はジョブとして扱う

1. **re-embed job**: 再埋め込み
2. **similarity rebuild job**: TopK 再生成
3. 「いつ・どのモデルで」をログと一緒に残す

## 4. 距離レイヤ（metric）の増設ルール

距離は気持ちよく増やせるが、増やしすぎると死ぬ。増設のガイドラインを固定する。

### A. metric は最大 3 種類までを「常用」にする

初期は 2 で十分。

| 常用 metric 例 | 根拠 |
|---------------|------|
| 作品意味距離 | embedding ベース |
| 人物共演距離 | グラフ統計ベース |
| IP 横断距離（将来） | Relations ベース |

### B. metric の根拠が違うものは混ぜない

| 根拠 | 例 |
|------|-----|
| **embedding 系**（意味の近さ） | テキスト embedding の cosine 距離 |
| **グラフ統計系**（共演・同制作陣・共通 IP） | Jaccard 係数、共起スコア |

これを混ぜるなら「別 metric」にする。重み付き合成は v0.2 以降の話。

### C. UI は「metric 切替」が基本形

合成スコアより、まず切替で世界を変える方が説明可能で運用が安定する。

## 5. クロスメディア拡張

原作マンガ → アニメ → 映画のような横断。将来の拡張パターンを 2 つ用意する。

### パターン A: Relations だけで表現（最小）

- `adaptation_of` / `same_ip` / `remake` / `related` を使って結ぶ
- すぐ始められる。v0.1 の延長でいける

```
entity(manga:1) --[adaptation_of]--> entity(anime:1)
entity(anime:1) --[adaptation_of]--> entity(movie:1)
```

### パターン B: IP（作品世界）という上位エンティティを導入（強い）

- `category` に `ip` を追加
- 「映画」「アニメ」「原作マンガ」は `ip` にぶら下げる
- 「同一世界観」が安定して扱える（検索も可視化も綺麗になる）

```
entity(ip:1, title="進撃の巨人")
  ├── entity(manga:1) --[belongs_to_ip]--> entity(ip:1)
  ├── entity(anime:1) --[belongs_to_ip]--> entity(ip:1)
  └── entity(movie:1) --[belongs_to_ip]--> entity(ip:1)
```

**推奨**: 最初は A で十分。横断が増えてきたら B へ移行が自然。

## 6. スケール対応（件数増加）

将来の件数増に対して、壊れない運用の拡張ポイント。

### A. Similarity は常に TopK

- 保存量は `O(N * K)` で頭打ちになる
- K は最初 100、必要に応じて 50〜200 で調整

### B. ANN インデックスは metric 単位で持つ

| インデックス | 用途 |
|------------|------|
| `movie_semantic` のインデックス | 映画の意味距離検索 |
| `music_semantic` のインデックス（将来） | 音楽の意味距離検索 |

更新頻度やチューニングが metric ごとに違うため、分離する。

### C. 更新は「増分」が基本、全体は「たまに」

| 更新タイプ | 頻度 | 内容 |
|-----------|------|------|
| **増分更新** | 随時 | 新規 / 更新分だけ embedding & TopK |
| **全体リビルド** | 月 1 回 or 大更新後 | 全体再計算で帳尻合わせ |

## 7. 説明責任とデバッグのための運用拡張

部下が運用するならここが効く。「なぜこうなった？」に答えられる形。

### A. ジョブ管理（必須）

- `job_id` を発行し、ETL・embed・similarity 生成を追跡する
- 失敗時にどこまで終わったかが分かる

```
job_log (
  job_id      TEXT PRIMARY KEY,
  job_type    TEXT NOT NULL,         -- "etl" / "embed" / "similarity"
  category    TEXT,
  source      TEXT,
  started_at  TIMESTAMP NOT NULL,
  finished_at TIMESTAMP,
  status      TEXT NOT NULL,         -- "running" / "completed" / "failed"
  stats       JSON                   -- {"processed": 1500, "errors": 3}
)
```

### B. 監視指標（最低限）

| 指標 | 意味 |
|------|------|
| 取り込み件数 | ETL の正常動作確認 |
| 更新件数 | 差分の量 |
| 欠損率（description なし等） | データ品質 |
| similarity の平均スコア分布 | embedding 品質 |
| TopK 埋まり率 | 十分なデータ量があるか |
| 外部ソース別の成功率 | API の安定性 |
| レート制限発生率 | スロットリングの影響 |

### C. バージョニング

| 対象 | 管理方法 |
|------|---------|
| schema_version | マイグレーションファイルで管理 |
| embedding_model_version | `embeddings.model` カラムで管理 |
| similarity_metric_version | metric 名の命名規約（`__v1` サフィックス）で管理 |

## 付録A: 部下向け一言サマリ

> この基盤は、**共通の台帳（Core）** + **カテゴリ別の詳細（Domain）** + **事実のつながり（Relations）** + **意味の近さ（Similarities）** でできている。
>
> カテゴリや情報源を増やす時は、**Domain と ETL プラグインを増やすだけ**でよく、Core と ID のルールは一切崩さない。
>
> 距離は全対全にせず **TopK だけ**を持ち、**増分更新 + たまに全体リビルド**でスケールさせる。
