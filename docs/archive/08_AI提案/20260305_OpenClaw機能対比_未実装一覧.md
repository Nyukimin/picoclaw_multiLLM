# OpenClaw 機能対比 & PicoClaw 未実装一覧

**作成日**: 2026-03-05
**ブランチ**: proposal/clean-architecture
**目的**: OpenClaw チートシートを基準に PicoClaw の機能カバレッジを棚卸し

---

## 1. 現状サマリ

### PicoClaw が実装済みの機能

| カテゴリ | 機能 | 実装場所 |
|---------|------|----------|
| Gateway 起動 | HTTP サーバ + LINE Webhook | `cmd/picoclaw/main.go` |
| LINE 連携 | Webhook 受信・署名検証・応答送信・メディア処理 | `internal/adapter/line/` |
| マルチ LLM ルーティング | Chat/Worker/Coder 自動振り分け + 明示コマンド (`/code1`, `/code2`, `/code3`) | `internal/infrastructure/routing/` |
| Worker 即時実行 | Coder が生成した patch を Worker が自動適用 | `internal/application/service/worker_execution_service.go` |
| 分散実行 | SSH + JSON Transport、スタンドアロン Agent モード | `internal/infrastructure/transport/`, `internal/application/orchestrator/distributed_orchestrator.go` |
| ハートビート | 定期タスク実行 + LINE Push 通知 | `internal/application/heartbeat/` |
| メモリ (FileStore) | 長期メモリの読み書き・日次カットオーバー | `internal/domain/memory/`, `internal/infrastructure/persistence/memory/` |
| 会話エンジン | ConversationEngine v5.1 (RecallPack + Persona) | `internal/domain/conversation/`, `internal/infrastructure/persistence/conversation/` |
| MCP 連携 | MCP クライアント（ツール呼び出し） | `internal/infrastructure/mcp/` |
| セッション管理 | JSON ベースセッション永続化 | `internal/infrastructure/persistence/session/` |
| IdleChat | アイドル時の自発的会話 | `internal/application/idlechat/` |
| LLM プロバイダー | Ollama / Claude / DeepSeek / OpenAI | `internal/infrastructure/llm/` |
| Web 検索 | Google Custom Search API 統合 | `internal/infrastructure/tools/` |

### PicoClaw 独自の強み（OpenClaw にない機能）

- **分散実行 (v4.0)**: SSH Transport による複数マシン並列実行
- **Worker 即時実行**: Coder → Proposal → patch 自動適用の完全自動パイプライン
- **Clean Architecture**: Domain/Application/Infrastructure/Adapter の 4 層分離
- **マルチ Coder**: DeepSeek(CODE1) / OpenAI(CODE2) / Claude(CODE3) の使い分け

## 2. 未実装一覧

### 2.1 CLI 管理コマンド（12 項目）

OpenClaw は CLI でシステム全体を管理できる。PicoClaw は `gateway run` 相当のみ。

| # | OpenClaw コマンド | 機能 | 優先度 | 備考 |
|---|---|---|---|---|

| 2 | `gateway restart / stop` | Gateway の再起動・停止 | B | systemd で代替可 |
| 3 | `gateway install` | デーモンインストール | C | systemd unit で代替済み |
| 4 | `channels add / list` | チャネルの追加・一覧 | B | config.yaml 手動編集で代替 |

## 3. 優先度の考え方

### A（高）: 運用品質に直結

- `status / --deep` — 問題発生時に状態を素早く確認
- `health --json` — Health Check API（Ollama 監視の基盤）
- `/status` — チャット内からの状態確認
- `/stop` — 長時間実行の中断
- サブエージェント — AGENT.md で宣言済み、自動化の中核

### B（中）: 利便性向上

- CLI 管理コマンド群（gateway status, channels, models, memory, doctor）
- チャット内コマンド（/context, /compact, /new）
- 追加チャネル（Telegram, Discord, Slack）
- cron ジョブ
- 上流移植の残作業（ContextBuilder, SkillsLoader）

### C（低）: あれば嬉しい

- 音声 (TTS)
- ブラウザ操作
- 初期セットアップウィザード
- 自己更新
- iMessage / WhatsApp
