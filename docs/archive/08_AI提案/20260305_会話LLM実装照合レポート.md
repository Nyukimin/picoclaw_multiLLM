## 1. 目的と設計原則

**目的**: 単発応答ではなく「時間軸で育つ対話」を成立させる。LLM 自体はステートレスとして扱い、会話・記憶・人格・ユーザー状態を外部構造で維持する。

### 設計原則（固定）

| # | 原則 | 実装状態 |
|---|------|---------|
| 1 | 会話ログ ≠ 知識。全部を VectorDB に入れない。 | ✅ 3層分離済み |
| 2 | 記憶には寿命と粒度がある（短期/中期/長期/KB を分ける）。 | ✅ Redis/DuckDB/Qdrant |
| 3 | 1ターンの処理は「入力→想起→判断→生成→記録」で閉じる。 | ✅ BeginTurn→Chat→EndTurn |
| 4 | Spawn 禁止。Subagent は同期呼び出しとしてのみ使う。 | ✅ Spawn なし（Subagent は未実装） |

## 2. 用語定義（会話とは何か）

| 用語 | 定義 | 実装 |
|------|------|------|
| **Message** | 発話の最小単位。speaker と本文、メタ情報を持つ。 | ✅ `conversation.Message` |
| **Turn** | ユーザー入力1回を起点に、想起→判断→生成→記録が完了するまで。 | ✅ BeginTurn/EndTurn |
| **Thread** | 話題の塊。短期記憶の単位。6〜8往復（最大12メッセージ保持）。 | ✅ `conversation.Thread` |
| **Session** | 割り込み復帰/再起動復元に必要なまとまり。24h〜7d。 | ✅ `conversation.SessionConversation` |
| **Conversation** | Thread の連なり。Thread 終了時に要約されて積み上がる。 | ✅ FlushThread で要約→中期格納 |

### PicoClaw 実装で必須の ID

| ID | 定義 | 実装状態 |
|----|------|---------|
| `job_id` | ユーザー要求1件の処理単位 | ✅ `domain/task/jobid.go` |
| `turn_id` | ターンの識別子 | ❌ 未実装 |
| `thread_id` | 文脈の識別子 | ✅ `Thread.ID` |
| `session_id` | セッションの識別子 | ✅ `SessionConversation.ID` |
| `event_id` | ログ上の処理単位識別 | ❌ 未実装 |

## 3. 全体構造（Chat / Worker / Coder の責務分離）

### 3.1 コンポーネント対応表（固定）

#### Chat（会話担当・入口）

| 責務 | 実装状態 | 実装場所 |
|------|---------|---------|
| ユーザー I/F（LINE/Discord/Slack/Web） | ✅ LINE のみ | `adapter/line/` |
| 会話の見た目の生成（口調・安全・出力整形） | ✅ Persona 注入 | `conversation/persona.go` |
| Worker に渡すための入力正規化（job_id/turn_id 付与） | ⚠️ job_id のみ | `task/jobid.go` |
| 最終応答の提示、Thread の見た目上の連続性維持 | ✅ RecallPack | `conversation/recall_pack.go` |

#### Worker（中核オーケストレータ）

| 責務 | 実装状態 | 実装場所 |
|------|---------|---------|
| Thread 継続/終了判定 | ❌ 自動判定なし | -- |
| 想起パック組み立て（短期→中期→長期→KB） | ⚠️ Mio が直接実行 | `agent/mio.go` (Chat 内) |
| ルーティング判断 | ✅ | `agent/mio.go` (DecideAction) |
| ツール実行 | ✅ | `infrastructure/tools/` |
| Coder への依頼生成・結果の検証・統合 | ✅ | `application/orchestrator/` |
| 記録（要約・メタ・埋め込み・TTL 管理） | ✅ | `conversation/real_manager.go` |

**重要なギャップ**: 仕様では Worker が想起パック生成を担当するが、現在の実装では **Mio（Chat）が ConversationEngine を直接呼び出して** BeginTurn/EndTurn を実行している。Worker への委譲は未実施。

#### Coder（実装・重作業担当）

| 責務 | 実装状態 | 実装場所 |
|------|---------|---------|
| コード生成、リファクタ、テスト観点 | ✅ | `domain/agent/coder.go` |
| Skill/CLI/スクリプト生成 | ✅ Proposal 経由 | `domain/proposal/` |
| 外部 LLM を使う高難度推論 | ✅ Claude/DeepSeek/OpenAI | `infrastructure/llm/` |
| 返却物は差分/ファイル/実行手順（機械可読寄り） | ✅ PatchCommand | `domain/patch/` |

## 4. 1ターンの処理フロー（Spawn 禁止に最適化）

### 仕様フロー vs 現在の実装

| # | 仕様 | 現在の実装 | ギャップ |
|---|------|-----------|---------|
| 1 | **Chat**: ユーザー入力受信、job_id/turn_id 発行 | LINE Handler → Orchestrator | turn_id なし |
| 2 | **Worker**: Thread 判定（継続 or 新規開始） | ❌ 自動判定なし | Thread は手動管理もなし |
| 3 | **Worker**: 想起（Recall）短期→中期→長期→KB | Mio.Chat() 内で BeginTurn が実行 | Worker ではなく Chat が実行 |
| 4 | **Worker**: 行動計画（Plan） | DecideAction でルーティング | 会話だけ/調査/実装の分岐は実装済み |
| 5 | **Worker→Coder** 同期 Subagent | Orchestrator 経由で Coder 呼び出し | JSON I/F は未構造化 |
| 6 | **Worker**: 統合・検証 | WorkerExecutionService で patch 適用 | 返却物の整合チェックは基本的 |
| 7 | **Chat**: 最終応答生成 | Mio.Chat() で LLM 応答生成 | Persona 注入済み |
| 8 | **Worker**: 記録 | EndTurn で Store + FlushThread | event_id なし |

### 現在の実際のフロー

```
LINE Message
  → LINE Handler (adapter/line/)
  → MessageOrchestrator.Handle()
    → MioAgent.DecideAction()        -- ルーティング判断
    → route == CHAT:
        MioAgent.Chat()
          → ConversationEngine.BeginTurn()  -- 想起パック生成
          → LLMProvider.Generate()          -- 応答生成
          → ConversationEngine.EndTurn()    -- 記録
    → route == CODE*:
        CoderAgent.GenerateProposal()
        WorkerExecutionService.Execute()
  → LINE Handler: 応答送信
```

## 5. 記憶レイヤー仕様（寿命と粒度）

### 5.1 レイヤー一覧

| レイヤー | 物理ストア | 寿命 | 目的 | 実装状態 |
|---------|-----------|------|------|---------|
| **短期記憶** | Redis (TTL 24h) | Thread 生存期間 | 直近文脈、ルーティング | ✅ RedisStore |
| **中期記憶** | DuckDB (warm, 7d) | 7日 | 再起動復帰、直近会話検索 | ✅ DuckDBStore |
| **長期記憶** | Qdrant VectorDB | 原則無期限 | 要約・嗜好・方針の意味検索 | ✅ VectorDBStore |
| **知識ベース (KB)** | VectorDB (namespace `kb:<domain>`) | 原則無期限 | RAG 用資料 | ❌ namespace 分離未実装 |
| **Persona State** | Go 構造体 (RAM) | プロセス生存期間 | 口調・距離・禁則 | ✅ PersonaState |
| **User Profile** | 長期記憶 + KV | 原則無期限 | 嗜好・優先順位・プロジェクト文脈 | ⚠️ 型定義のみ（自動抽出なし） |

**重要**: 短期/中期/長期/KB は共有資産。キャラごとに分断しない。キャラ差は Persona State で出す。

### 5.2 Thread 開始/終了条件

#### 開始条件

| 条件 | 実装状態 |
|------|---------|
| 初回入力 | ✅ CreateThread |
| 明確な新トピック宣言（「別件」「ところで」） | ❌ 未実装 |
| ドメインが変わった | ❌ 未実装 |
| 前 Thread が終了している | ❌ 未実装（自動終了なし） |

#### 終了条件

| 条件 | 実装状態 |
|------|---------|
| 明示切替語 | ❌ 未実装 |
| 埋め込み類似度が閾値未満（例: cos < 0.75） | ❌ 未実装 |
| ターン上限到達 | ❌ 未実装 |
| 無入力タイムアウト（例: 10分） | ❌ 未実装 |

#### 終了時処理

| 処理 | 実装状態 |
|------|---------|
| Thread 要約（事実/決定事項/未決を分離） | ✅ FlushThread + LLMSummarizer |
| keywords 抽出 | ✅ ExtractKeywords |
| embedding 生成 | ✅ OllamaEmbedder |
| 中期（DuckDB）へ格納 | ✅ DuckDBStore.SaveThreadSummary |
| 長期昇格判定（方針/嗜好/継続案件/禁則のみ） | ⚠️ IsNovelInformation はあるが昇格ロジックは基本的 |

## 6. 想起パック（Recall Pack）の注入仕様

各ターンで `RecallPack` を作って LLM 入力に注入する。

| フィールド | 内容 | 上限 | 実装状態 |
|-----------|------|------|---------|
| `ShortContext` | 直近メッセージ | 最大 12 | ✅ |
| `MidSummaries` | 直近スレッド要約 | 最大 3 | ✅ |
| `LongFacts` | 長期からの重要点（箇条書き） | 最大 3 | ✅ |
| `KBSnippets` | KB からの根拠断片（出典メタ付き） | 最大 2 | ⚠️ 型あり、KB namespace 未分離 |
| `Persona` | キャラクター設定 | -- | ✅ |
| `UserProfile` | ユーザーの好み・傾向 | -- | ⚠️ 型あり、自動蓄積なし |
| `Constraints` | 禁則・優先順位・出力フォーマット規約 | -- | ⚠️ 型のみ（トークン制約） |

**RecallPack は Worker が生成し、Chat は「見せ方」だけを担当する**（仕様）。
**現在の実装**: Mio（Chat）が ConversationEngine.BeginTurn() を直接呼び出して生成している。

## 8. 非機能要件

| 要件 | 仕様 | 実装状態 |

| **可観測性** | 全ログに job_id/turn_id/thread_id/session_id/event_id を付与 | ⚠️ job_id のみ |

## 9. 未実装項目と実装ギャップ

### 9.1 優先度 A（運用品質に直結）

| # | 項目 | 影響 | 推奨アクション |
|---|------|------|--------------|
| 1 | **Thread 自動判定** | 話題が変わっても同じ Thread に積み続ける | 埋め込み類似度 + タイムアウトで Thread 開始/終了を自動化 |
| 2 | **turn_id / event_id** | ログの追跡粒度が粗い | task.Task に turn_id を追加、ログに event_id を付与 |
| 3 | **UserProfile 自動抽出** | ユーザーの好みが蓄積されない | EndTurn 時に LLM でプロファイル要素を抽出 |

### 9.2 優先度 B（設計整合性）

| # | 項目 | 影響 | 推奨アクション |
|---|------|------|--------------|
| 4 | **Worker による想起パック生成** | 仕様の責務分離に違反 | Mio.Chat() → Worker 経由に変更（大きなリファクタ） |
| 5 | **KB namespace 分離** | ドメイン知識と会話記憶が混在 | VectorDB の namespace を `user:<uid>` と `kb:<domain>` に分離 |
| 6 | **Worker→Coder JSON I/F 構造化** | Coder への依頼が非構造的 | CodingRequest 型を定義して Orchestrator に導入 |
| 7 | **Constraints 実装** | 禁則・出力フォーマットが未制御 | RecallPack.Constraints にルールを注入するロジック追加 |

### 9.3 優先度 C（将来拡張）

| # | 項目 | 推奨アクション |
|---|------|--------------|
| 8 | 情報流出制御 | Worker がコンテキスト最小化フィルタを適用 |
| 9 | 長期昇格の精緻化 | 方針/嗜好/継続案件/禁則の分類 LLM を追加 |
| 10 | Thread 要約の事実/決定/未決の分離 | Summarizer のプロンプトを構造化 |

### 9.4 設計判断メモ: Worker 委譲について

仕様では Worker が想起パック生成を担当するが、現在の PicoClaw アーキテクチャでは:

- **Mio（Chat）が ConversationEngine を直接保持** (`mio.go:21`)
- BeginTurn/EndTurn は Mio.Chat() 内で呼ばれる (`mio.go:69-121`)
- Worker（Shiro）は会話記憶を持たない

この設計は「Mio が会話の主体」という PicoClaw の基本思想に基づいている。仕様通りに Worker に委譲する場合、MessageOrchestrator レベルでの大幅なリファクタが必要。

**現実的な妥協案**: 想起パック生成は Mio に残し、Worker への委譲は「Mio が RecallPack を生成した後に Worker/Coder に必要部分だけ渡す」形にする。
