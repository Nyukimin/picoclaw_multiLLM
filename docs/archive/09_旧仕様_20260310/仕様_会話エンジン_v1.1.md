# RenCrow 会話エンジン仕様 v1.1

**バージョン**: 1.1
**作成日**: 2026-03-05
**ステータス**: Draft
**前提**: v5.0 会話LLMシステム（3層記憶実装済み）

---

## 1. 概要

### 1.1 目的

RenCrow の会話品質を「記憶はあるが使っていない」状態から「記憶を活用した文脈ある会話」へ進化させる。

**現状の課題**:
- ConversationManager の Recall 結果が**捨てられている**（`_, err := m.conversationMgr.Recall(...)`）
- LLM プロンプトに過去文脈が**注入されていない**
- キャラクター（ミオ）のペルソナが**定義されていない**
- ユーザープロファイル（好み・傾向）が**蓄積されていない**

**解決策**: ConversationEngine を導入し、Recall → RecallPack → プロンプト注入の流れを確立する。

### 1.2 設計方針

| 方針 | 説明 |
|------|------|
| **Mio が頭脳** | 意思決定・対話管理は Mio（Chat Agent）が担当。Worker は道具係のまま |
| **ConversationEngine = Mio の記憶脳** | 既存 ConversationManager をラップし、RecallPack 生成を追加 |
| **RecallPack = LLM 注入フォーマット** | 想起結果を構造化して LLM プロンプトに注入 |
| **段階的導入** | 既存の3層記憶インフラをそのまま活用、新規インフラ不要 |
| **後方互換** | `conversation.enabled: false` で従来動作にフォールバック |

## 2. アーキテクチャ

### 2.1 コンポーネント構成

```
ユーザー入力
    │
    ▼
┌─────────────────────────────────────────────┐
│  Mio (Chat Agent) — 意思決定・対話管理       │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │  ConversationEngine（新規）             │  │
│  │  ┌──────────────┐  ┌───────────────┐  │  │
│  │  │ BeginTurn()  │  │  EndTurn()    │  │  │
│  │  │ ・Recall     │  │  ・Store      │  │  │
│  │  │ ・RecallPack │  │  ・FlushThread│  │  │
│  │  │ ・Persona注入│  │  ・Profile更新│  │  │
│  │  └──────────────┘  └───────────────┘  │  │
│  │         │                              │  │
│  │         ▼                              │  │
│  │  ConversationManager（既存）            │  │
│  │  ┌──────┐ ┌────────┐ ┌──────────┐     │  │
│  │  │Redis │ │DuckDB  │ │Qdrant    │     │  │
│  │  │短期  │ │中期    │ │長期      │     │  │
│  │  └──────┘ └────────┘ └──────────┘     │  │
│  └────────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 2.2 1ターンの処理フロー

```
1. ユーザー入力受信
2. ConversationEngine.BeginTurn(sessionID, userMessage)
   a. ConversationManager.Recall() → 過去文脈取得
   b. Thread 判定（継続 or 新規作成）
   c. RecallPack 構築
   d. PersonaState 読み込み
   e. UserProfile 読み込み
3. RecallPack + Persona + UserProfile → LLM プロンプト組み立て
4. LLM 呼び出し → 応答生成
5. ConversationEngine.EndTurn(sessionID, response)
   a. ConversationManager.Store(userMessage)
   b. ConversationManager.Store(mioResponse)
   c. Thread 終了判定 → FlushThread（要約・embedding・VectorDB 保存）
   d. UserProfile 更新（プロファイル抽出）
6. 応答返却
```

## 3. RecallPack（想起パック）

### 3.1 定義

RecallPack は Recall 結果を構造化し、LLM プロンプトに注入するためのフォーマット。

```go
type RecallPack struct {
    // 短期文脈: 現在の Thread 内の直近メッセージ（最大12件）
    ShortContext []conversation.Message

    // 中期要約: 同一セッション内の過去 Thread 要約（最大3件）
    MidSummaries []conversation.ThreadSummary

    // 長期事実: VectorDB から類似検索した過去の知識（最大3件）
    LongFacts []string

    // KB スニペット: ドメイン知識ベースからの関連情報（最大2件）
    KBSnippets []string

    // ペルソナ: キャラクター設定
    Persona PersonaState

    // ユーザープロファイル: ユーザーの好み・傾向
    UserProfile UserProfile

    // 制約: トークン上限・応答スタイル等
    Constraints PromptConstraints
}
```

### 3.2 プロンプト注入テンプレート

```
[システムプロンプト]
あなたは「ミオ（澪）」です。
{Persona.SystemPrompt}

[ユーザープロファイル]
{UserProfile の要約テキスト}

[過去の会話から思い出したこと]
{MidSummaries の要約テキスト}
{LongFacts の要約テキスト}

[参考知識]
{KBSnippets の内容}

[直近の会話]
{ShortContext のメッセージ履歴}

[ユーザーの質問]
{userMessage}
```

### 3.3 トークン予算

| セクション | 最大トークン | 優先度 |
|-----------|------------|--------|
| システムプロンプト + Persona | 200 | 必須 |
| UserProfile | 100 | 高 |
| ShortContext（直近会話） | 1,500 | 必須 |
| MidSummaries（中期要約） | 500 | 中 |
| LongFacts（長期事実） | 300 | 低 |
| KBSnippets（ドメイン知識） | 300 | 低 |
| ユーザー入力 | 500 | 必須 |
| 応答余裕 | 600 | 必須 |
| **合計** | **~4,000** | - |

**注**: Ollama chat-v1 の MaxContext は 8,192 トークン。プロンプト注入は ~4,000 に収める。

## 4. PersonaState（ペルソナ）

### 4.1 定義

```go
type PersonaState struct {
    Name         string `json:"name"`          // "ミオ"
    SystemPrompt string `json:"system_prompt"` // キャラクター設定プロンプト
    Tone         string `json:"tone"`          // "friendly", "formal", "casual"
    Mood         string `json:"mood"`          // "neutral", "cheerful", "thoughtful"
}
```

### 4.2 ミオのデフォルトペルソナ

```
あなたは「ミオ（澪）」という名前の AI アシスタントです。
性格: 明るく親切で、ユーザーの質問に丁寧に答えます。
口調: フレンドリーだが丁寧語を基本とします。
特徴:
- 過去の会話を覚えていて、文脈を踏まえた応答をします
- わからないことは素直に「わかりません」と言います
- 技術的な質問には正確に、雑談には楽しく応答します
```

## 5. UserProfile（ユーザープロファイル）

### 5.1 定義

```go
type UserProfile struct {
    UserID      string            `json:"user_id"`
    Preferences map[string]string `json:"preferences"` // "好きな映画ジャンル": "SF"
    Facts       []string          `json:"facts"`        // "エンジニア", "Go言語を使用"
    UpdatedAt   time.Time         `json:"updated_at"`
}
```

### 5.2 プロファイル抽出

Thread 終了時に LLM でプロファイル情報を抽出:

```
以下の会話から、ユーザーに関する永続的な事実や好みを抽出してください。
一時的な情報（「今日は疲れた」等）は除外してください。

【会話】
（Thread.Turns の内容）

【抽出結果（JSON）】
{"preferences": {"key": "value"}, "facts": ["fact1", "fact2"]}
```

### 5.3 保存先

- **短期**: Redis（`profile:<user_id>`、TTL: 7日）
- **永続**: VectorDB の `user:<user_id>` namespace にメタデータとして保存

## 6. Thread 判定ロジック

### 6.1 Thread 継続/新規判定

```
ユーザー入力を受信:
  1. アクティブ Thread があるか？
     → なし: 新規 Thread 作成
     → あり: 継続判定へ
  2. 継続判定:
     a. 話題切替語チェック（「ところで」「別件」「話変わるけど」）
        → 検出: 新規 Thread 作成
     b. 最後のメッセージからの経過時間
        → 10分超: 新規 Thread 作成
     c. ターン数チェック
        → 12ターン超: 新規 Thread 作成（古い Thread は FlushThread）
     d. ドメイン変化チェック（embedding 類似度）
        → コサイン類似度 < 0.75: 新規 Thread 作成
  3. 上記いずれにも該当しない → 既存 Thread に追加
```
