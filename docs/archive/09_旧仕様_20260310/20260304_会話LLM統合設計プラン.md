# 会話LLM統合設計プラン（Draft v0.9 → PicoClaw v5.0）

**作成日**: 2026-03-04
**目的**: 会話LLMシステムをPicoClawのChat部に統合し、記憶を持つ継続会話を実現

---

## 1. 統合の基本方針

### 1.1 キャラクター対応

会話LLM仕様の**キャラクター**を、PicoClawの**Agent**に対応させます。

| 会話LLM仕様 | PicoClaw Agent | LLM | 責務 | 会話参加条件 |
|-------------|----------------|-----|------|-------------|
| **ルミナ** | Chat（Mio） | Ollama (chat-v1) | ユーザー対話、即答、ルーティング | **常時参加** |
| **ノクス** | Worker（Shiro） | Ollama (worker-v1) | 実行・道具係、検索 | **タスクが空いている時のみ** |
| **クラリス等** | Coder（Aka/Ao/Gin）等 | DeepSeek/OpenAI/Claude | 仕様設計、実装、推論 | **将来的に参加可能（最大数未定）** |

**設計判断**:
- **Phase 1**: Chat（Mio）が主会話担当、Worker（Shiro）はタスク実行と空き時間の会話
- **Phase 2**: Coder達も会話に参加できる仕組みを用意（将来拡張）
- **スケーラビリティ**: キャラクター数は可変（最大数未定）

### 1.2 記憶対象

**記憶の基本原則**:
- ✅ **ユーザー ⇄ Chat（Mio）の会話** - 常に記憶対象
- ✅ **Chat ⇄ Worker/Coder の会話** - **Chatが知らない情報なら記憶対象**
- ❌ **Worker ⇄ Coder の内部対話** - 記憶対象外（実行ログとして保存）

**判定基準: 「Chatが知らない情報」とは？**

Thread終了時に以下を判定:
1. **VectorDB類似検索**: 既存の長期記憶と比較（embedding）
2. **類似度閾値**: 類似度 < 0.8 なら「新規情報」と判定
3. **記憶**: 新規情報なら長期記憶へ保存

**例**:
- ❌ Worker: 「ファイルを作成しました」→ 既知の手順、記憶不要
- ✅ Worker: 「このライブラリはバージョン3.0以降で破壊的変更があります」→ 新規知識、記憶
- ✅ Coder: 「この設計パターンはメモリ効率が良いです」→ 新規知識、記憶

**理由**:
- Chatの知識ベースを継続的に拡充
- ユーザーに提供できる情報の質を向上
- Worker/Coderから得た知見を次回以降の会話に活用

### 1.3 アーキテクチャ統合

**現在（v4.0）**:
```
ユーザー
  ↓
LINE Adapter
  ↓
MessageOrchestrator（Mio）
  ↓ ルーティング
Worker / Coder（実行）
```

**統合後（v5.0）**:
```
ユーザー
  ↓
LINE Adapter
  ↓
ConversationOrchestrator（LangGraph）
  ├─ Router Node（発話者決定: Mio/Shiro）
  ├─ Mio Node（Chat）
  ├─ Shiro Node（Worker）
  └─ Memory Manager（短期/中期/長期/KB）
  ↓
Worker / Coder（実行）
```

## 2. 記憶レイヤー設計

### 2.1 レイヤー一覧

| レイヤー | 物理ストア | TTL/保持期間 | 目的 |
|---------|-----------|-------------|------|
| **短期記憶** | LangGraph State（RAM） | 1 Thread（6〜8ターン） | アクティブ会話の文脈 |
| **中期記憶** | Redis（hot）→ DuckDB（warm） | 24h（Redis）→ 7d（DuckDB） | セッション継続、割り込み復帰 |
| **長期記憶** | VectorDB `user:<uid>` + MetaDB | 無期限（削除可能） | プロファイル、好み、Thread要約 |
| **知識ベース** | VectorDB `kb:<domain>` | 無期限（ETL更新） | RAG用ドメイン資料 |

### 2.2 短期記憶（Thread State）

**保持構造**（LangGraph State）:
```json
{
  "thread_id": 42,
  "session_id": "sess_20260304_001",
  "domain": "movie",
  "turns": [
    {"speaker": "user", "msg": "おすすめの映画教えて", "ts": "2026-03-04T10:00:00Z"},
    {"speaker": "mio", "msg": "どんなジャンルがお好きですか？", "ts": "2026-03-04T10:00:05Z"},
    {"speaker": "user", "msg": "SF映画", "ts": "2026-03-04T10:00:10Z"}
  ],
  "targets": ["mio"],
  "ct": {"mio": 0, "shiro": 1},
  "auto_prompt": 0
}
```

**保持数**: 最新12メッセージ（6〜8往復相当）

### 2.3 Thread開始/終了条件

**開始条件**（いずれか）:
- 初回入力
- 新トピック提示（キーワード: 「ところで」「別件」「質問変えて」）
- キャラ側提案へのユーザー応答

**終了条件**（いずれか）:
- 話題切替語検出
- コサイン類似度 < 0.75（埋め込み評価）
- ドメイン変化（CHAT → RESEARCH 等）
- 10分無入力 or ターン上限（12ターン超）

**終了時処理**:
1. Thread要約生成（LLM）
2. キーワード抽出（LLM）
3. Embedding生成（VectorDB用）
4. 中期記憶へフラッシュ（Redis → DuckDB）

### 2.4 中期記憶（Redis / DuckDB）

**Redis（hot）キー設計**:
```
sess:<session_id> = {
  "history": [...],        # 最近のThread要約リスト
  "agenda": "...",         # 継続中のタスク
  "targets": ["mio"],      # 最後に話していたキャラ
  "last_thread_id": 42
}
TTL: 24h
```

**DuckDB（warm）テーブル設計**:
```sql
CREATE TABLE session_thread (
  thread_id INTEGER PRIMARY KEY,
  session_id TEXT NOT NULL,
  ts_start TIMESTAMP NOT NULL,
  ts_end TIMESTAMP,
  domain TEXT,
  summary TEXT,
  keywords TEXT[],
  embedding FLOAT[],
  INDEX idx_session (session_id),
  INDEX idx_domain (domain),
  INDEX idx_ts (ts_start)
);
```

**保持期間**: 7日（DuckDB）、24時間（Redis hot cache）

### 2.5 長期記憶（VectorDB + MetaDB）

**VectorDB namespace**: `user:<uid>`（例: `user:line_U123456`）

**保存対象**:
- Thread要約
- 重要プロファイル（「好きな映画: SF」等）
- 繰り返し出現するトピック
- KPI履歴（満足度、エンゲージメント）

**取得方法**:
- 類似検索（embedding）
- メタフィルタ（domain, timestamp, kpi_tags）

### 2.6 知識ベース（Domain KB）

**VectorDB namespace**: `kb:<domain>`（例: `kb:movie`, `kb:history`）

**データソース**:
- Worker の RESEARCH 検索結果（Google Custom Search）
- ETL による外部データ取り込み（週次/月次）

**用途**:
- Chat の即答時に参照（RAG）
- Worker の実行時に参照（ツール情報、FAQ）

## 3. LangGraph 設計

### 3.1 ノード構成

```
ConversationOrchestrator (LangGraph)
  ├─ Router Node（発話者決定）
  ├─ Mio Node（Chat Agent）
  ├─ Shiro Node（Worker Agent）
  ├─ Memory Manager（想起・記録）
  └─ Adapter Manager（検索・KB参照）
```

### 3.2 Router Node（発話者決定）

**優先順位**（上から優先）:
1. **ユーザー指名**: `@ミオ`, `@シロ` → 明示的指定
2. **Agent状態フィルタ**:
   - Chat（Mio）: 常時参加可能
   - Worker（Shiro）: **タスクが空いている時のみ**参加可能（`idle_status == true`）
   - Coder（将来）: 条件未定（Phase 2で設計）
3. **ドメイン適性**: `adapter_priority`（検索 → Shiro, 即答 → Mio）
4. **ラウンドロビン + クールタイム**: 連投抑制（`ct`）
5. **スムーズ補正**: 直前発話者と同一を避ける

**Workerのタスク状態管理**:
```go
type AgentStatus struct {
    AgentName string    `json:"agent_name"`
    IsIdle    bool      `json:"is_idle"`       // タスクが空いているか
    LastTask  time.Time `json:"last_task"`     // 最後のタスク完了時刻
}
```

**実装**:
- 既存の `internal/infrastructure/routing/classifier.go` を拡張
- 新規: `internal/domain/agent/router.go`（Router Node実装）
- 新規: `internal/domain/agent/status.go`（Agent状態管理）

### 3.3 Mio Node（Chat Agent）

**責務**:
- ユーザー対話
- 即答（検索 + KB参照）
- ルーティング決定（Worker/Coder への委譲）

**実装**:
- 既存: `internal/domain/agent/mio.go`
- 拡張: 記憶想起（Memory Manager連携）

### 3.4 Shiro Node（Worker Agent）

**責務**:
- 実行・道具係
- RESEARCH 検索（バックグラウンドDB構築）
- Coder オーケストレーション

**実装**:
- 既存: `internal/domain/agent/shiro.go`
- 拡張: 記憶記録（実行結果を中期記憶へ）

### 3.5 Memory Manager

**責務**:
- 想起（Recall）: 短期 → 中期 → 長期 → KB の順に検索
- 記録（Store）: ターン終了時、Thread終了時
- **新規性判定**: Chatが知らない情報かを判定
- 忘却（Forget）: TTL管理、削除API

**実装**:
- 新規: `internal/domain/memory/manager.go`
- 新規: `internal/infrastructure/persistence/memory/`（Redis/DuckDB/VectorDB）

**想起フロー**:
```
1. 短期（Thread turns）から最新メッセージ取得
2. 中期（Redis → DuckDB）からSession要約取得
3. 長期（VectorDB）から類似Thread検索
4. KB（VectorDB）からドメイン知識検索
→ 想起パック（role: "memory"）を短期に注入
```

**新規性判定フロー**（Thread終了時）:
```
1. Thread内のメッセージから重要情報を抽出（LLM）
2. 抽出情報のEmbedding生成
3. VectorDB類似検索（長期記憶 + KB）
4. 類似度 < 0.8 なら「新規情報」と判定
5. 新規情報なら長期記憶へ保存
```

**API例**:
```go
// 新規性判定
func (m *MemoryManager) IsNovelInformation(ctx context.Context, msg Message) (bool, error)

// 記憶判定（ユーザー⇄Chat or 新規情報）
func (m *MemoryManager) ShouldMemorize(ctx context.Context, thread Thread) (bool, error)
```

### 3.6 Adapter Manager

**責務**:
- Web検索（Google Custom Search API）
- KB検索（VectorDB）
- 結果要約と出典メタ付与

**実装**:
- 既存: `internal/infrastructure/tools/runner.go`（Web検索）
- 拡張: KB検索機能追加

## 4. データモデル

### 4.1 Message（メッセージ）

```go
type Message struct {
	Speaker   string    `json:"speaker"`    // user, mio, shiro, system, tool, memory
	Msg       string    `json:"msg"`        // 本文
	Timestamp time.Time `json:"ts"`         // タイムスタンプ
	Meta      map[string]interface{} `json:"meta,omitempty"` // 引用元、検索URL、ドメイン等
}
```

### 4.2 Thread（スレッド）

```go
type Thread struct {
	ID        int64     `json:"thread_id"`
	SessionID string    `json:"session_id"`
	Domain    string    `json:"domain"`      // movie, tech, general等
	Turns     []Message `json:"turns"`       // 最新12件
	Targets   []string  `json:"targets"`     // 指名キャラ
	Cooldown  map[string]int `json:"ct"`     // クールタイム
	StartTime time.Time `json:"ts_start"`
	EndTime   *time.Time `json:"ts_end,omitempty"`
}
```

### 4.3 Session（セッション）

```go
type Session struct {
	ID           string    `json:"session_id"`
	UserID       string    `json:"user_id"`       // LINE User ID等
	History      []ThreadSummary `json:"history"`  // Thread要約リスト
	Agenda       string    `json:"agenda"`        // 継続中のタスク
	LastThreadID int64     `json:"last_thread_id"`
	CreatedAt    time.Time `json:"created_at"`
	UpdatedAt    time.Time `json:"updated_at"`
}
```

### 4.4 ThreadSummary（Thread要約）

```go
type ThreadSummary struct {
	ThreadID  int64     `json:"thread_id"`
	Domain    string    `json:"domain"`
	Summary   string    `json:"summary"`    // LLM生成要約
	Keywords  []string  `json:"keywords"`   // LLM抽出キーワード
	Embedding []float32 `json:"embedding"`  // VectorDB用
	StartTime time.Time `json:"ts_start"`
	EndTime   time.Time `json:"ts_end"`
}
```

## 6. 非機能要件

### 6.1 可観測性

**必須項目**:
- すべてのログに `turn_id`, `thread_id`, `session_id` を付与
- Thread開始/終了をログ出力
- 想起パックの内容をログ出力（デバッグ用）

**実装**:
- `internal/adapter/logging/structured_logger.go`（既存）を拡張

### 6.2 フォールトトレランス

**検索失敗時の挙動**:
- Web検索失敗 → 「出典なし」を明示し、推測は避ける
- VectorDB失敗 → 短期記憶のみで継続

**実装**:
- エラーハンドリング（`log.Warn` + フォールバック）

### 6.3 データ寿命

**TTL管理**:
- Redis: 24h自動削除
- DuckDB: 7d後にアーカイブ（オプション）
- VectorDB: 手動削除（削除API提供）

**実装**:
- Redis TTL設定
- DuckDB定期クリーンアップ（cron）

## 8. リスクと対策

### 8.1 リスク1: メモリ使用量増加

**リスク**: LangGraph State + Redis でメモリ <10MB を超える可能性

**対策**:
- Thread保持数を最小化（12メッセージ = 約2-3KB）
- Redis は別プロセス（メモリ制約外）
- 定期的なメモリプロファイリング

### 8.2 リスク2: VectorDB応答遅延

**リスク**: 長期記憶検索で応答時間が増加

**対策**:
- VectorDB検索はバックグラウンド実行（goroutine）
- タイムアウト設定（500ms）
- キャッシュ（Redis）

### 8.3 リスク3: Thread要約の品質

**リスク**: LLM生成要約が不正確

**対策**:
- プロンプトエンジニアリング（Few-shot例示）
- 要約結果の手動レビュー（初期）
- フィードバックループ（ユーザー評価）
