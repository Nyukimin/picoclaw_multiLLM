# コンテキスト収集・プロンプト構築アーキテクチャ

最終更新: 2026-02-21  
対象コード: `pkg/agent/context.go`, `pkg/agent/memory.go`, `pkg/agent/loop.go`, `pkg/agent/router.go`

---

## 概要

RenCrowは、複数の情報ソースからコンテキストを収集し、ルート（CHAT/CODE/PLAN等）に応じてプロンプトを動的に構築してLLMに送信する。本ドキュメントはその全体フローを記述する。

---

## 1. 情報ソース一覧

| # | ソース | 収集場所 | 担当関数 | 概要 |
|---|--------|----------|----------|------|
| A | Identity（固定情報） | ハードコード | `getIdentity()` | 時刻・ランタイム・ルール |
| B | ブートストラップファイル | `workspace/*.md` | `LoadBootstrapFiles()` | ペルソナ・ユーザー情報等 |
| C | スキル | `workspace/skills/`, `~/.picoclaw/skills/`, `./skills/` | `BuildSkillsSummary()` | スキル定義のサマリ |
| D | メモリ | `workspace/memory/` | `GetMemoryContext()` | 長期記憶・日次ノート |
| E | セッション履歴 | `workspace/sessions/{key}.json` | `SessionManager` | 会話履歴・要約 |
| F | ツール定義 | `pkg/tools/` 各ツール | `buildToolsSection()` | 利用可能ツール一覧 |

## 2. 各ソースの詳細

### A. Identity（`getIdentity()`）

`pkg/agent/context.go` の `getIdentity()` で生成される固定セクション。

**含まれる情報:**

| 項目 | 内容 | 例 |
|------|------|-----|
| 現在時刻 | `time.Now()` のフォーマット | `2026-02-21 14:30 (Friday)` |
| ランタイム | OS・アーキテクチャ・Go版 | `linux amd64, Go go1.22.0` |
| ワークスペースパス | 絶対パス | `/home/user/picoclaw/workspace` |
| 各種パス案内 | Memory, Daily Notes, Skills, Persona | パス一覧 |
| 運用ルール | 8GB制約、HDD永続化、サブエージェント禁止等 | — |

**CHATルート固有の追加:**
- Mioペルソナ優先ルール
- CHAT Delegation Protocol（Worker/Coderへの委任形式）

### B. ブートストラップファイル（`LoadBootstrapFiles()`）

`workspace/` 配下の5つのMarkdownファイルを順に読み込む。ファイルが存在しない場合はスキップされる。

| 読み込み順 | ファイル | 用途 |
|-----------|----------|------|
| 1 | `CHAT_PERSONA.md` | チャットペルソナ（Mioキャラクター設定） |
| 2 | `AGENTS.md` | エージェントの行動指針・ガイドライン |
| 3 | `SOUL.md` | コアな性格特性（親切・正直・透明性） |
| 4 | `USER.md` | ユーザー情報（名前・好み・タイムゾーン） |
| 5 | `IDENTITY.md` | システム名・目的・能力・哲学 |

各ファイルは `## {filename}\n\n{content}\n\n` の形式でプロンプトに挿入される。

### C. スキル情報（`BuildSkillsSummary()`）

3つのディレクトリからスキルを検索し、XML形式のサマリをプロンプトに載せる。

**検索優先度:**

```
1. workspace/skills/          ← ワークスペース固有（最優先）
2. ~/.picoclaw/skills/        ← グローバル
3. ./skills/                  ← ビルトイン
```

サマリにはスキル名・説明・所在・ソース種別が含まれる。LLMが必要に応じて `read_file` ツールで `SKILL.md` の全文を読む設計。

### D. メモリ（`GetMemoryContext()`）

`pkg/agent/memory.go` の `MemoryStore` が管理する2種類の記憶。

| 種類 | ファイル | 内容 |
|------|----------|------|
| 長期記憶 | `workspace/memory/MEMORY.md` | セッション横断で保持する重要情報 |
| 日次ノート | `workspace/memory/YYYYMM/YYYYMMDD.md` | 日ごとの記録（直近3日分を収集） |

```
## Long-term Memory
{MEMORY.md の内容}

---

## Recent Daily Notes
{過去3日分の日次ノート（--- 区切り）}
```

### E. セッション履歴（`SessionManager`）

`workspace/sessions/{sessionKey}.json` に永続化された会話履歴。

- **履歴（History）**: 過去のuser/assistant/toolメッセージ配列
- **要約（Summary）**: 履歴が長くなった際にLLMで生成した圧縮テキスト

### F. ツール定義（`buildToolsSection()`）

`ToolRegistry` に登録されたツールのサマリを動的生成する。

**CHATルートではツールセクション非表示**（会話専用のため）。

**登録ツール一覧:**

| ツール名 | 機能 |
|----------|------|
| `read_file` | ファイル読み取り |
| `write_file` | ファイル書き込み |
| `list_dir` | ディレクトリ一覧 |
| `edit_file` | ファイル編集 |
| `append_file` | ファイル追記 |
| `exec` | シェルコマンド実行 |
| `web_search` | Web検索（Brave/DuckDuckGo/Perplexity） |
| `web_fetch` | Webページ取得 |
| `i2c` | I2Cハードウェア制御 |
| `spi` | SPIハードウェア制御 |
| `message` | ユーザーへ直接メッセージ送信 |

## 3. プロンプト構築フロー

### 3.1 全体フロー図

```
ユーザーメッセージ受信
    │
    ▼
Router.Decide()  →  ルート決定（CHAT / CODE / PLAN / ANALYZE / OPS / RESEARCH）
    │
    ▼
runAgentLoop()
    │
    ├─ updateToolContexts()         ← ツールにchannel/chatID設定
    │
    ├─ SessionManager.GetHistory()  ← 会話履歴取得
    ├─ SessionManager.GetSummary()  ← 要約取得
    │
    ├─ ContextBuilder.BuildMessages()
    │   │
    │   ├─ BuildSystemPrompt(route)
    │   │   ├─ getIdentity(route)         ← Identity + Tools
    │   │   ├─ LoadBootstrapFiles()       ← *.md ファイル群
    │   │   ├─ BuildSkillsSummary()       ← スキルXMLサマリ
    │   │   └─ GetMemoryContext()         ← MEMORY.md + 日次ノート
    │   │
    │   ├─ Session Info (channel, chatID)
    │   ├─ Summary (あれば)
    │   └─ buildUserContentWithMedia()    ← 添付ファイル処理
    │
    ▼
LLMへ送信: [System] + [History] + [User Message]
```

### 3.2 最終メッセージ配列の構造

```
┌─────────────────────────────────────────────────┐
│ messages[0]: System Message                     │
│  ├── # picoclaw 🦞                             │
│  │   ├── Current Time                           │
│  │   ├── Runtime                                │
│  │   ├── Workspace                              │
│  │   ├── Available Tools（CHAT以外）            │
│  │   └── Important Rules                        │
│  │                                              │
│  ├── --- separator ---                          │
│  │                                              │
│  ├── ## CHAT_PERSONA.md                         │
│  ├── ## AGENTS.md                               │
│  ├── ## SOUL.md                                 │
│  ├── ## USER.md                                 │
│  ├── ## IDENTITY.md                             │
│  │                                              │
│  ├── --- separator ---                          │
│  │                                              │
│  ├── # Skills（XMLサマリ）                      │
│  │                                              │
│  ├── --- separator ---                          │
│  │                                              │
│  ├── # Memory                                   │
│  │   ├── ## Long-term Memory                    │
│  │   └── ## Recent Daily Notes                  │
│  │                                              │
│  ├── ## Current Session（channel, chatID）      │
│  └── ## Summary of Previous Conversation        │
│                                                 │
├─────────────────────────────────────────────────┤
│ messages[1..N]: History                         │
│  └── 過去のuser / assistant / tool メッセージ   │
│                                                 │
├─────────────────────────────────────────────────┤
│ messages[N+1]: Current User Message             │
│  ├── テキスト本文                               │
│  ├── [ATTACHMENTS]（画像・文書がある場合）      │
│  │   ├── ファイルパス一覧                       │
│  │   ├── テキスト抜粋（最大12KB）               │
│  │   └── [type=image] タグ                      │
│  └── [ATTACHMENT_POLICY]（処理方針）            │
└─────────────────────────────────────────────────┘
```

各セクションは `\n\n---\n\n` で区切られる。

## 4. ルーティングとルート別挙動

### 4.1 ルーティング判定フロー

`Router.Decide()` が以下の優先度でルートを決定する。

```
優先度1: 明示コマンド   (/chat, /code, /plan, /analyze, /ops, /research)
    ↓ マッチしない場合
優先度2: ルールマッチ   (キーワード・パターン検知)
    ↓ マッチしない場合
優先度3: LLMクラシファイア (LLM自身による分類判定、信頼度閾値あり)
    ↓ 信頼度不足の場合
優先度4: フォールバック  (デフォルト: CHAT)
```

### 4.2 ルール判定の検知パターン

| ルート | 検知パターン |
|--------|------------|
| CODE | バッククォート3連、`diff --git`、`Traceback`、ファイル拡張子（`.go`, `.py` 等） |
| OPS | `systemctl`, `journalctl`, `docker`, `ssh`, `kubectl` |
| ANALYZE | `集計`, `傾向`, `統計`, `analyze`, `csv`, `json` |
| RESEARCH | URL、`出典`, `最新`, `比較`, `research` |
| PLAN | `仕様`, `設計`, `構成`, `段取り`, `plan`, `architecture` |

### 4.3 ルート別の差異

| ルート | ツール提供 | LLMモデル | 要約有効 | 備考 |
|--------|-----------|-----------|----------|------|
| CHAT | なし | `routing.llm.chat_model` | **無効** | Mioペルソナ、会話のみ |
| CODE | あり | `routing.llm.coder_model` | 有効 | ファイル操作・コード実行可 |
| PLAN | あり | `routing.llm.worker_model` | 有効 | 計画立案向け |
| ANALYZE | あり | `routing.llm.worker_model` | 有効 | データ分析向け |
| OPS | あり | `routing.llm.worker_model` | 有効 | 運用操作向け |
| RESEARCH | あり | `routing.llm.worker_model` | 有効 | 調査・検索向け |

### 4.4 CHATデリゲーション

CHATルートのLLMが重い作業を検知した場合、以下の形式で委任を宣言できる。

```
DELEGATE: CODE|PLAN|ANALYZE|OPS|RESEARCH
TASK:
{具体的な作業指示}
```

委任フロー:
1. Mio（CHAT）がデリゲート形式を出力
2. 新規セッションでWorker/Coderが作業実行
3. 結果をMioに戻し、最終回答を生成

## 5. 添付ファイル処理

### 5.1 メディア参照

画像ファイル（`.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`）はLLMにメディア参照として渡される。

### 5.2 テキスト抜粋

テキスト系ファイル（`.md`, `.txt`, `.json`, `.yaml`, `.go`, `.py`, `.ts` 等）は最大12,000文字まで抜粋を挿入。

### 5.3 添付ポリシー（`ATTACHMENT_POLICY`）

ユーザーメッセージのキーワードから意図を推定し、処理方針を自動付与する。

| 推定意図 | 検知キーワード | ポリシー内容 |
|----------|---------------|-------------|
| `save_data` | 保存、保管、アーカイブ | データ保存前提で整理 |
| `knowledge_add` | ナレッジ、知識追加 | 構造化してknowledge配下に保存 |
| `image_prompt` | 描画、画像生成、プロンプトにして | 生成プロンプトに変換 |
| `image_analysis` | 分析、解析、見て | 観測事実と推定を分離して説明 |
| `doc_prompt` | プロンプトとして、テンプレート化 | 指示文に整形 |
| `default` | 上記に該当しない | 内容確認のみ実施し、次のアクションを確認 |

## 6. コンテキスト圧縮

### 6.1 通常要約（`maybeSummarize()`）

| トリガー条件 | アクション |
|-------------|-----------|
| 履歴メッセージ数 > 20 | LLMで要約生成 |
| トークン推定値 > コンテキストウィンドウの75% | LLMで要約生成 |

- 直近4件のメッセージは保持
- 10件超の場合は前半・後半を別々に要約し、マージ
- 要約はセッションのSummaryフィールドに保存

### 6.2 緊急圧縮（`forceCompression()`）

LLMからコンテキスト超過エラーが返された場合に発動。

- 履歴の前半50%を即時ドロップ
- `[System: Emergency compression dropped N oldest messages]` を挿入
- 最大2回リトライ

### 6.3 トークン推定

```
推定トークン数 = 文字数（RuneCount） × 2 / 5
```

CJK文字を考慮し、1トークンあたり約2.5文字で計算。
