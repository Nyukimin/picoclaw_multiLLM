## 0. エグゼクティブサマリ

### 0.1 目的

OpenClaw が提供する機能群を基準に、PicoClaw への移植計画を策定する。
単純なコピーではなく、PicoClaw の Clean Architecture（v3.0）と分散実行（v4.0）に適合する形で再設計・実装する。

### 0.2 現状

| 指標 | 値 |
|------|-----|
| OpenClaw 機能数（チートシート基準） | 44 |
| PicoClaw 実装済み | 17 (39%) |
| PicoClaw 独自機能（OpenClaw にない） | 4 |
| 未実装（優先度 A） | 5 |
| 未実装（優先度 B） | 15 |
| 未実装（優先度 C） | 11 |

### 0.3 PicoClaw 独自の強み（移植不要）

- **分散実行 (v4.0)**: SSH Transport による複数マシン並列実行
- **Worker 即時実行**: Coder -> Proposal -> patch 自動適用パイプライン
- **マルチ Coder**: CODE1(DeepSeek) / CODE2(OpenAI) / CODE3(Claude) の使い分け
- **Clean Architecture 4 層分離**: Domain/Application/Infrastructure/Adapter

### 0.4 移植の6フェーズ

| Phase | 内容 | 項目数 | 優先度 |
|-------|------|--------|--------|
| Phase 1 | 上流移植の完了（中断作業） | 4 | A |

| Phase 4 | チャネル拡張 | 3 | B |
| Phase 5 | 自動化拡張 | 3 | A-B |
| Phase 6 | 音声 (TTS) | 1 | C |

### 1.2 アーキテクチャ上の制約

現在の `cmd/picoclaw/main.go` はサブコマンドを持たない単一エントリーポイント。
CLI コマンド追加には **サブコマンドフレームワーク** の導入が必要。

**選択肢**:
- (A) `cobra` ライブラリ: Go 標準のサブコマンドフレームワーク、広く採用
- (B) `os.Args` 手動パース: 依存ゼロ、PicoClaw の軽量思想に合致
- (C) `flag` + 手動サブコマンド: 標準ライブラリのみ

**推奨**: **(B) `os.Args` 手動パース**
- PicoClaw の設計思想（<10MB メモリ、最小依存）に適合
- サブコマンド数は限定的（10 個未満）、cobra は過剰

## 2. 移植対象の分類と判定

### 2.1 判定基準

| 判定 | 意味 | 条件 |
|------|------|------|
| **移植** | OpenClaw 相当の機能を実装 | PicoClaw の運用・利便性に直結 |
| **適応** | PicoClaw 固有の形で再設計 | OpenClaw とは異なるアプローチが適切 |
| **見送り** | 当面実装しない | 優先度が低い、または代替手段あり |

## 5. Phase 3: チャット内コマンド

### 5.1 設計方針

チャット内コマンドは LINE メッセージとして受信し、ルーティング前に `ChatCommandHandler` でインターセプトする。

**配置**:
```
internal/domain/agent/chat_command.go           -- ChatCommand interface, Registry
internal/application/chatcommand/handler.go     -- ChatCommandHandler（ルーティング前処理）
internal/application/chatcommand/handler_test.go
internal/application/chatcommand/commands.go    -- 各コマンドの実装
```

**処理フロー**:
```
LINE Message
  -> MessageOrchestrator.Handle()
  -> ChatCommandHandler.TryHandle(message)
     -> /status  -> StatusCommand.Execute()  -> 直接応答
     -> /stop    -> StopCommand.Execute()    -> コンテキストキャンセル
     -> /compact -> CompactCommand.Execute() -> カットオーバー実行
     -> nil      -> 通常ルーティングへ
```

### 5.2 `/status`

Gateway・LLM・セッションの要約をチャット内に返す。
`picoclaw status` の出力をテキストメッセージとして返却。

### 5.3 `/stop`

処理中のリクエストをキャンセルする。

**実装方針**:
- `context.WithCancel` で各リクエストの Context を管理
- `/stop` 受信時に該当セッションの Context をキャンセル
- 進行中の LLM 呼び出しが中断される

### 5.4 `/compact`

手動でコンテキスト圧縮（カットオーバー）を実行。

**実装方針**:
- ConversationEngine の ThreadSummary 生成をトリガー
- 短期メモリを要約して圧縮
- 「コンテキストを圧縮しました（X messages -> summary）」と応答

### 5.5 `/context [list|detail]`

RecallPack の現在の内容を表示。デバッグ・透明性のため。

**`/context list`**:
```
Current Context:
  System prompt: 1,200 chars
  Recent messages: 8
  Recalled memories: 2
  Thread summary: yes (last: 30m ago)
  Total tokens (est.): ~2,400
```

### 5.6 `/new`

セッションをリセットし、新しい会話を開始。

**実装方針**:
- 現在のセッションをクローズ（要約を保存）
- 新しいセッション ID を発行
- 「新しいセッションを開始しました」と応答

## 6. Phase 4: チャネル拡張

### 6.1 ChannelAdapter 抽象化

現在の LINE 固有実装を抽象化し、複数チャネルを統一的に扱う。

**配置**:
```
internal/adapter/channel.go                    -- ChannelAdapter interface
internal/adapter/line/adapter.go               -- LINEChannelAdapter（既存handler改修）
internal/adapter/telegram/adapter.go           -- TelegramChannelAdapter
internal/adapter/discord/adapter.go            -- DiscordChannelAdapter
internal/adapter/slack/adapter.go              -- SlackChannelAdapter
```

**ChannelAdapter インターフェース**:

```go
// internal/adapter/channel.go
package adapter

import "context"

// ChannelAdapter はチャットプラットフォームとの統合を抽象化する
type ChannelAdapter interface {
    // Name はチャネル名を返す（"line", "telegram", "discord", "slack"）
    Name() string

    // Start はチャネルのリスナーを起動する
    Start(ctx context.Context) error

    // Stop はチャネルのリスナーを停止する
    Stop() error

    // SendMessage はチャネルにメッセージを送信する
    SendMessage(ctx context.Context, chatID string, message string) error
}
```

**config.yaml 拡張**:

```yaml
channels:
  line:
    enabled: true
    channel_secret: "${LINE_CHANNEL_SECRET}"
    access_token: "${LINE_CHANNEL_TOKEN}"
  telegram:
    enabled: false
    bot_token: "${TELEGRAM_BOT_TOKEN}"
  discord:
    enabled: false
    bot_token: "${DISCORD_BOT_TOKEN}"
  slack:
    enabled: false
    bot_token: "${SLACK_BOT_TOKEN}"
    app_token: "${SLACK_APP_TOKEN}"
```

### 6.2 Telegram

- Bot API (`github.com/go-telegram-bot-api/telegram-bot-api/v5`) を使用
- Long polling（Webhook 不要で手軽）
- テキスト + 画像メッセージ対応

### 6.3 Discord

- `github.com/bwmarrin/discordgo` を使用
- WebSocket Gateway で常時接続
- テキスト + 画像メッセージ対応
- スレッド対応（セッション管理と連携）

### 6.4 Slack

- Socket Mode（Webhook 不要）
- `github.com/slack-go/slack` を使用
- テキスト + Block Kit 対応
- スレッド対応

**目的**: 定期実行タスクのスケジューリング。ハートビートとは独立した汎用 cron。

**配置**:
```
internal/domain/cron/schedule.go              -- Schedule, Job 型定義
internal/application/cron/service.go          -- CronService（スケジュール管理 + 実行）
internal/application/cron/service_test.go
```

**config.yaml**:

```yaml
cron:
  enabled: true
  jobs:
    - name: daily_summary
      schedule: "0 23 * * *"        # 毎日23:00
      action: chat
      message: "今日の活動をまとめて"
    - name: weekly_review
      schedule: "0 10 * * 1"        # 毎週月曜10:00
      action: chat
      message: "今週の予定を確認して"
```

**実装方針**:
- `time.Ticker` ベースの軽量スケジューラ（cron ライブラリ不使用）
- cron 式のパース: 分・時・日・月・曜日の 5 フィールド
- ジョブ実行時は Orchestrator 経由でメッセージを処理

## 9. Clean Architecture 配置ルール

### 9.1 層別ルール

| 層 | ディレクトリ | 依存できる先 | 禁止 |
|----|-------------|-------------|------|
| Domain | `internal/domain/` | なし（純粋 Go） | 外部パッケージ import 禁止 |
| Application | `internal/application/` | Domain | Infrastructure 直接参照禁止 |
| Infrastructure | `internal/infrastructure/` | Domain | Application 参照禁止 |
| Adapter | `internal/adapter/` | Application, Domain | Infrastructure 直接参照禁止 |
| main | `cmd/picoclaw/` | 全層 | DI のみ |

### 9.2 新規パッケージの配置指針

```
機能を追加するとき:
  1. インターフェースは Domain 層に定義
  2. 実装は Infrastructure 層に配置
  3. ユースケース（複数コンポーネントの協調）は Application 層
  4. 外部入出力（HTTP, CLI, Webhook）は Adapter 層
  5. DI（組み立て）は cmd/picoclaw/
```
