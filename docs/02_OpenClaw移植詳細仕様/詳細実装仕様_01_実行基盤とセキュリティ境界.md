### 0.1 OpenClaw から読み取る設計原則

1. 実行は「会話」ではなく「アクション単位」で監査可能であること。  
2. ツール実行はポリシー境界（許可・確認・拒否）で制御すること。  
3. Gateway は複数チャネル/ノードからの要求を同一の安全モデルで裁くこと。  
4. CLI は運用時の観測・診断・復旧の標準導線を持つこと。

## 1. スコープ

### 1.1 この詳細仕様で実装する範囲

- Tool 実行ポリシーエンジン（allow/ask/deny）
- Approval フロー（自動/手動承認の分岐）
- 実行サンドボックス境界（workspace 制限、破壊操作制限）
- 実行監査ログ（request -> decision -> execution -> result）
- 運用 CLI 最小セット（`status`, `health`, `doctor`）

### 1.2 この仕様では扱わない範囲

- TTS/音声生成本体
- Telegram/Discord/Slack などチャネル追加
- Subagent の実行戦略最適化

---

## 2. 要件定義

### 2.1 機能要件

1. すべてのツール呼び出し前に `PolicyDecision` を必ず生成する。  
2. `PolicyDecision=ask` の場合、承認が得られるまで実行してはならない。  
3. `PolicyDecision=deny` の場合、代替手段を返し、再試行はポリシー更新後のみ許可。  
4. 実行開始時点で `ExecutionRecord` を永続化し、失敗時も必ず終了イベントを残す。  
5. CLI で現在の安全状態（承認待ち件数、deny件数、実行中件数）を取得できる。

### 2.2 非機能要件

1. 判定遅延: Policy 判定 p95 < 30ms（ローカル条件）  
2. 監査完全性: 実行イベント欠損率 0%（起動中）  
3. 冪等性: 同一 `job_id + action_id` 再送で二重実行しない  
4. 可観測性: すべての実行に `trace_id` を付与

## 3. ドメインモデル

### 3.1 値オブジェクト

- `PolicyDecision`
  - `decision`: `allow | ask | deny`
  - `reason`: string
  - `matched_rule_id`: string
- `ExecutionAction`
  - `job_id`, `action_id`, `tool`, `arguments`, `requested_by`
- `ExecutionRecord`
  - `status`: `pending | waiting_approval | running | succeeded | failed | denied`
  - `started_at`, `finished_at`, `error`, `trace_id`

### 3.2 集約

- `ExecutionAggregate`
  - 不変条件:
    1. `deny` 状態から `running` に遷移不可
    2. `waiting_approval` は `approve` または `cancel` 以外へ遷移不可
    3. `succeeded/failed/denied` は終端状態

---

## 4. アプリケーション層ユースケース

### 4.1 `RequestToolExecution` ユースケース

1. `ExecutionAction` を受領  
2. `PolicyEngine.Evaluate(action)` を実行  
3. 判定に応じて分岐
   - `allow`: `ToolRunner.Run()`
   - `ask`: 承認キューへ投入、`waiting_approval`
   - `deny`: 終端 `denied`
4. `ExecutionRecord` を保存

### 4.2 `ApproveExecution` ユースケース

1. 承認キューから対象 action を取得  
2. TTL を超過していないことを検証（既定: 10 分）  
3. `ToolRunner.Run()` 実行、結果保存

---

## 5. Infrastructure 実装方針

### 5.1 Policy Engine

- 配置: `internal/infrastructure/security/policy_engine.go`
- 入力: `ExecutionAction`, `SecurityConfig`
- ルール優先順位:
  1. 強制 deny（破壊コマンド、workspace 外書き込み）
  2. 強制 ask（ネットワーク外部送信、機密ファイル参照）
  3. allow（安全ツール）

### 5.2 Sandbox Guard

- 配置: `internal/infrastructure/security/sandbox_guard.go`
- ガード内容:
  - path canonicalize + workspace prefix check
  - 禁止コマンドシグネチャ検査（`rm -rf`, `git reset --hard` など）
  - 実行タイムアウト上限適用

### 5.3 Audit Store

- 配置: `internal/infrastructure/persistence/execution/`
- 初期実装: JSONL append + ローテーション
- 後続拡張: SQLite backend 差し替え可能な `ExecutionRepository` interface

## 6. Adapter/CLI 仕様

### 6.1 追加コマンド

- `picoclaw status`
  - 実行中ジョブ数、承認待ち件数、直近失敗数を表示
- `picoclaw health --json`
  - policy engine / audit store / runner のヘルスを JSON 返却
- `picoclaw doctor`
  - 設定矛盾（例: 承認必須なのに承認経路未設定）を検出

### 6.2 出力契約

- テキスト出力は人間向け、`--json` は機械向け（フィールド固定）
- エラーは必ず `code` と `hint` を持つ

---

## 7. 設定仕様

`config.yaml` に以下を追加。

```yaml
security:
  policy_mode: strict          # strict | balanced
  approval_ttl_minutes: 10
  deny_commands:
    - "rm -rf"
    - "git reset --hard"
  workspace_enforced: true
  audit:
    enabled: true
    backend: jsonl
    path: "logs/execution_audit.jsonl"
```

## 8. TDD 実装計画（必須）

### 8.1 先に書くテスト

1. Domain: 状態遷移不変条件テスト  
2. Policy: `allow/ask/deny` 判定テスト  
3. Sandbox: workspace 外書き込み拒否テスト  
4. UseCase: 承認待ち -> 承認 -> 実行成功の遷移テスト  
5. CLI: `status/health/doctor` の契約テスト

### 8.2 受け入れ基準

- 単体テスト成功率 100%（対象パッケージ）
- 追加テストの失敗再現 -> 実装修正 -> 再成功の順序をCIログで確認できる
- E2Eで `deny` された操作が実行されないことを証明できる
