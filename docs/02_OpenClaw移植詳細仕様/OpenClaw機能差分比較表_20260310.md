## 比較表

| 領域 | OpenClaw | RenCrow現状 | 差分評価 |
|---|---|---|---|
| Unified Entry | 複数Platformを単一入口で処理 | `/entry` + `platform/channel`正規化実装済み | △ 入口は整備済み、運用粒度が未統一 |
| Platform同期 | Chat UI/CLI/連携アプリを同一進行モデルで同期 | `entry.stage` 発火、Viewer + Chrome bridge status/eventsあり | △ 通知粒度設定・冪等化が未完 |
| SSE再接続 | 再接続時の連続性を担保 | `seq` + `Last-Event-ID` 対応（viewer/chrome） | ◯ 主要要件は実装済み |
| Ops CLI | status/health/doctor/logs/gateway/channelsを統一運用 | `--json` 契約含め主要コマンド実装済み | △ restart運用方針など未決事項あり |
| Security Profile | strict系中心の明示権限モデル | strict/balanced/dev + sandbox_level実装 | ◯ モデル層は実装済み |
| Network制御 | allowlist/blocked等の実行時制御 | `network_scope` + `network_allowlist` 実装済み | △ container実行基盤は未統合 |
| 危険操作拒否 | 破壊操作を強制deny | `rm -rf`, `git reset --hard`, workspace外書込を拒否 | ◯ 実装済み |
| 監査イベント | decision/violationを実行単位で記録 | `security.decision` / `security.violation` 実装済み | ◯ 実装済み |
| Approvalモード | never/on-demand/always | 実装済み（TTL付き） | ◯ 実装済み |
| Contract Layer | 依頼→実行契約へ正規化 | `application/contract` 実装済み | △ reject再生成の深掘りは継続 |
| Autonomous Executor | Plan→Apply→Verify→Repair→Verify | 最小ループ実装、`/entry`経路で適用済み | △ 失敗分類テンプレート強化余地 |
| Evidence Layer | 実行証跡の保存・可視化 | `execution_report` 保存 + Viewer recent/detail/summary + CLI evidence | ◯ 実装済み |
| Tools体系 | Manifest/Registry/ExecutionEnvelopeを第一級化 | Manifest/Registryは着手済み、体系は分散気味 | △ 統一整理が必要 |
| Nodes/Capability | ノード能力ベース選定 | `Capability` モデル + `NodeSelector` 実装、`RouteCODE` で能力選定適用済み | △ 全ルート/全ノードへの適用と動的能力更新は未完 |
| チャネル網羅 | 複数チャネル同一モデル運用 | LINE/Telegram/Discord/Slack基盤あり | △ 署名検証・再送/重複対策の統一を要強化 |
| TTS実装能力 | 指示で導入完了まで持っていく能力 | `/entry` 経路で Plan→Apply→Verify→Repair + synth/playback + evidence 記録を実装済み | △ `sbv2` 中心。Azure/Eleven/local の本実装と実機E2E証跡が未完 |

## 差分評価の凡例

- `◯`: 現行要件を概ね満たす
- `△`: 一部実装済み。運用・機能要件に未充足あり
- `❌`: 主要機能が未実装

---

## 優先対応（推奨）

1. `TTS実装能力` のProvider拡張（Azure/Eleven/local）と実機E2E証跡の確立
2. `Nodes/Capability` の適用範囲拡張（CODE以外・動的能力更新）
3. `Platform同期` の冪等化（重複 `request_id` の一意制御）
4. `Tools体系` の統一（ExecutionEnvelopeと監査一体化）
