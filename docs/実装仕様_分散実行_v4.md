## 0. エグゼクティブサマリ

- v3.0（Clean Architecture）で単一プロセス内の実装が完成
- `docs/Chat_Worker_Coder_アーキテクチャ.md` で分散実行の理想像が定義
- しかし、**実装可能な技術仕様が不足**し、分散実行機能は未実装

**v4.0の目的**:
- v3.0の上に**分散実行機能を追加**
- Agent間通信をSSH + JSON化
- 複数CPU/マシンでAgentを並列実行
- **v3.0との完全な互換性**を維持（設定フラグで切り替え）

## 1. 背景と目的

### 1.1 現状の課題

#### 1.1.1 単一プロセスでのボトルネック

**v3.0の制約**:
- すべてのAgent（Mio、Shiro、Aka、Ao、Gin）が同一プロセス内で動作
- LLM呼び出し（重い処理）が順次実行され、並列化不可能
- Ollama常駐による単一マシンへの負荷集中

**具体的な問題**:
```
ユーザー: 「3つのファイルを同時に作成して」
  ↓
Mio → Shiro → Gin（ファイル1作成）→ 完了
            → Gin（ファイル2作成）→ 完了  ← 順次実行
            → Gin（ファイル3作成）→ 完了  ← 並列化できない
  ↓
合計時間: 15秒（5秒 × 3ファイル）
```

**v4.0の並列実行（Worker内PatchCommand並列）**:
```
ユーザー: 「3つのファイルを同時に作成して」
  ↓
Mio → Worker → Coder3（Proposal生成: 3ファイル分のPatchCommand）
  ↓
Worker: PatchCommand並列実行
  ├─ [goroutine1] file_edit/create: ファイル1
  ├─ [goroutine2] file_edit/create: ファイル2
  └─ [goroutine3] file_edit/create: ファイル3
  ↓
合計時間: Coder3生成5秒 + ファイル作成（並列）1秒 = 6秒
v3.0比: Coder3生成5秒 + ファイル作成（直列）3秒 = 8秒 → 25%改善
```

**v4.1の並列実行（タスク分割、将来）**:
```
ユーザー: 「3つのサービスを設計して」
  ↓
Mio → タスク分割 → [CPU1] Coder1（サービスA）並列実行
                  → [CPU2] Coder2（サービスB）並列実行
                  → [CPU3] Coder3（サービスC）並列実行
  ↓
合計時間: max(Coder1, Coder2, Coder3) = 5秒（並列実行）
```

#### 1.1.2 スケーラビリティの限界

| 項目 | v3.0の制約 | 影響 |
|------|----------|------|
| **メモリ** | 単一プロセスで全Agentを管理 | LLM呼び出しの応答バッファが集中 |
| **CPU** | LLM呼び出しがシリアル実行 | CPU使用率 70%でボトルネック |
| **ネットワーク** | Ollama接続が単一マシンに集中 | ネットワーク帯域がボトルネック |
| **拡張性** | マシン追加してもスケールしない | 性能向上の上限 |

#### 1.1.3 保守性の課題

- Agent単位のアップデート不可能（全体を再起動）
- 障害時の影響範囲が全Agent（1つが落ちると全停止）
- ログが単一プロセスに集約され、デバッグ困難

```
  CPU2: (未使用)
```

**v4.0スコープ**: Transport層の並列化 + Worker内PatchCommand並列実行

```
[v4.0] Worker内PatchCommand並列実行:

タスク: 「hello.goを作成してテストも実行して」
  ↓
Mio → Worker → Coder3（Proposal生成）→ patch返却
  ↓
Worker: PatchCommand並列実行
  ├─ [goroutine1] file_edit: hello.go作成
  ├─ [goroutine2] file_edit: hello_test.go作成
  └─ 依存関係のあるコマンドは順次実行（shell_command: go test）
  ↓
合計時間: ファイル作成は並列、テスト実行は直列
```

**v4.1スコープ（将来）**: タスク分割による複数Coder並列実行

```

  ↓
Mio → タスク分割判定 → [並列実行]
               ├─ [CPU1] Aka: サービスAの設計
               ├─ [CPU2] Ao: サービスBの実装
               └─ [CPU3] Gin: サービスCの実装
  ↓
合計時間: max(Aka, Ao, Gin) = 10秒
v3.0では: Aka + Ao + Gin = 30秒
```

**注意**: v4.0では1タスク=1Coderの原則を維持する。タスク分割ロジック（どこで分割するか、分割結果の統合方法）はv4.1で設計する。

#### 1.2.2.1 Type-Based Phased Execution Model（v4.0）

Worker内PatchCommand並列実行の依存関係は、**コマンドType別の3フェーズ実行モデル**で管理する。

```
Phase 1: file_edit（並列実行）
  ├─ [goroutine] create hello.go
  ├─ [goroutine] create hello_test.go
  ├─ [goroutine] mkdir internal/pkg
  └─ sync.WaitGroup.Wait()  ← 全file_edit完了を待機
         ↓
Phase 2: shell_command（順次実行）
  ├─ run "go fmt ./..."
  └─ run "go test ./..."
         ↓
Phase 3: git_operation（順次実行）
  ├─ add -A
  └─ commit -m "..."
```

**設計原則**:

1. **Phase 1（file_edit）**: 全`file_edit`コマンドをgoroutineで並列実行。ファイルI/Oは独立しているため安全に並列化できる。ただし同一ファイルへの複数操作（create後にappend等）がある場合は、**同一Targetのコマンドは出現順に直列化**する。
2. **Phase 2（shell_command）**: `shell_command`は副作用が予測不可能（プロセス起動、ファイル読み書き等）のため、**常に順次実行**する。Phase 1の全完了後に開始。
3. **Phase 3（git_operation）**: `git_operation`は排他的操作（`.git/index` ロック）のため、**常に順次実行**する。Phase 2の全完了後に開始。

**フェーズ分割の実装**:

```go
// partitionByPhase はPatchCommandをフェーズ別に分割する
func partitionByPhase(commands []patch.PatchCommand) (
	fileEdits []patch.PatchCommand,
	shellCmds []patch.PatchCommand,
	gitOps []patch.PatchCommand,
) {
	for _, cmd := range commands {
		switch cmd.Type {
		case patch.TypeFileEdit:
			fileEdits = append(fileEdits, cmd)
		case patch.TypeShellCommand:
			shellCmds = append(shellCmds, cmd)
		case patch.TypeGitOperation:
			gitOps = append(gitOps, cmd)
		}
	}
	return
}

// groupByTarget は同一Targetのfile_editをグループ化する
// 同一ファイルへの複数操作（create→append等）を順序保証する
func groupByTarget(fileEdits []patch.PatchCommand) map[string][]patch.PatchCommand {
	groups := make(map[string][]patch.PatchCommand)
	for _, cmd := range fileEdits {
		groups[cmd.Target] = append(groups[cmd.Target], cmd)
	}
	return groups
}
```

**StopOnErrorとの相互作用**:
- `StopOnError=true`の場合: Phase 1で1つでもエラーが発生したら、他のgoroutineの完了を待った後、Phase 2/3をスキップする
- `StopOnError=false`の場合: 全Phaseを実行し、エラーを集約して報告する

**AutoCommitとの相互作用**:
- Git auto-commit（実行前）は Phase 1 開始前に実行（v3と同じ）
- Git auto-commit（実行後）は Phase 3 完了後に実行（v3と同じ）
- Phase 3 に明示的なgit_operationがある場合、auto-commitとの競合を防ぐため、auto-commitは Phase 3 の**後**に実行する

### 1.3 設計原則（不変）

**v3.0からの継承**:
1. **Clean Architecture 4層構造**: Domain → Application → Infrastructure → Adapter
2. **責務分離**: Mio（全体）、Shiro（Coderオーケストレーター）、Coder（実装）
3. **日本語会話**: Agent間通信は日本語メッセージ（JSONパラメータ）
4. **DDD**: エンティティ、値オブジェクト、リポジトリパターン

**v4.0の追加設計原則**:
1. **Transport層抽象化**: Local/Remote透過的切り替え
2. **疎結合**: Agent間依存は通信インターフェースのみ
3. **v3.0互換性**: 設定フラグで既存動作を維持
4. **段階的移行**: Phase 1-3で段階的に機能追加

### 1.4 Agent名称の統一定義

ドキュメント・設定・コードで使用するAgent名称を以下のように統一定義する。

| Transport ID（通信用） | 設定キー（config.yaml） | ニックネーム（ドキュメント用） | v3コード上の変数名 | 役割 |
|------------------------|------------------------|-------------------------------|-------------------|------|
| `"Mio"` | `mio` | Mio | `mio` | Chat/ルーティング |
| `"Worker"` | `worker` | Shiro | `shiro` | 実行/オーケストレーション |
| `"Coder1"` | `coder1` | Aka | `coder1` | 仕様設計（DeepSeek） |
| `"Coder2"` | `coder2` | Ao | `coder2` | 実装（OpenAI） |
| `"Coder3"` | `coder3` | Gin | `coder3` | 高品質コーディング（Claude） |

**使い分けルール**:
- **`Message.From` / `Message.To`**: Transport ID を使用（例: `"Worker"`, `"Coder3"`）
- **`config.yaml` の `distributed.transports`**: 設定キーを使用（例: `worker`, `coder3`）
- **ドキュメント本文・アーキテクチャ図**: ニックネーム使用可だが、初出時にTransport IDを併記

**注意**: v3コードの `shiro` 変数名は変更しない（後方互換性維持）。Transport IDのみ `"Worker"` を正とする。

## 2. アーキテクチャ概要

### 2.1 v3.0（現在）の構成

```
[LINE/Slack] → [Adapter] → [Application (Orchestrator)]
                              ↓（直接関数呼び出し）
                            [Domain (Mio/Shiro/Coder)]
                              ↓
                            [Infrastructure (LLM Provider)]
```

**特徴**:
- すべてのAgentが同一プロセス内で動作
- Agent間通信はGo関数呼び出し（`mio.DecideAction()`, `shiro.Execute()`等）
- ローカル変数としてTask、Proposalを受け渡し

### 2.2 v4.0（実装済み）の構成

```
[fujitsu-ubunts: Ubuntu]
  [LINE] → [Adapter] → [Orchestrator]
                          ↓（localhost 関数呼び出し）
                        [Mio]  ←→ [Shiro (Worker)]
```

#### 2.4.1 Orchestrator移行設計

**課題**: v3の`MessageOrchestrator`は同期的な関数呼び出し（`MioAgent.DecideAction()`, `ShiroAgent.Execute()`, `CoderAgent.Generate()`）で設計されている。v4のTransport層は非同期メッセージング（`Send()`/`Receive()`）であり、設計パラダイムが根本的に異なる。

**移行戦略**: Adapter パターンによる段階的移行

```
[v3 MessageOrchestrator]
  └─ MioAgent.DecideAction(ctx, task) → (Decision, error)   ← 同期関数呼び出し
  └─ ShiroAgent.Execute(ctx, task) → (string, error)         ← 同期関数呼び出し
  └─ CoderAgent.Generate(ctx, task, prompt) → (string, error) ← 同期関数呼び出し

[v4 DistributedOrchestrator]
  └─ Transport.Send(ctx, msg) → error                        ← 非同期メッセージング
  └─ Transport.Receive(ctx) → (Message, error)               ← ブロッキング受信
```

**実装方針**:

1. **v3 MessageOrchestrator は維持**（`distributed.enabled: false` 時に使用）
2. **v4 DistributedOrchestrator を新規作成**（`distributed.enabled: true` 時に使用）
3. **共通インターフェースで切り替え**:

```go
// Orchestrator は v3/v4 共通のインターフェース
type Orchestrator interface {
    ProcessMessage(ctx context.Context, req ProcessMessageRequest) (ProcessMessageResponse, error)
}

// v3: 同期関数呼び出し（既存）
type MessageOrchestrator struct { /* ... */ }

// v4: Transport層経由の非同期メッセージング（新規）
type DistributedOrchestrator struct {
    sessionRepo     SessionRepository
    router          *MessageRouter      // Transport層のルーター
    responseTimeout time.Duration       // レスポンスタイムアウト
}

// v4のProcessMessage: Send → Receive パターン
func (o *DistributedOrchestrator) ProcessMessage(ctx context.Context, req ProcessMessageRequest) (ProcessMessageResponse, error) {
    // 1. セッションをロードまたは作成
    sess, err := o.loadOrCreateSession(ctx, req.SessionID, req.Channel, req.ChatID)
    if err != nil {
        return ProcessMessageResponse{}, err
    }

    // 2. タスクメッセージを構築
    jobID := task.NewJobID()
    msg := transport.Message{
        From:      "Mio",
        To:        "Mio",  // 自分自身（ルーティング判定用）
        SessionID: sess.ID(),
        JobID:     jobID.String(),
        Message:   req.UserMessage,
    }

    // 3. Mio（ルーティング）→ Worker/Coder → 結果受信
    //    Transport層がメッセージを中継し、結果がReceive()で返る
    resultMsg, err := o.sendAndWaitForResult(ctx, msg)
    if err != nil {
        return ProcessMessageResponse{}, err
    }

    // 4. 結果をレスポンスに変換
    return o.buildResponse(resultMsg, jobID), nil
}
```

#### 2.4.2 Worker並列実行の実装設計

**課題**: v3の`workerExecutionService.ExecuteProposal()`はPatchCommandを`for`ループで順次実行している（`worker_execution_service.go:63-72`）。v4.0ではType-Based Phased Execution Model（§1.2.2.1）に基づく並列実行が必要。

**移行戦略**: 設定フラグによる切り替え

```go
// WorkerConfig に並列実行設定を追加
type WorkerConfig struct {
    // ... v3既存フィールド（AutoCommit, StopOnError, etc.）...

    // v4.0 追加
    ParallelExecution bool `yaml:"parallel_execution"` // true で並列実行モード
    MaxParallelism    int  `yaml:"max_parallelism"`     // file_edit並列数上限（デフォルト: 4）
}
```

**実装**: `worker_execution_service.go` に追加

```go
import (
    "sync"
    // ... 既存インポート ...
)

// ExecuteProposal は Proposal の Patch を解析・実行する（v4.0拡張版）
func (w *workerExecutionService) ExecuteProposal(
    ctx context.Context,
    jobID task.JobID,
    p *proposal.Proposal,
) (*patch.PatchExecutionResult, error) {
    // 1. Patchをパース
    commands, err := patch.ParsePatch(p.Patch())
    if err != nil {
        return nil, fmt.Errorf("patch parse error: %w", err)
    }

    // 2. 実行前サマリ表示
    if w.config.ShowExecutionSummary {
        w.showExecutionSummary(jobID, commands)
    }

    // 3. Git auto-commit（実行前）
    if w.config.AutoCommit {
        preCommitHash, err := w.autoCommitChanges(ctx, jobID, "Before patch execution")
        if err != nil {
            return nil, fmt.Errorf("pre-execution auto-commit failed: %w", err)
        }
        fmt.Printf("[Worker] Pre-commit succeeded: %s\n", preCommitHash)
    }

    // 4. 実行（並列 or 順次）
    var result *patch.PatchExecutionResult
    if w.config.ParallelExecution {
        result = w.executeParallel(ctx, jobID, commands)
    } else {
        result = w.executeSequential(ctx, jobID, commands)
    }

    // 5. Git auto-commit（実行後）
    if w.config.AutoCommit && result.ExecutedCmds > 0 {
        postCommitHash, err := w.autoCommitChanges(ctx, jobID,
            fmt.Sprintf("Patch execution: %d commands", result.ExecutedCmds))
        if err == nil {
            result = result.WithGitCommit(postCommitHash)
        }
    }

    // 6. サマリ生成
    summary := fmt.Sprintf("実行: %d 件, 成功: %d 件, 失敗: %d 件",
        len(commands), result.ExecutedCmds, result.FailedCmds)
    result = result.WithSummary(summary)

    return result, nil
}

// executeSequential は v3互換の順次実行（既存ロジック）
func (w *workerExecutionService) executeSequential(
    ctx context.Context,
    jobID task.JobID,
    commands []patch.PatchCommand,
) *patch.PatchExecutionResult {
    result := patch.NewPatchExecutionResult()
    for i, cmd := range commands {
        cmdResult := w.executeCommand(ctx, jobID, cmd, i)
        result.AddResult(cmdResult)
        if !cmdResult.Success && w.config.StopOnError {
            break
        }
    }
    return result
}

// executeParallel は Type-Based Phased Execution（v4.0新規）
func (w *workerExecutionService) executeParallel(
    ctx context.Context,
    jobID task.JobID,
    commands []patch.PatchCommand,
) *patch.PatchExecutionResult {
    result := patch.NewPatchExecutionResult()

    // Phase分割
    fileEdits, shellCmds, gitOps := partitionByPhase(commands)

    // === Phase 1: file_edit（並列実行）===
    if len(fileEdits) > 0 {
        phase1Results := w.executeFileEditsParallel(ctx, jobID, fileEdits)
        for _, r := range phase1Results {
            result.AddResult(r)
        }

        // StopOnError: Phase 1でエラー → Phase 2/3をスキップ
        if w.config.StopOnError && result.HasFailures() {
            return result
        }
    }

    // === Phase 2: shell_command（順次実行）===
    for i, cmd := range shellCmds {
        cmdResult := w.executeCommand(ctx, jobID, cmd, len(fileEdits)+i)
        result.AddResult(cmdResult)
        if !cmdResult.Success && w.config.StopOnError {
            return result
        }
    }

    // === Phase 3: git_operation（順次実行）===
    for i, cmd := range gitOps {
        cmdResult := w.executeCommand(ctx, jobID, cmd, len(fileEdits)+len(shellCmds)+i)
        result.AddResult(cmdResult)
        if !cmdResult.Success && w.config.StopOnError {
            return result
        }
    }

    return result
}

// executeFileEditsParallel はfile_editコマンドを並列実行する
// 同一Targetへの複数操作は出現順に直列化する
func (w *workerExecutionService) executeFileEditsParallel(
    ctx context.Context,
    jobID task.JobID,
    fileEdits []patch.PatchCommand,
) []patch.CommandResult {
    // 同一Targetのコマンドをグループ化
    groups := groupByTarget(fileEdits)

    // 結果を格納するスライス（順序保持のためインデックスマップを使用）
    allResults := make([]patch.CommandResult, len(fileEdits))
    indexMap := make(map[string][]int) // target → 元のインデックス配列

    idx := 0
    for target, cmds := range groups {
        indices := make([]int, len(cmds))
        for i := range cmds {
            indices[i] = idx
            idx++
        }
        indexMap[target] = indices
    }

    // 並列度制限付きgoroutine実行
    maxParallel := w.config.MaxParallelism
    if maxParallel <= 0 {
        maxParallel = 4 // デフォルト
    }
    sem := make(chan struct{}, maxParallel)

    var wg sync.WaitGroup

    for target, cmds := range groups {
        indices := indexMap[target]

        wg.Add(1)
        sem <- struct{}{} // セマフォ取得

        go func(target string, cmds []patch.PatchCommand, indices []int) {
            defer wg.Done()
            defer func() { <-sem }() // セマフォ解放

            // 同一Target内は順次実行（create→append等の順序保証）
            for i, cmd := range cmds {
                cmdResult := w.executeCommand(ctx, jobID, cmd, indices[i])
                allResults[indices[i]] = cmdResult

                // 同一Target内でエラー → 残りをスキップ
                if !cmdResult.Success && w.config.StopOnError {
                    break
                }
            }
        }(target, cmds, indices)
    }

    wg.Wait()
    return allResults
}

// partitionByPhase はPatchCommandをType別に3フェーズに分割する
func partitionByPhase(commands []patch.PatchCommand) (
    fileEdits []patch.PatchCommand,
    shellCmds []patch.PatchCommand,
    gitOps []patch.PatchCommand,
) {
    for _, cmd := range commands {
        switch cmd.Type {
        case patch.TypeFileEdit:
            fileEdits = append(fileEdits, cmd)
        case patch.TypeShellCommand:
            shellCmds = append(shellCmds, cmd)
        case patch.TypeGitOperation:
            gitOps = append(gitOps, cmd)
        }
    }
    return
}

// groupByTarget は同一Targetのfile_editをグループ化する
func groupByTarget(fileEdits []patch.PatchCommand) map[string][]patch.PatchCommand {
    groups := make(map[string][]patch.PatchCommand)
    for _, cmd := range fileEdits {
        groups[cmd.Target] = append(groups[cmd.Target], cmd)
    }
    return groups
}
```

**実装**:

```go
package transport

import (
	"context"
	"time"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/proposal"
	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/patch"

// Message は Agent間の通信メッセージ
type Message struct {
	From      string                      `json:"from"`       // 送信元Agent（"Mio", "Worker", "Coder3"）
	To        string                      `json:"to"`         // 送信先Agent
	SessionID string                      `json:"session_id"` // セッション識別子
	JobID     string                      `json:"job_id"`     // Job識別子
	Message   string                      `json:"message"`    // 日本語メッセージ
	Context   map[string]interface{}      `json:"context"`    // コンテキスト（履歴、パラメータ等）
	Proposal  *ProposalPayload            `json:"proposal,omitempty"` // Proposal（任意）
	Result    *ResultPayload              `json:"result,omitempty"`   // 実行結果（任意）
	Timestamp string                      `json:"timestamp"`  // タイムスタンプ（RFC3339）
}

// ProposalPayload は Proposal のペイロード
type ProposalPayload struct {
	Plan     string          `json:"plan"`
	Patch    json.RawMessage `json:"patch"`     // PatchCommand配列のJSON（下記参照）
	Risk     string          `json:"risk"`
	CostHint string          `json:"costHint"`
}

// Patchフィールドのシリアライゼーション契約:
//
// v3の Proposal.Patch() は LLM が生成した文字列（JSON配列またはMarkdown）を返す。
// v4の ProposalPayload.Patch は json.RawMessage 型を使用する。
//
// json.RawMessage の利点:
// - Transport層のJSON エンコード時に二重エンコーディングが発生しない
//   （string型だと {"patch": "[{...}]"} のようにエスケープされる）
// - json.RawMessage は {"patch": [{...}]} のようにそのまま埋め込まれる
// - 受信側は json.Unmarshal(payload.Patch, &commands) でPatchCommand配列に変換可能
//
// 変換例:
//   // Coder側: v3 Proposal → v4 ProposalPayload
//   proposalPayload := &ProposalPayload{
//       Plan:     p.Plan(),
//       Patch:    json.RawMessage(p.Patch()), // 文字列→RawMessage（コピーのみ）
//       Risk:     p.Risk(),
//       CostHint: p.CostHint(),
//   }
//
//   // Worker側: v4 ProposalPayload → v3 PatchCommand配列
//   var commands []patch.PatchCommand
//   if err := json.Unmarshal(proposalPayload.Patch, &commands); err != nil {
//       // Markdown形式等の場合は patch.ParsePatch(string(proposalPayload.Patch)) を使用
//   }

// ResultPayload は 実行結果のペイロード
// v3 PatchExecutionResult との対応: Status↔Success, ExecutedCmds↔ExecutedCmds,
// FailedCmds↔FailedCmds, CommandResults↔Results, GitCommit↔GitCommit
type ResultPayload struct {
	Status         string           `json:"status"`                    // "success", "partial_success", "failed"
	ExecutedCmds   int              `json:"executed_cmds"`             // 実行したコマンド数
	FailedCmds     int              `json:"failed_cmds"`              // 失敗したコマンド数
	CreatedFiles   []string         `json:"created_files,omitempty"`
	ModifiedFiles  []string         `json:"modified_files,omitempty"`
	DeletedFiles   []string         `json:"deleted_files,omitempty"`
	GitCommit      string           `json:"git_commit,omitempty"`
	ElapsedTime    string           `json:"elapsed_time"`
	ErrorMessage   string           `json:"error_message,omitempty"`
	ErrorCode      string           `json:"error_code,omitempty"`      // エラーコード（§8.2.3参照）
	CommandResults []CommandResultPayload `json:"command_results,omitempty"` // 各コマンドの実行結果
}

// CommandResultPayload は個別コマンドの実行結果
// v3 patch.CommandResult との対応
type CommandResultPayload struct {
	Type    string `json:"type"`    // "file_edit", "shell_command", "git_operation"
	Action  string `json:"action"`  // "create", "update", "delete", "run", etc.
	Target  string `json:"target"`  // ファイルパスまたはコマンド
	Success bool   `json:"success"`
	Output  string `json:"output,omitempty"`
	Error   string `json:"error,omitempty"`
}

// Transport は Agent間通信の抽象化
type Transport interface {
	// Send はメッセージを送信
	Send(ctx context.Context, msg Message) error

	// Receive はメッセージを受信（ブロッキング）
	Receive(ctx context.Context) (Message, error)

	// Close はTransportを閉じる
	Close() error

	// IsHealthy はTransportの健全性を確認
	IsHealthy() bool
}
```

### 3.2 LocalTransport 実装

**ファイル**: `internal/infrastructure/transport/local.go`

**設計ポイント**:
- チャネルベースの同一プロセス内通信
- スレッドセーフ（doneチャネル + sync.Mutex でデッドロック回避）
- バッファリング（チャネル容量100）

**実装**:
```go
package transport

import (
	"context"
	"errors"
	"sync"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// LocalTransport はローカル（同一プロセス内）通信の実装
//
// ロック戦略:
//   - closedフラグの読み書きのみ sync.Mutex で保護
//   - Send/Receive はロックを保持したままチャネル操作しない（デッドロック防止）
//   - チャネル自体はスレッドセーフなため、ロック不要
//   - Close()はdoneチャネルでSend/Receiveのブロッキングを解除
type LocalTransport struct {
	inbound  chan transport.Message
	outbound chan transport.Message
	done     chan struct{} // Close通知用（selectで使用）
	closed   bool
	mu       sync.Mutex // closedフラグ保護用（RWMutexは不要）
}

// NewLocalTransport は LocalTransport を作成
func NewLocalTransport() *LocalTransport {
	return &LocalTransport{
		inbound:  make(chan transport.Message, 100),
		outbound: make(chan transport.Message, 100),
		done:     make(chan struct{}),
		closed:   false,
	}
}

// Send はメッセージを送信
// ロックを保持したままチャネル操作しないため、デッドロックしない
func (t *LocalTransport) Send(ctx context.Context, msg transport.Message) error {
	select {
	case t.outbound <- msg:
		return nil
	case <-ctx.Done():
		return ctx.Err()
	case <-t.done:
		return errors.New("transport is closed")
	}
}

// Receive はメッセージを受信（ブロッキング）
// Close()が呼ばれるとdoneチャネル経由で即座にアンブロックされる
func (t *LocalTransport) Receive(ctx context.Context) (transport.Message, error) {
	select {
	case msg := <-t.inbound:
		return msg, nil
	case <-ctx.Done():
		return transport.Message{}, ctx.Err()
	case <-t.done:
		return transport.Message{}, errors.New("transport is closed")
	}
}

// Close はTransportを閉じる
// doneチャネルのcloseにより、ブロッキング中のSend/Receiveが即座に解除される
func (t *LocalTransport) Close() error {
	t.mu.Lock()
	defer t.mu.Unlock()

	if t.closed {
		return nil
	}

	t.closed = true
	close(t.done)     // Send/Receiveのブロッキングを解除
	close(t.inbound)  // forwardLoopへの終了通知
	close(t.outbound) // forwardLoopへの終了通知
	return nil
}

// IsHealthy はTransportの健全性を確認
func (t *LocalTransport) IsHealthy() bool {
	t.mu.Lock()
	defer t.mu.Unlock()
	return !t.closed
}

// GetOutboundChannel は outbound チャネルを返す（MessageRouter用）
func (t *LocalTransport) GetOutboundChannel() <-chan transport.Message {
	return t.outbound
}

// PutInboundMessage は inbound チャネルにメッセージを送る（MessageRouter用）
// doneチャネルでClose済みを検知するため、ロック不要
func (t *LocalTransport) PutInboundMessage(msg transport.Message) error {
	select {
	case t.inbound <- msg:
		return nil
	case <-t.done:
		return errors.New("transport is closed")
	}
}
```

### 3.3 MessageRouter 実装

**ファイル**: `internal/infrastructure/transport/router.go`

**設計ポイント**:
- Agent名 → Transport のマッピング管理
- スレッドセーフ（sync.RWMutex）
- 動的なAgent登録/削除
- **Agent毎に専用goroutineでoutboundを監視**（busy-loop回避）

**実装**:
```go
package transport

import (
	"context"
	"fmt"
	"log/slog"
	"sync"

	domaintransport "github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// MessageRouter は Agent間のメッセージをルーティング
type MessageRouter struct {
	transports map[string]*LocalTransport // Agent名 → Transport
	mu         sync.RWMutex
	ctx        context.Context
	cancel     context.CancelFunc
	wg         sync.WaitGroup // goroutine終了待ち
	logger     *slog.Logger
}

// NewMessageRouter は MessageRouter を作成
func NewMessageRouter(logger *slog.Logger) *MessageRouter {
	ctx, cancel := context.WithCancel(context.Background())
	return &MessageRouter{
		transports: make(map[string]*LocalTransport),
		ctx:        ctx,
		cancel:     cancel,
		logger:     logger,
	}
}

// RegisterAgent は Agent を登録し、専用の転送goroutineを起動
func (r *MessageRouter) RegisterAgent(name string, t *LocalTransport) {
	r.mu.Lock()
	defer r.mu.Unlock()
	r.transports[name] = t

	// Agent毎に1つのgoroutineでoutboundチャネルを監視
	// チャネル受信でブロックするため、CPU消費なし
	r.wg.Add(1)
	go r.forwardLoop(name, t)
}

// UnregisterAgent は Agent を削除（goroutineはctx.Done()で終了）
func (r *MessageRouter) UnregisterAgent(name string) {
	r.mu.Lock()
	defer r.mu.Unlock()
	delete(r.transports, name)
}

// Stop はルーティングを停止し、全goroutineの終了を待つ（タイムアウト付き）
func (r *MessageRouter) Stop() {
	r.cancel()

	done := make(chan struct{})
	go func() {
		r.wg.Wait()
		close(done)
	}()

	select {
	case <-done:
		r.logger.Info("router.stopped", "status", "clean")
	case <-time.After(10 * time.Second):
		r.logger.Warn("router.stop_timeout",
			"message", "some goroutines did not exit within 10s")
	}
}

// forwardLoop はAgent専用の転送ループ（1 Agent = 1 goroutine）
// チャネル受信でブロックするため、メッセージがないときCPUを消費しない
func (r *MessageRouter) forwardLoop(name string, src *LocalTransport) {
	defer r.wg.Done()
	outCh := src.GetOutboundChannel()

	for {
		select {
		case <-r.ctx.Done():
			return
		case msg, ok := <-outCh:
			if !ok {
				// チャネルがcloseされた（Transport終了）
				return
			}
			r.deliverMessage(msg)
		}
	}
}

// deliverMessage はメッセージを宛先Agentに配送
// 配送失敗時は送信元にエラー応答メッセージを返送する
func (r *MessageRouter) deliverMessage(msg domaintransport.Message) {
	r.mu.RLock()
	targetTransport, ok := r.transports[msg.To]
	r.mu.RUnlock()

	if !ok {
		r.logger.Error("router.agent_not_found",
			"from", msg.From,
			"to", msg.To,
			"job_id", msg.JobID,
		)
		// 送信元にエラー応答を返送（§8.3.3 フォールバック戦略）
		r.sendDeliveryError(msg, "agent_not_found",
			fmt.Sprintf("target agent %q is not registered", msg.To))
		return
	}

	if err := targetTransport.PutInboundMessage(msg); err != nil {
		r.logger.Error("router.delivery_failed",
			"from", msg.From,
			"to", msg.To,
			"error", err,
		)
		// 送信元にエラー応答を返送
		r.sendDeliveryError(msg, "delivery_failed", err.Error())
	}
}

// sendDeliveryError は配送失敗を送信元に通知するエラーメッセージを返送
func (r *MessageRouter) sendDeliveryError(originalMsg domaintransport.Message, errorCode, errorDetail string) {
	r.mu.RLock()
	senderTransport, ok := r.transports[originalMsg.From]
	r.mu.RUnlock()

	if !ok {
		// 送信元も見つからない場合はログのみ
		r.logger.Error("router.sender_not_found_for_error_response",
			"from", originalMsg.From,
			"to", originalMsg.To,
			"error_code", errorCode,
		)
		return
	}

	errorMsg := domaintransport.Message{
		From:      "router",
		To:        originalMsg.From,
		Type:      "result",
		JobID:     originalMsg.JobID,
		Timestamp: time.Now().UTC().Format(time.RFC3339),
		Payload: domaintransport.ResultPayload{
			Status:       "error",
			ErrorMessage: fmt.Sprintf("delivery failed: %s - %s", errorCode, errorDetail),
			ErrorCode:    errorCode,
			ElapsedTime:  "0s",
		},
	}

	if err := senderTransport.PutInboundMessage(errorMsg); err != nil {
		r.logger.Error("router.error_response_failed",
			"target", originalMsg.From,
			"error", err,
		)
	}
}
```

**旧設計との差分**:

| 項目 | 旧（busy-loop） | 新（per-agent goroutine） |
|------|-----------------|--------------------------|
| CPU使用率 | 100%（defaultケースで即座にループ再開） | ~0%（チャネル受信でブロック） |
| ロック範囲 | mapイテレーション中にRLock/RUnlock交互 | deliverMessage内で最小限のRLock |
| map変更時の安全性 | イテレーション中のmap変更でpanicリスク | RLockはdeliverMessage内のみ、mapイテレーションなし |
| goroutine数 | 1（全Agent共有） | Agent数と同じ（通常5個以下） |

### 3.4 メッセージ配送保証

#### 3.4.1 配送保証レベル

| トランスポート | 保証レベル | 根拠 |
|----------------|-----------|------|
| **LocalTransport** | **At-most-once** | チャネルベース、永続化なし。チャネル満杯時はエラー返却（ドロップではない） |
| **SSHTransport** | **At-most-once + リトライ** | `SendWithRetry()` で最大3回リトライ。リトライ間で重複排除なし |

**設計判断**: PicoClaw は低レイテンシ・軽量を重視するため、at-most-once を基本とする。Exactly-once はメッセージブローカー（Kafka等）が必要となり、運用コストに見合わないため採用しない。

#### 3.4.2 チャネル満杯時の動作

```go
// PutInboundMessage はメッセージを受信キューに投入
// チャネル満杯の場合はブロックせずエラーを返す
func (t *LocalTransport) PutInboundMessage(msg transport.Message) error {
    select {
    case t.inbound <- msg:
        return nil
    default:
        return fmt.Errorf("inbound channel full for agent (capacity: %d)", cap(t.inbound))
    }
}
```

**チャネル容量**: `inbound` / `outbound` ともに容量 `100`（§3.2 参照）。通常運用で満杯になることは想定していないが、バーストトラフィック対策として以下を実施:

- `deliverMessage()` がエラーを受信した場合、`sendDeliveryError()` で送信元に通知（§3.3 参照）
- ログイベント `router.delivery_failed` を記録し、モニタリングで検知可能にする

#### 3.4.3 配送失敗時のフロー

```
送信元Agent → MessageRouter.deliverMessage()
                  ↓ 宛先不明
              sendDeliveryError() → 送信元にエラー応答
                  ↓
              送信元が再試行 or ユーザーに通知

送信元Agent → SSHTransport.SendWithRetry()
                  ↓ 接続エラー
              reconnect() → 指数バックオフ3回リトライ
                  ↓ 全失敗
              Orchestrator.executeWithFallback() → 代替Agent試行（§8.3.3）
```

#### 3.4.4 メッセージ順序保証

- **同一Agent間**: FIFO保証（チャネルの特性による）
- **異なるAgent間**: 順序保証なし（各Agentは独立したgoroutineで処理）
- **並列実行時**: 同一JobIDの結果は `DistributedOrchestrator` で集約時にタイムスタンプ順にソート

### 4.1 SSHTransport 実装

**ファイル**: `internal/infrastructure/transport/ssh.go`

**技術選択**:
- `golang.org/x/crypto/ssh` パッケージ使用
- SSH接続 → リモートで `picoclaw-agent --standalone` 起動
- stdin/stdout で JSON 送受信（1行1メッセージ）

**設計ポイント**:
- 再接続ロジック（3回リトライ、指数バックオフ）
- タイムアウト（接続: 10秒、通信: 30秒）
- ヘルスチェック（定期的なping/pong）
- エラーハンドリング（接続切断時の復旧）

**実装**:
```go
package transport

import (
	"context"
	"encoding/json"
	"errors"
	"fmt"
	"io"
	"net"
	"sync"
	"time"

	"golang.org/x/crypto/ssh"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// SSHTransport はSSH経由のリモート通信の実装
//
// ロック戦略:
//   - mu: closedフラグとconnect/reconnectの排他制御
//   - sendMu: encoder への排他アクセス（Send と reconnect の競合防止）
//   - decoder は Receive 内の goroutine でのみ使用（1つずつ順次実行）
type SSHTransport struct {
	host       string
	user       string
	privateKey string
	remoteCmd  string // リモートで実行するコマンド

	client  *ssh.Client
	session *ssh.Session
	stdin   io.WriteCloser
	stdout  io.Reader
	stderr  io.Reader

	decoder *json.Decoder
	encoder *json.Encoder

	closed       bool
	reconnecting bool           // 再接続中フラグ（mu で保護）
	mu           sync.Mutex     // closedフラグ・接続状態・reconnectingの保護
	sendMu       sync.Mutex     // encoder への排他アクセス
}

// SSHConfig は SSHTransport の設定
type SSHConfig struct {
	Host          string
	User          string
	PrivateKey    string
	RemoteCommand string
	Timeout       time.Duration
}

// NewSSHTransport は SSHTransport を作成
func NewSSHTransport(cfg SSHConfig) (*SSHTransport, error) {
	t := &SSHTransport{
		host:       cfg.Host,
		user:       cfg.User,
		privateKey: cfg.PrivateKey,
		remoteCmd:  cfg.RemoteCommand,
		closed:     false,
	}

	if err := t.connect(); err != nil {
		return nil, fmt.Errorf("failed to connect: %w", err)
	}

	return t, nil
}

// connect はSSH接続を確立
func (t *SSHTransport) connect() error {
	// SSH鍵の読み込み
	key, err := ssh.ParsePrivateKey([]byte(t.privateKey))
	if err != nil {
		return fmt.Errorf("failed to parse private key: %w", err)
	}

	// SSH設定
	sshConfig := &ssh.ClientConfig{
		User: t.user,
		Auth: []ssh.AuthMethod{
			ssh.PublicKeys(key),
		},
		HostKeyCallback: ssh.InsecureIgnoreHostKey(), // TODO: 本番ではホスト鍵検証
		Timeout:         10 * time.Second,
	}

	// SSH接続
	client, err := ssh.Dial("tcp", t.host, sshConfig)
	if err != nil {
		return fmt.Errorf("failed to dial: %w", err)
	}
	t.client = client

	// セッション作成
	session, err := client.NewSession()
	if err != nil {
		return fmt.Errorf("failed to create session: %w", err)
	}
	t.session = session

	// stdin/stdout/stderr取得
	stdin, err := session.StdinPipe()
	if err != nil {
		return fmt.Errorf("failed to get stdin: %w", err)
	}
	t.stdin = stdin

	stdout, err := session.StdoutPipe()
	if err != nil {
		return fmt.Errorf("failed to get stdout: %w", err)
	}
	t.stdout = stdout

	stderr, err := session.StderrPipe()
	if err != nil {
		return fmt.Errorf("failed to get stderr: %w", err)
	}
	t.stderr = stderr

	// JSON エンコーダー/デコーダー
	t.encoder = json.NewEncoder(t.stdin)
	t.decoder = json.NewDecoder(t.stdout)

	// リモートコマンド実行
	if err := session.Start(t.remoteCmd); err != nil {
		return fmt.Errorf("failed to start remote command: %w", err)
	}

	return nil
}

// decodeResult は Receive のgoroutine間でデータ競合なく結果を受け渡すための型
type decodeResult struct {
	msg domaintransport.Message
	err error
}

// Send はメッセージを送信
// ロック戦略: sendMu でエンコーダへの排他アクセスを保証
func (t *SSHTransport) Send(ctx context.Context, msg domaintransport.Message) error {
	t.mu.Lock()
	if t.closed {
		t.mu.Unlock()
		return errors.New("transport is closed")
	}
	t.mu.Unlock()

	// エンコーダへの排他アクセス（reconnect()との競合防止）
	t.sendMu.Lock()
	defer t.sendMu.Unlock()

	done := make(chan error, 1)
	go func() {
		done <- t.encoder.Encode(msg)
	}()

	select {
	case err := <-done:
		return err
	case <-ctx.Done():
		return ctx.Err()
	}
}

// Receive はメッセージを受信
// データ競合の防止: goroutineの結果をdecodeResult構造体として
// チャネル経由で受け渡し、共有変数への並行アクセスを排除
func (t *SSHTransport) Receive(ctx context.Context) (domaintransport.Message, error) {
	t.mu.Lock()
	if t.closed {
		t.mu.Unlock()
		return domaintransport.Message{}, errors.New("transport is closed")
	}
	t.mu.Unlock()

	// 結果をチャネル経由で受け取り、共有変数への並行アクセスを排除
	resultCh := make(chan decodeResult, 1)
	go func() {
		var msg domaintransport.Message
		err := t.decoder.Decode(&msg)
		resultCh <- decodeResult{msg: msg, err: err}
	}()

	select {
	case result := <-resultCh:
		if result.err != nil {
			return domaintransport.Message{}, result.err
		}
		return result.msg, nil
	case <-ctx.Done():
		// 注意: goroutineはDecodeでブロックし続ける可能性がある。
		// 呼び出し側はClose()でstdinをcloseし、Decodeを強制終了させること。
		return domaintransport.Message{}, ctx.Err()
	}
}

// Close はTransportを閉じる
func (t *SSHTransport) Close() error {
	t.mu.Lock()
	defer t.mu.Unlock()

	if t.closed {
		return nil
	}

	t.closed = true

	if t.stdin != nil {
		t.stdin.Close()
	}
	if t.session != nil {
		t.session.Close()
	}
	if t.client != nil {
		t.client.Close()
	}

	return nil
}

// IsHealthy はTransportの健全性を確認
func (t *SSHTransport) IsHealthy() bool {
	t.mu.Lock()
	defer t.mu.Unlock()

	if t.closed || t.client == nil {
		return false
	}

	// ping/pong テスト（簡易ヘルスチェック）
	_, _, err := t.client.SendRequest("keepalive@openssh.com", true, nil)
	return err == nil
}
```

```go
// AgentHandler はメッセージを処理するAgent共通インターフェース
type AgentHandler interface {
	HandleMessage(ctx context.Context, msg domaintransport.Message) (domaintransport.Message, error)
}

func main() {
	// フラグ定義
	standalone := flag.Bool("standalone", false, "Run in standalone mode (stdin/stdout JSON)")
	agentType := flag.String("agent", "", "Agent type: worker, coder1, coder2, coder3")
	configPath := flag.String("config", "", "Config file path (default: PICOCLAW_CONFIG env or ./config.yaml)")
	flag.Parse()

	if !*standalone {
		log.Fatal("This command requires --standalone flag")
	}
	if *agentType == "" {
		log.Fatal("Agent type is required (--agent=worker|coder1|coder2|coder3)")
	}

	// ログを stderr に出力（stdout は通信用）
	log.SetOutput(os.Stderr)
	log.Printf("Starting picoclaw-agent in standalone mode (agent=%s)", *agentType)

	// 設定読み込み
	cfgPath := *configPath
	if cfgPath == "" {
		cfgPath = os.Getenv("PICOCLAW_CONFIG")
	}
	if cfgPath == "" {
		cfgPath = "./config.yaml"
	}
	cfg, err := config.LoadConfig(cfgPath)
	if err != nil {
		log.Fatalf("Failed to load config: %v", err)
	}

	// Agent初期化
	handler, err := initAgent(*agentType, cfg)
	if err != nil {
		log.Fatalf("Failed to init agent: %v", err)
	}
	log.Printf("Agent %s initialized successfully", *agentType)

	// シグナルハンドリング（Graceful Shutdown）
	sigCh := make(chan os.Signal, 1)
	signal.Notify(sigCh, syscall.SIGTERM, syscall.SIGINT)

	// stdin/stdout でJSON通信ループ
	// json.Decoder を使用（bufio.Scanner の 64KB 制限を回避）
	decoder := json.NewDecoder(os.Stdin)
	encoder := json.NewEncoder(os.Stdout)

	// メッセージ処理ループ（1メッセージずつ直列処理 → Sessionスレッドセーフ性を保証）
	doneCh := make(chan struct{})
	go func() {
		defer close(doneCh)
		for decoder.More() {
			var msg domaintransport.Message
			if err := decoder.Decode(&msg); err != nil {
				log.Printf("ERROR: Failed to decode message: %v", err)
				continue
			}

			log.Printf("Received: from=%s to=%s job_id=%s", msg.From, msg.To, msg.JobID)

			// メッセージ処理
			ctx, cancel := context.WithTimeout(context.Background(), 300*time.Second)
			result, err := handler.HandleMessage(ctx, msg)
			cancel()

			if err != nil {
				log.Printf("ERROR: HandleMessage failed: %v", err)
				// エラー応答を返す
				result = domaintransport.Message{
					From:      msg.To,
					To:        msg.From,
					SessionID: msg.SessionID,
					JobID:     msg.JobID,
					Message:   fmt.Sprintf("エラー: %v", err),
					Result: &domaintransport.ResultPayload{
						Status:       "failed",
						ErrorMessage: err.Error(),
						ElapsedTime:  "0s",
					},
					Timestamp: time.Now().UTC().Format(time.RFC3339),
				}
			}

			if err := encoder.Encode(result); err != nil {
				log.Printf("ERROR: Failed to encode result: %v", err)
			}
		}
	}()

	// シグナルまたはstdin EOF を待機
	select {
	case sig := <-sigCh:
		log.Printf("Received signal: %s, shutting down...", sig)
	case <-doneCh:
		log.Println("stdin closed, shutting down...")
	}

	// --- Graceful Shutdown 手順 ---
	// 1. 新規メッセージ受付停止（stdinの読み取りは既に停止 or sigで終了）
	log.Println("Graceful shutdown: stopping new message acceptance")

	// 2. 処理中タスクの完了待ち（タイムアウト付き）
	shutdownTimeout := 30 * time.Second
	shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), shutdownTimeout)
	defer shutdownCancel()

	// doneCh が閉じていない場合（シグナルで終了した場合）、処理中メッセージの完了を待つ
	select {
	case <-doneCh:
		log.Println("Graceful shutdown: all in-flight messages completed")
	case <-shutdownCtx.Done():
		log.Printf("Graceful shutdown: timeout after %v, forcing exit", shutdownTimeout)
	}

	// 3. AgentHandler のクリーンアップ（リソース解放）
	if closer, ok := handler.(io.Closer); ok {
		if err := closer.Close(); err != nil {
			log.Printf("Graceful shutdown: handler cleanup error: %v", err)
		}
	}

	log.Println("picoclaw-agent finished")
}

// initAgent はAgent種別に応じた初期化を行う
func initAgent(agentType string, cfg *config.Config) (AgentHandler, error) {
	switch agentType {
	case "worker":
		return initWorker(cfg)
	case "coder1":
		return initCoder1(cfg)
	case "coder2":
		return initCoder2(cfg)
	case "coder3":
		return initCoder3(cfg)
	default:
		return nil, fmt.Errorf("unknown agent type: %s", agentType)
	}
}

// === Worker ===

type workerHandler struct {
	shiroAgent       *agent.ShiroAgent
	executionService service.WorkerExecutionService
}

func initWorker(cfg *config.Config) (*workerHandler, error) {
	// Worker はOllama(共通モデル) + ToolRunner + MCPClient + WorkerExecutionService
	ollamaProvider := ollama.NewOllamaProvider(cfg.Ollama.BaseURL, cfg.Ollama.Model)
	toolRunner := tools.NewToolRunner()
	mcpClient := mcp.NewMCPClient()
	shiroAgent := agent.NewShiroAgent(ollamaProvider, toolRunner, mcpClient)
	executionService := service.NewWorkerExecutionService(cfg.Worker)

	return &workerHandler{
		shiroAgent:       shiroAgent,
		executionService: executionService,
	}, nil
}

func (h *workerHandler) HandleMessage(ctx context.Context, msg domaintransport.Message) (domaintransport.Message, error) {
	start := time.Now()

	// Proposalが含まれている場合 → Worker即時実行
	if msg.Proposal != nil {
		// Proposal からドメインオブジェクトに変換
		p := proposal.Reconstruct(msg.Proposal.Plan, msg.Proposal.Patch, msg.Proposal.Risk, msg.Proposal.CostHint)
		if p == nil || !p.IsValid() {
			return domaintransport.Message{}, fmt.Errorf("invalid proposal in message")
		}

		// JobID復元
		jobID, err := task.ParseJobID(msg.JobID)
		if err != nil {
			return domaintransport.Message{}, fmt.Errorf("invalid job_id: %w", err)
		}

		// Worker即時実行
		result, err := h.executionService.ExecuteProposal(ctx, jobID, p)
		if err != nil {
			return domaintransport.Message{}, fmt.Errorf("execution failed: %w", err)
		}

		// 結果をメッセージに変換
		return domaintransport.Message{
			From:      "Worker",
			To:        msg.From,
			SessionID: msg.SessionID,
			JobID:     msg.JobID,
			Message:   fmt.Sprintf("実行完了: %d件成功, %d件失敗", result.ExecutedCmds, result.FailedCmds),
			Result:    convertToResultPayload(result),
			Timestamp: time.Now().UTC().Format(time.RFC3339),
		}, nil
	}

	// Proposalなし → ShiroAgent.Execute()
	t := task.NewTask(task.JobID{}, msg.Message, "", "")
	response, err := h.shiroAgent.Execute(ctx, t)
	if err != nil {
		return domaintransport.Message{}, err
	}

	return domaintransport.Message{
		From:      "Worker",
		To:        msg.From,
		SessionID: msg.SessionID,
		JobID:     msg.JobID,
		Message:   response,
		Result: &domaintransport.ResultPayload{
			Status:      "success",
			ElapsedTime: time.Since(start).String(),
		},
		Timestamp: time.Now().UTC().Format(time.RFC3339),
	}, nil
}

// HandleMessage はCoder共通のメッセージ処理
func (h *coderHandler) HandleMessage(ctx context.Context, msg domaintransport.Message) (domaintransport.Message, error) {
	// タスク構築
	t := task.NewTask(task.JobID{}, msg.Message, "", "")

	// Proposal生成
	p, err := h.coderAgent.GenerateProposal(ctx, t)
	if err != nil {
		return domaintransport.Message{}, fmt.Errorf("proposal generation failed: %w", err)
	}

	if p == nil {
		return domaintransport.Message{
			From:      h.agentName,
			To:        msg.From,
			SessionID: msg.SessionID,
			JobID:     msg.JobID,
			Message:   "Proposalを生成できませんでした",
			Timestamp: time.Now().UTC().Format(time.RFC3339),
		}, nil
	}

	// Proposal付きメッセージを返す
	return domaintransport.Message{
		From:      h.agentName,
		To:        msg.From,
		SessionID: msg.SessionID,
		JobID:     msg.JobID,
		Message:   "Proposal生成完了",
		Proposal: &domaintransport.ProposalPayload{
			Plan:     p.Plan(),
			Patch:    p.Patch(),
			Risk:     p.Risk(),
			CostHint: p.CostHint(),
		},
		Timestamp: time.Now().UTC().Format(time.RFC3339),
	}, nil
}

// convertToResultPayload は v3 PatchExecutionResult を v4 ResultPayload に変換する
func convertToResultPayload(r *patch.PatchExecutionResult) *domaintransport.ResultPayload {
	status := "success"
	if r.HasFailures() {
		if r.SuccessRate() > 0 {
			status = "partial_success"
		} else {
			status = "failed"
		}
	}

	// ファイル操作結果の分類
	var created, modified, deleted []string
	for _, cmdResult := range r.Results {
		if !cmdResult.Success || cmdResult.Command.Type != patch.TypeFileEdit {
			continue
		}
		switch cmdResult.Command.Action {
		case patch.ActionCreate, patch.ActionMkdir:
			created = append(created, cmdResult.Command.Target)
		case patch.ActionUpdate, patch.ActionAppend, patch.ActionRename, patch.ActionCopy:
			modified = append(modified, cmdResult.Command.Target)
		case patch.ActionDelete:
			deleted = append(deleted, cmdResult.Command.Target)
		}
	}

	errorMsg := ""
	if r.HasFailures() {
		errorMsg = r.Summary
	}

	return &domaintransport.ResultPayload{
		Status:        status,
		CreatedFiles:  created,
		ModifiedFiles: modified,
		DeletedFiles:  deleted,
		GitCommit:     r.GitCommit,
		ElapsedTime:   fmt.Sprintf("%ds", r.ExecutedCmds), // 簡易版
		ErrorMessage:  errorMsg,
	}
}
```

### 4.3 TransportFactory 実装

**ファイル**: `internal/infrastructure/transport/factory.go`

**設計ポイント**:
- 設定ファイル（`config.yaml`）に基づき、Local/SSH を透過的に切り替え
- SSH鍵の読み込み（秘密鍵パス指定）
- ホスト鍵検証（`known_hosts` 活用）

**実装**:
```go
package transport

import (
	"fmt"
	"os"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// TransportType は Transport の種類
type TransportType string

const (
	TransportTypeLocal TransportType = "local"
	TransportTypeSSH   TransportType = "ssh"
)

// AgentTransportConfig は Agent ごとの Transport 設定
type AgentTransportConfig struct {
	Type       TransportType
	RemoteHost string
	RemoteUser string
	SSHKeyPath string
}

// TransportFactory は Transport を生成
type TransportFactory struct {
	configs map[string]AgentTransportConfig // Agent名 → 設定
}

// NewTransportFactory は TransportFactory を作成
func NewTransportFactory(configs map[string]AgentTransportConfig) *TransportFactory {
	return &TransportFactory{
		configs: configs,
	}
}

// CreateTransport は Agent名に応じた Transport を生成
func (f *TransportFactory) CreateTransport(agentName string) (transport.Transport, error) {
	cfg, ok := f.configs[agentName]
	if !ok {
		return nil, fmt.Errorf("no config for agent: %s", agentName)
	}

	switch cfg.Type {
	case TransportTypeLocal:
		return NewLocalTransport(), nil

	case TransportTypeSSH:
		// SSH鍵の読み込み
		privateKey, err := os.ReadFile(cfg.SSHKeyPath)
		if err != nil {
			return nil, fmt.Errorf("failed to read SSH key: %w", err)
		}

		sshCfg := SSHConfig{
			Host:          cfg.RemoteHost,
			User:          cfg.RemoteUser,
			PrivateKey:    string(privateKey),
			RemoteCommand: fmt.Sprintf("picoclaw-agent --standalone --agent=%s", agentName),
		}

		return NewSSHTransport(sshCfg)

	default:
		return nil, fmt.Errorf("unknown transport type: %s", cfg.Type)
	}
}
```

### 5.1 Session 同期

**ファイル**: `internal/domain/session/distributed.go`

**設計ポイント**:
- 各 Agent が独立した Memory を持つ
- 階層的な会話ログ伝播（Coder → Worker → Mio）
- Mioが全体の会話ログを統合して把握

**実装**:
```go
package session

import (
	"sync"
	"time"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// AgentMemory は各Agentが持つ独立したメモリ
// MaxConversations で保持する会話数の上限を設定し、メモリ使用量を制御する
type AgentMemory struct {
	AgentID          string
	Conversations    []transport.Message
	Context          map[string]interface{}
	SyncPoint        time.Time
	MaxConversations int // 保持する最大会話数（0 = デフォルト100）
	mu               sync.RWMutex
}

// DefaultMaxConversations はAgentMemoryのデフォルト最大会話数
// 1メッセージ ≈ 500B と仮定: 100件 × 5 Agent = 250KB（十分小さい）
const DefaultMaxConversations = 100

// NewAgentMemory は AgentMemory を作成
func NewAgentMemory(agentID string) *AgentMemory {
	return &AgentMemory{
		AgentID:          agentID,
		Conversations:    make([]transport.Message, 0, DefaultMaxConversations),
		Context:          make(map[string]interface{}),
		SyncPoint:        time.Now().UTC(),
		MaxConversations: DefaultMaxConversations,
	}
}

// AddConversation は会話を追加（上限超過時は古いメッセージを破棄）
func (m *AgentMemory) AddConversation(msg transport.Message) {
	m.mu.Lock()
	defer m.mu.Unlock()

	m.Conversations = append(m.Conversations, msg)

	// 上限超過時は古いメッセージを破棄（FIFO）
	maxConv := m.MaxConversations
	if maxConv <= 0 {
		maxConv = DefaultMaxConversations
	}
	if len(m.Conversations) > maxConv {
		// 先頭（最古）を切り捨て。GCフレンドリーにスライスをコピー
		overflow := len(m.Conversations) - maxConv
		copy(m.Conversations, m.Conversations[overflow:])
		m.Conversations = m.Conversations[:maxConv]
	}
}

// ConversationCount は現在の会話数を返す
func (m *AgentMemory) ConversationCount() int {
	m.mu.RLock()
	defer m.mu.RUnlock()
	return len(m.Conversations)
}

// GetConversations は会話ログを取得（コピーを返す）
func (m *AgentMemory) GetConversations() []transport.Message {
	m.mu.RLock()
	defer m.mu.RUnlock()
	result := make([]transport.Message, len(m.Conversations))
	copy(result, m.Conversations)
	return result
}

// CentralMemory は Mioが持つ全体統合メモリ
type CentralMemory struct {
	SessionID     string
	AgentMemories map[string]*AgentMemory // Agent名 → Memory
	mu            sync.RWMutex
}

// NewCentralMemory は CentralMemory を作成
func NewCentralMemory(sessionID string) *CentralMemory {
	return &CentralMemory{
		SessionID:     sessionID,
		AgentMemories: make(map[string]*AgentMemory),
	}
}

// RegisterAgent は Agentのメモリを登録
func (c *CentralMemory) RegisterAgent(agentID string, memory *AgentMemory) {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.AgentMemories[agentID] = memory
}

// GetAgentMemory は Agent のメモリを取得
func (c *CentralMemory) GetAgentMemory(agentID string) (*AgentMemory, bool) {
	c.mu.RLock()
	defer c.mu.RUnlock()
	memory, ok := c.AgentMemories[agentID]
	return memory, ok
}

// GetAllConversations はすべての会話ログを統合して取得
func (c *CentralMemory) GetAllConversations() []transport.Message {
	c.mu.RLock()
	defer c.mu.RUnlock()

	var all []transport.Message
	for _, memory := range c.AgentMemories {
		all = append(all, memory.GetConversations()...)
	}

	// タイムスタンプでソート（time.Time比較で正確な順序を保証）
	// 注: Validate()でUTC強制済みのため文字列比較でも正しいが、
	//      安全のためtime.Timeに変換して比較する
	sort.Slice(all, func(i, j int) bool {
		ti, _ := time.Parse(time.RFC3339, all[i].Timestamp)
		tj, _ := time.Parse(time.RFC3339, all[j].Timestamp)
		return ti.Before(tj)
	})

	return all
}
```

```go
package transport

import (
	"context"
	"sync"

	"github.com/Nyukimin/picoclaw_multiLLM/internal/domain/transport"
)

// TransportPool は Transport の接続プール
type TransportPool struct {
	transports map[string]transport.Transport // Agent名 → Transport
	factory    *TransportFactory
	mu         sync.RWMutex
}

// NewTransportPool は TransportPool を作成
func NewTransportPool(factory *TransportFactory) *TransportPool {
	return &TransportPool{
		transports: make(map[string]transport.Transport),
		factory:    factory,
	}
}

// GetTransport は Agent名に対応するTransportを取得（プール管理）
func (p *TransportPool) GetTransport(agentName string) (transport.Transport, error) {
	p.mu.RLock()
	t, ok := p.transports[agentName]
	p.mu.RUnlock()

	if ok && t.IsHealthy() {
		return t, nil
	}

	// 新規作成または再作成
	p.mu.Lock()
	defer p.mu.Unlock()

	// 既存のTransportを削除
	if ok {
		t.Close()
		delete(p.transports, agentName)
	}

	// 新規作成
	newTransport, err := p.factory.CreateTransport(agentName)
	if err != nil {
		return nil, err
	}

	p.transports[agentName] = newTransport
	return newTransport, nil
}

// CloseAll はすべてのTransportを閉じる
func (p *TransportPool) CloseAll() {
	p.mu.Lock()
	defer p.mu.Unlock()

	for _, t := range p.transports {
		t.Close()
	}
	p.transports = make(map[string]transport.Transport)
}
```

### 5.6 IdleChat オーケストレーション設計

**ファイル**: `internal/application/idlechat/orchestrator.go`

#### 5.6.1 設計思想

- 全Agentは**同一のOllamaモデル**（picoclaw-v1）をベースとする
- Agent固有の個性は**システムプロンプト**で切り替える（モデル切り替え不要）
- タスク処理中はIdleChatを**自動停止**し、リソース競合を防ぐ
- IdleChat開始条件は「`chatBusy == false` かつ `workerBusy == false` かつ `idleDuration >= intervalMin`」
- 通常の `CHAT` では外部検索（`web_search`）の利用を許可する
- `idleChat` では外部検索（Web/API検索）を禁止し、内部文脈のみで会話する
- `idleChat` の話題は `chat(mio)` が提示し、話題比率は `user 30% / current 20% / tech 20% / worker_db 20% / random 10%`
- `chat` と `worker` は積極的に興味を示し、深掘り対話を行う
- 会話ループ検知時は一度停止して `worker(shiro)` が要約保存し、新規話題で再開する
- セッション終端時は `worker` が「`何月何日の何の話題まとめ`」形式で要約を保存する
- 参加Agentは設定（`idle_chat.participants`）で制御。デフォルトはMio + Shiroのみ
- Coder系Agent（Aka/Ao/Gin）も`participants`に追加すれば参加可能

```go
// Start はIdleChatモニタリングを開始
func (o *IdleChatOrchestrator) Start(ctx context.Context) {
	o.mu.Lock()
	if o.running {
		o.mu.Unlock()
		return
	}
	o.running = true
	o.mu.Unlock()

	go o.monitorLoop(ctx)
}

// Stop はIdleChatを停止
func (o *IdleChatOrchestrator) Stop() {
	o.mu.Lock()
	defer o.mu.Unlock()
	if o.running {
		close(o.stopCh)
		o.running = false
	}
}

// NotifyTaskActivity はタスク処理を通知（アイドルタイマーリセット）
func (o *IdleChatOrchestrator) NotifyTaskActivity() {
	o.mu.Lock()
	defer o.mu.Unlock()
	o.lastTask = time.Now()
}

// monitorLoop はアイドル状態を監視し、雑談を開始する
func (o *IdleChatOrchestrator) monitorLoop(ctx context.Context) {
	ticker := time.NewTicker(30 * time.Second) // 30秒ごとにチェック
	defer ticker.Stop()

	for {
		select {
		case <-ctx.Done():
			return
		case <-o.stopCh:
			return
		case <-ticker.C:
			o.mu.Lock()
			idleDuration := time.Since(o.lastTask)
			o.mu.Unlock()

			threshold := time.Duration(o.intervalMin) * time.Minute
			if idleDuration >= threshold {
				o.runConversation(ctx)
				// 雑談後にタイマーリセット
				o.mu.Lock()
				o.lastTask = time.Now()
				o.mu.Unlock()
			}
		}
	}
}

// runConversation は1回の雑談セッションを実行
func (o *IdleChatOrchestrator) runConversation(ctx context.Context) {
	if len(o.participants) < 2 {
		return
	}

	// ランダムにトピックを選択
	topics := []string{
		"最近のタスクで面白かったこと",
		"効率的な作業方法について",
		"今日の天気と気分",
		"好きなプログラミング言語の話",
		"休日の過ごし方",
	}
	topic := topics[rand.Intn(len(topics))]

	// 最初の発言者をランダム選択
	currentSpeaker := o.participants[rand.Intn(len(o.participants))]
	message := fmt.Sprintf("（雑談トピック: %s）", topic)

	fmt.Printf("[IdleChat] 雑談開始: トピック=%s, 参加者=%v\n", topic, o.participants)

	for turn := 0; turn < o.maxTurns; turn++ {
		// タスクが来たら即停止
		select {
		case <-ctx.Done():
			return
		case <-o.stopCh:
			return
		default:
		}

		// 現在の発言者のシステムプロンプトを取得
		personality, ok := o.personalities[currentSpeaker]
		if !ok {
			continue
		}

		// LLMで応答生成
		response, err := o.provider.Chat(ctx, personality.SystemPrompt, message)
		if err != nil {
			fmt.Printf("[IdleChat] エラー: %v\n", err)
			break
		}

		fmt.Printf("[IdleChat] %s: %s\n", currentSpeaker, truncate(response, 100))

		// 次の発言者を選択（現在の発言者以外からランダム）
		message = response
		currentSpeaker = o.nextSpeaker(currentSpeaker)
	}

	fmt.Printf("[IdleChat] 雑談終了 (%d ターン)\n", o.maxTurns)
}

// nextSpeaker は次の発言者を選択（ラウンドロビン）
func (o *IdleChatOrchestrator) nextSpeaker(current string) string {
	for i, p := range o.participants {
		if p == current {
			return o.participants[(i+1)%len(o.participants)]
		}
	}
	return o.participants[0]
}

// truncate は文字列を指定長で切り詰める
func truncate(s string, maxLen int) string {
	runes := []rune(s)
	if len(runes) <= maxLen {
		return s
	}
	return string(runes[:maxLen]) + "..."
}
```

#### 5.6.3 タスク/雑談の排他制御

IdleChatとタスク処理は**同一のOllamaモデル**を使用するため、排他制御が必要。

```
タスク到着時のフロー:

[タスク到着] → [NotifyActivity + chatBusy=true]
                    → [route判定]
                    → [route!=CHAT ? workerBusy=true : 変更なし]
                    → [タスク処理]
                    → [workerBusy=false（必要時）]
                    → [chatBusy=false]

IdleChat開始判定:
[chatBusy=false かつ workerBusy=false かつ idleDuration>=threshold] → [IdleChat開始]
```

**Orchestratorとの統合**（`message_orchestrator.go` での使用例）:

```go

    if o.idleNotifier != nil {
        o.idleNotifier.NotifyActivity()
        o.idleNotifier.SetChatBusy(true)
        defer o.idleNotifier.SetChatBusy(false)
    }

    decision := route(...)
    if o.idleNotifier != nil && decision.Route != routing.RouteCHAT {
        o.idleNotifier.SetWorkerBusy(true)
        defer o.idleNotifier.SetWorkerBusy(false)
    }
    return execute(...)
}
```

**配線**:
- `cmd/picoclaw/main.go` で `distOrch.SetIdleNotifier(idleChatOrch)` を設定して連携する。

## 7. Session/Memory分散管理

### 7.1 Session管理の設計

**v3.0（単一プロセス）**:
```
[Mio/Worker/Coder] → [共有Session] → [共有Memory]
```

**v4.0（分散実行）**:
```
[Mio]   → [MioMemory]
[Worker] → [WorkerMemory]
[Coder3] → [Coder3Memory]
  ↓
階層的に伝播
  ↓
[CentralMemory（Mio）] ← すべての会話ログを統合
```

#### 7.1.1 SessionID 生成ルール

**形式**: `YYYYMMDD-{channel}-{chatID}`

| フィールド | 説明 | 例 |
|-----------|------|-----|
| `YYYYMMDD` | 日付（UTC） | `20260302` |
| `channel` | メッセージチャネル | `line`, `slack` |
| `chatID` | チャネル固有のユーザー/チャット識別子 | `U123456`, `C789012` |

**例**: `20260302-line-U123456`

**生成責務**: Adapter層（LINE Handler / Slack Handler）が生成する。

```go
// GenerateSessionID はメッセージ受信時にSessionIDを生成する
// Adapter層（LINE/Slack Handler）で呼び出す
func GenerateSessionID(channel, chatID string) string {
    return fmt.Sprintf("%s-%s-%s",
        time.Now().UTC().Format("20060102"),
        channel,
        chatID,
    )
}
```

**一意性保証**:
- 同一ユーザー + 同一日 → 同一SessionID（日次カットオーバーによるリセット）
- 日付が変わると新しいSessionIDが生成される（セッションの自然分離）
- 分散環境でも一意性が保証される（chatID はチャネル側で一意）

**分散環境での注意**:
- SessionID の生成はホスト（マシン1: Ubuntu）の Adapter 層でのみ行われる
- リモート Agent（Coder3）は SessionID を生成せず、メッセージ内の `session_id` をそのまま使用する
- Session の永続化（JSONファイル）はホスト側の SessionRepository が一元管理する

#### 7.1.2 Sessionのスレッドセーフ性

**課題**: v3の`Session`構造体（`internal/domain/session/session.go`）にはmutexが存在しない。v3では単一goroutineからのアクセスを前提としていたが、v4では以下の理由でスレッドセーフ性が必要になる。

- Transport層のReceive goroutineが複数のメッセージを並行処理する可能性
- Worker内のPatchCommand並列実行中にSessionの更新が発生する可能性
- 会話ログの伝播（§7.2）中にSessionへの読み書きが競合する可能性

**設計方針**: Agent単位のSession所有権

```
[v4.0 設計]
各AgentがローカルにSessionを所有し、書き込みは所有者のみ行う。
他AgentからのSessionデータはメッセージ（Context）経由で読み取り専用で受信する。

結果:
- Session構造体にmutexは追加しない（単一所有者の原則）
- Agentプロセス内では、メッセージ処理を直列化（1メッセージずつ処理）
- 並列アクセスが必要になった場合のみ、sync.RWMutex を導入
```

**実装ガイドライン**:

1. **v4.0初期**: Agentごとに1つのメッセージ処理goroutineを維持（直列処理）
   - `Receive()` → 処理 → `Send()` のループを1 goroutineで実行
   - Session への競合アクセスは発生しない

2. **v4.0後期（必要に応じて）**: sync.RWMutex 導入
   ```go
   type Session struct {
       mu        sync.RWMutex           // v4.0後期: 並行アクセス保護
       id        string
       channel   string
       chatID    string
       history   []task.Task
       memory    map[string]interface{}
       createdAt time.Time
       updatedAt time.Time
   }
   ```

3. **原則**: mutex導入は「計測してボトルネックが確認された場合のみ」とする（YAGNI）

### 7.2 階層的な会話ログ伝播

**伝播フロー**:
1. **Coder3 → Worker**: Coder3の会話ログがWorkerに伝播
2. **Worker → Mio**: Worker、Coder3の会話ログがMioに伝播
3. **Mioが全体把握**: すべての会話ログを受け取り、モニタリング

**実装例**:

```go
// Worker がメッセージを受信したとき、Coderの会話ログを取得
func (w *WorkerAgent) ReceiveFromCoder(msg transport.Message) {
    // Coderの会話ログを自分のMemoryに追加
    w.memory.AddConversation(msg)

    // Coderのメモリも取得（任意）
    if msg.Context != nil {
        if coderHistory, ok := msg.Context["coder_history"].([]interface{}); ok {
            for _, h := range coderHistory {
                w.memory.AddExternalConversation(h)
            }
        }
    }
}

// Worker が Mioに報告するとき、自分とCoderの会話ログを含める
func (w *WorkerAgent) SendToMio(msg transport.Message) {
    msg.Context["worker_history"] = w.memory.GetConversations()
    msg.Context["coder_history"] = w.memory.GetExternalConversations()

    w.transport.Send(context.Background(), msg)
}
```

### 7.4 競合解決

**問題**: 複数Agentが同時に同じファイルを編集する場合

**解決策**:
1. **楽観的ロック**: ファイルのハッシュ値をPatchCommand.Metadataに格納し、書き込み前にチェック
2. **ワークスペース分離**: Agent別のワークスペースディレクトリ（実験的）
3. **Git auto-commit**: 各編集をcommitし、衝突時はユーザーに通知

**楽観的ロックの実装**:

v3の`PatchCommand`構造体（`internal/domain/patch/command.go`）は変更せず、既存の`Metadata`フィールドを活用する。
Coderが`expected_hash`をMetadataに設定し、Workerが実行前に検証する。

```go
// Coder が Proposal生成時、既存ファイルの更新コマンドにハッシュを付与
cmd := patch.NewPatchCommand(patch.TypeFileEdit, patch.ActionUpdate, "main.go", newContent)
cmd = cmd.WithMetadata("expected_hash", computeFileHash("main.go"))

// Worker の patch 実行前にファイル状態を確認
func (w *workerExecutionService) verifyFileHash(cmd patch.PatchCommand) error {
    if !cmd.IsFileEdit() {
        return nil // file_edit以外はスキップ
    }

    expectedHash, ok := cmd.GetMetadata("expected_hash")
    if !ok || expectedHash == "" {
        return nil // ハッシュ未設定はスキップ（新規作成等）
    }

    actualHash := computeFileHash(cmd.Target) // cmd.Target がファイルパス
    if actualHash != expectedHash {
        return fmt.Errorf("file conflict: %s (expected %s, got %s)", cmd.Target, expectedHash, actualHash)
    }
    return nil
}

// computeFileHash はファイルのSHA256ハッシュを計算
func computeFileHash(path string) string {
    data, err := os.ReadFile(path)
    if err != nil {
        return "" // ファイルが存在しない場合は空文字
    }
    hash := sha256.Sum256(data)
    return hex.EncodeToString(hash[:8]) // 先頭8バイト（16文字）
}
```

**注意**: PatchCommand構造体自体は変更しない（§2.4の「不変」リストに含まれる）。`Metadata`フィールド（`map[string]string`）の慣習的なキーとして`expected_hash`を使用する。

```go
// ホスト鍵検証（本番環境）
func createHostKeyCallback() (ssh.HostKeyCallback, error) {
    knownHostsPath := filepath.Join(os.Getenv("HOME"), ".ssh", "known_hosts")
    return knownhosts.New(knownHostsPath)
}

// SSH接続（ホスト鍵検証あり）
sshConfig := &ssh.ClientConfig{
    User: t.user,
    Auth: []ssh.AuthMethod{
        ssh.PublicKeys(key),
    },
    HostKeyCallback: createHostKeyCallback(), // ← 本番環境で必須
    Timeout:         10 * time.Second,
}
```

### 8.2 メッセージ検証

#### 8.2.1 必須フィールドチェック

```go
// Message.Validate() の実装（再掲）
func (m *Message) Validate() error {
    if m.From == "" {
        return errors.New("from is required")
    }
    if m.To == "" {
        return errors.New("to is required")
    }
    if m.SessionID == "" {
        return errors.New("session_id is required")
    }
    if m.JobID == "" {
        return errors.New("job_id is required")
    }
    if m.Message == "" {
        return errors.New("message is required")
    }

    // タイムスタンプの形式チェック
    t, err := time.Parse(time.RFC3339, m.Timestamp)
    if err != nil {
        return fmt.Errorf("timestamp is invalid: %w", err)
    }

    // UTC強制: タイムゾーン混在を防止し、文字列ソートの正確性を保証
    // UTC以外（例: +09:00）の場合はエラー
    if t.Location() != time.UTC {
        return fmt.Errorf("timestamp must be in UTC (got %s), use time.Now().UTC().Format(time.RFC3339)", m.Timestamp)
    }

    return nil
}
```

```go
// SessionID/JobIDのフォーマット検証
func (m *Message) ValidateIDs() error {
    // SessionID形式: YYYYMMDD-{channel}-{chatID}（例: "20260301-line-U123456"）
    // v3既存形式を維持: 日付(8桁)-チャネル名(小文字)-チャットID

        return fmt.Errorf("invalid session_id format: %s (expected YYYYMMDD-channel-chatID)", m.SessionID)
```

### 8.3 エラーハンドリング

#### 8.3.1 接続エラー（再接続）

**戦略**: 指数バックオフで3回リトライ

```go
// 再接続ロジック
// reconnect はSSH接続を再確立する
//
// 設計原則: ロックの保持時間を最小化する
// - sleep（指数バックオフ待ち）はロック外で行う
// - ロックは接続確立（connect()呼び出し）時のみ保持する
// - reconnecting フラグで多重再接続を防止する
func (t *SSHTransport) reconnect() error {
    // 再接続中フラグの排他制御
    t.mu.Lock()
    if t.reconnecting {
        t.mu.Unlock()
        return fmt.Errorf("reconnection already in progress")
    }
    t.reconnecting = true
    t.mu.Unlock()

    defer func() {
        t.mu.Lock()
        t.reconnecting = false
        t.mu.Unlock()
    }()

    maxRetries := 3
    baseDelay := 1 * time.Second

    for i := 0; i < maxRetries; i++ {
        delay := baseDelay * time.Duration(1<<uint(i)) // 指数バックオフ
        log.Printf("Reconnecting (attempt %d/%d) after %v...", i+1, maxRetries, delay)

        // ロック外でスリープ（他のgoroutineをブロックしない）
        time.Sleep(delay)

        // 接続確立時のみロックを取得
        t.mu.Lock()
        t.sendMu.Lock()
        err := t.connect()
        t.sendMu.Unlock()
        t.mu.Unlock()

        if err == nil {
            log.Printf("Reconnection successful")
            return nil
        }
        log.Printf("Reconnection attempt %d failed: %v", i+1, err)
    }

    return fmt.Errorf("failed to reconnect after %d attempts", maxRetries)
}

// SSHTransport の reconnecting フィールド追加
// type SSHTransport struct {
//     ...
//     reconnecting bool  // 再接続中フラグ（mu で保護）
//     ...
// }

// SendWithRetry は接続エラー時に再接続を試みてリトライする
func (t *SSHTransport) SendWithRetry(ctx context.Context, msg domaintransport.Message) error {
    err := t.Send(ctx, msg)
    if err != nil && isConnectionError(err) {
        // 再接続試行
        if reconnectErr := t.reconnect(); reconnectErr == nil {
            // リトライ
            return t.Send(ctx, msg)
        }
    }
    return err
}
```

#### 8.3.2 タイムアウト

**設定**:
- **接続タイムアウト**: 10秒
- **通信タイムアウト**: 30秒（通常）、300秒（Coder実行）

```go
// タイムアウト設定（コンテキスト使用）
func (t *SSHTransport) Send(ctx context.Context, msg transport.Message) error {
    // デフォルト30秒、Coder実行時は300秒
    timeout := 30 * time.Second
    if msg.To == "Coder1" || msg.To == "Coder2" || msg.To == "Coder3" {
        timeout = 300 * time.Second
    }

    ctx, cancel := context.WithTimeout(ctx, timeout)
    defer cancel()

    done := make(chan error, 1)
    go func() {
        done <- t.encoder.Encode(msg)
    }()

    select {
    case err := <-done:
        return err
    case <-ctx.Done():
        return fmt.Errorf("send timeout after %v", timeout)
    }
}
```

#### 8.3.3 リモートAgent到達不能時のフォールバック戦略

**問題**: リモートAgent（SSH経由のCoder等）が到達不能な場合、タスクが永久にハングする

**フォールバック階層**:

| レベル | 条件 | 動作 |
|--------|------|------|
| L1: リトライ | 接続エラー検出時 | `SendWithRetry()` で指数バックオフ3回リトライ（§8.3.1） |
| L2: 代替Agent | リトライ全失敗 | 同一ロールの別Agentにフォールバック（例: Coder1→Coder2） |
| L3: ローカル実行 | 代替Agentも不可 | v3互換のローカル実行にフォールバック |
| L4: エラー応答 | すべて失敗 | Mio経由でユーザーにエラー通知 |

**Orchestratorでの実装**:

```go
// executeWithFallback はリモートAgent到達不能時にフォールバックを試みる
func (o *DistributedOrchestrator) executeWithFallback(
    ctx context.Context,
    msg transport.Message,
    primaryAgent string,
    fallbackAgents []string,
) (transport.Message, error) {
    // L1: プライマリAgentに送信（SendWithRetryが内部でリトライ）
    result, err := o.sendAndWait(ctx, msg, primaryAgent)
    if err == nil {
        return result, nil
    }
    log.Printf("Primary agent %s failed: %v", primaryAgent, err)

    // L2: フォールバックAgentを順に試行
    for _, agent := range fallbackAgents {
        log.Printf("Trying fallback agent: %s", agent)
        msg.To = agent
        result, err = o.sendAndWait(ctx, msg, agent)
        if err == nil {
            log.Printf("Fallback to %s succeeded", agent)
            return result, nil
        }
        log.Printf("Fallback agent %s also failed: %v", agent, err)
    }

    // L3: ローカル実行フォールバック（v3互換）
    if o.localFallbackEnabled {
        log.Printf("All remote agents failed, falling back to local execution")
        return o.executeLocally(ctx, msg)
    }

    // L4: エラー応答
    return transport.Message{}, fmt.Errorf(
        "all agents failed: primary=%s, fallbacks=%v", primaryAgent, fallbackAgents)
}
```

**フォールバックマッピング**:

```go
// agentFallbackMap はAgent種別ごとのフォールバック順序を定義
var agentFallbackMap = map[string][]string{
    "Shiro":  {},                          // Workerは代替なし（ローカルフォールバックのみ）
    "Coder1": {"Coder2", "Coder3"},        // DeepSeek → OpenAI → Claude
    "Coder2": {"Coder1", "Coder3"},        // OpenAI → DeepSeek → Claude
    "Coder3": {"Coder1", "Coder2"},        // Claude → DeepSeek → OpenAI
}
```

**MessageRouter のエラー応答**: 配送失敗時は `sendDeliveryError()` で送信元に即座にエラー応答を返送する（§3.3 `deliverMessage()` 参照）。

#### 8.3.4 部分的失敗

**問題**: 複数のPatchCommand実行中、途中で失敗した場合

**対策**: トランザクション的な実行（ロールバック）

```go
// Worker の patch 実行（トランザクション）
func (w *WorkerExecutionService) ExecutePatchWithRollback(ctx context.Context, patch []PatchCommand) error {
    executedCommands := []PatchCommand{}

    for i, cmd := range patch {
        if err := executePatchCommand(cmd); err != nil {
            // ロールバック
            log.Printf("ERROR: Failed to execute command %d, rolling back...", i)
            for j := len(executedCommands) - 1; j >= 0; j-- {
                rollbackCommand(executedCommands[j])
            }
            return fmt.Errorf("patch execution failed at step %d: %w", i, err)
        }
        executedCommands = append(executedCommands, cmd)
    }

    return nil
}
```

### 8.4 グレースフルシャットダウン

#### 8.4.1 シャットダウンシーケンス

**トリガー**: `SIGTERM` / `SIGINT` 受信、またはstdin EOF

```
[シグナル受信]
  ↓
1. 新規メッセージ受付停止
  ↓
2. 処理中タスク完了待ち（タイムアウト: 30秒）
  ↓
3. MessageRouter.Stop()（goroutine終了待ち、タイムアウト: 10秒）
  ↓
4. Transport クローズ（SSH接続切断）
  ↓
5. AgentHandler クリーンアップ（リソース解放）
  ↓
6. プロセス終了
```

#### 8.4.2 コンポーネント別シャットダウン責務

| コンポーネント | 停止動作 | タイムアウト |
|----------------|----------|-------------|
| **Standalone Agent** (`main`) | stdin読取停止 → 処理中タスク完了待ち → handler.Close() | 30秒 |
| **MessageRouter** (`Stop`) | ctx.cancel() → wg.Wait() | 10秒 |
| **SSHTransport** (`Close`) | SSH session/client 切断 | 5秒 |
| **AgentHandler** (`Close`) | 進行中のLLM呼び出しキャンセル | ctx.Done() に依存 |

#### 8.4.3 実装上の注意

- **処理中タスクの扱い**: シャットダウン時に進行中のタスクは完了を待つが、タイムアウト後は強制終了する。`context.WithCancel` によりLLM APIコールも中断される
- **ログ出力**: シャットダウンの各フェーズで進捗をログ出力し、どのフェーズで停滞したか特定可能にする
- **exit code**: 正常シャットダウンは `0`、タイムアウト強制終了は `1` を返す
- **AgentHandler.Close()**: `io.Closer` インターフェースをオプショナルに実装。未実装の場合はスキップ

## 付録B: SSH接続設定例

### B.1 SSH鍵生成

```bash
# Ed25519鍵の生成（推奨）
ssh-keygen -t ed25519 -f ~/.ssh/id_picoclaw -C "picoclaw-agent"

# または RSA 4096bit
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_picoclaw -C "picoclaw-agent"

# パーミッション設定
chmod 600 ~/.ssh/id_picoclaw
chmod 644 ~/.ssh/id_picoclaw.pub
```

### B.2 公開鍵の配置

```bash
# リモートホストに公開鍵をコピー
ssh-copy-id -i ~/.ssh/id_picoclaw.pub picoclaw@192.168.1.100

# または手動で配置
cat ~/.ssh/id_picoclaw.pub | ssh picoclaw@192.168.1.100 \
  "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

### B.3 known_hosts の設定

```bash
# ホスト鍵を事前に登録
ssh-keyscan -H 192.168.1.100 >> ~/.ssh/known_hosts
ssh-keyscan -H 192.168.1.101 >> ~/.ssh/known_hosts
ssh-keyscan -H 192.168.1.102 >> ~/.ssh/known_hosts
ssh-keyscan -H 192.168.1.103 >> ~/.ssh/known_hosts
```

### B.4 SSH設定ファイル（オプション）

```
Host picoclaw-worker

    HostName 192.168.1.103
```

## 付録C: 用語集

### C.1 Transport層

| 用語 | 説明 |
|------|------|
| **Transport** | Agent間通信を抽象化したインターフェース |
| **LocalTransport** | 同一プロセス内通信の実装（チャネルベース） |
| **SSHTransport** | SSH経由のリモート通信の実装 |
| **MessageRouter** | Agent名に基づくメッセージルーティング |
| **TransportFactory** | Transport生成のファクトリ |
| **TransportPool** | Transport接続プール |

### C.2 メッセージ

| 用語 | 説明 |
|------|------|
| **Message** | Agent間通信の基本メッセージ |
| **Proposal** | Coderが生成する実装提案（plan/patch/risk） |
| **Result** | Workerの実行結果 |
| **Context** | メッセージのコンテキスト情報 |
| **SessionID** | セッション識別子（形式: `YYYYMMDD-channel-chatID`、例: `20260302-line-U123456`） |
| **JobID** | Job識別子（形式: `YYYYMMDD-HHMMSS-uuid8`、例: `20260302-153005-abc12345`） |

### C.3 Session/Memory

| 用語 | 説明 |
|------|------|
| **AgentMemory** | 各Agentが持つ独立したメモリ |
| **CentralMemory** | Mioが持つ全体統合メモリ |
| **会話ログ伝播** | Coder → Worker → Mio の階層的なログ伝播 |
| **日次カットオーバー** | 日次でSessionをリセットし、古いMemoryをアーカイブ |

### C.4 セキュリティ

| 用語 | 説明 |
|------|------|
| **公開鍵認証** | SSHの認証方式（パスワード認証は禁止） |
| **ホスト鍵検証** | `known_hosts` を使用したSSHホスト鍵検証 |
| **楽観的ロック** | ファイル編集時のETagチェック |
| **ワークスペース分離** | Agent別のワークスペースディレクトリ（実験的） |

### C.5 その他

| 用語 | 説明 |
|------|------|
| **v3.0** | Clean Architectureベースの単一プロセス実装 |
| **v4.0** | 分散実行対応版（Transport層追加） |
| **Phase 1-3** | v4.0の段階的実装フェーズ |
| **ブルーグリーンデプロイ** | v3.0/v4.0の並行稼働と段階的切り替え |
| **指数バックオフ** | 再接続時の待機時間を指数的に増加させる戦略 |
