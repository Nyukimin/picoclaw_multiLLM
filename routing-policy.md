# routing-policy.md

## このファイルの役割

このファイルは、RenCrow におけるルーティング判断の実務ルールを定義する。  
目的は、ユーザー入力を適切な役割と処理カテゴリへ振り分け、責務の混線を防ぐこと。

この文書は「どこへ渡すか」を決めるためのものであり、  
詳細実装や背景思想は `CLAUDE.md`、作業ルールは `Agents.md`、実装仕様は `docs/実装仕様_v3.md` を参照する。

---

## ルーティングの基本原則

ルーティングでは次を優先する。

1. 責務分離を守る
2. 安全側に倒す
3. 曖昧な場合は小さく始める
4. 実行より先に分類を誤らない
5. 迷ったら `CHAT` に戻す

ルーティングは「最も賢そうな LLM を使うこと」ではなく、  
「その仕事を正しい層に置くこと」を優先する。

---

## 役割の定義

### Chat
担当：
- ユーザー対話
- 意図の整理
- ルーティング判断
- 結果返却
- 軽い要約、確認、分類

やらないこと：
- 大きな実装
- 破壊的操作
- 実行の主体化
- 複雑なコード修正の抱え込み

### Worker
担当：
- 実行
- ファイル編集
- patch 適用
- コマンド実行
- テスト実行
- 実行結果の記録
- plan のステップ実施

やらないこと：
- 設計方針の独断決定
- 曖昧要件の解釈主導
- 長い仕様策定
- 実装案のゼロからの発明

### Coder
担当：
- 設計
- 分析
- 実装案作成
- `plan` 生成
- `patch` 生成
- リファクタリング案
- テスト追加案
- 原因調査

やらないこと：
- 原則として破壊的操作の直接実行
- 実行責務の肩代わり
- 運用判断の最終確定

---

## 処理カテゴリ

RenCrow では、入力を以下のカテゴリへ分類する。

- `CHAT`
- `PLAN`
- `ANALYZE`
- `OPS`
- `RESEARCH`
- `CODE`
- `CODE1`
- `CODE2`
- `CODE3`

---

## カテゴリ定義

### CHAT
対象：
- 雑談
- 方針相談
- 短い説明
- 軽い要約
- 判断保留が適切な曖昧入力
- どこへ回すべきか未確定の入力

担当：
- Chat

例：
- 「これってどう思う？」
- 「今の構成をざっくり説明して」
- 「まず何をすべき？」

### PLAN
対象：
- 作業計画の作成
- 手順分解
- 段取り整理
- 実装前の実行順序整理

担当：
- 原則 Coder
- 結果返却は Chat

例：
- 「この機能追加の実装手順を考えて」
- 「段階的な移行プランを作って」

### ANALYZE
対象：
- 原因調査
- ログ分析
- ボトルネック分析
- コード読解
- 差分比較
- 影響範囲の見積もり

担当：
- 原則 Coder
- 軽い読解だけなら Chat でも可

例：
- 「このエラーの原因を調べて」
- 「このモジュールの責務を分析して」

### OPS
対象：
- 実行系操作
- コマンド実行
- テスト実行
- ファイル変更適用
- 実運用タスク
- patch 適用
- ヘルスチェック

担当：
- Worker

例：
- 「テストを回して」
- 「この patch を適用して」
- 「Ollama の状態を確認して」

### RESEARCH
対象：
- 外部情報の調査
- 比較検討
- ドキュメント収集
- 仕様確認
- 周辺技術の探索

担当：
- 原則 Coder
- 結果返却は Chat

例：
- 「Codex CLI と Claude Code を比較して」
- 「このライブラリの使い方を調べて」

### CODE
対象：
- 汎用コーディング依頼
- まだ CODE1/CODE2/CODE3 の切り分けが不要な実装依頼

担当：
- Coder
- 必要に応じて内部で CODE1/CODE2/CODE3 相当へ振り分ける

例：
- 「この関数を直して」
- 「ユニットテストを書いて」

### CODE1
対象：
- 仕様整理寄り
- 骨格設計
- 初期案づくり
- ラフな plan 生成
- 発散的アイデア生成

担当：
- Coder1

例：
- 「実装前に設計案を複数出して」
- 「仕様のたたき台を作って」

### CODE2
対象：
- 実装寄りのコーディング
- 安定した修正
- テスト追加
- 既存コードへの自然な差分作成

担当：
- Coder2

例：
- 「この不具合を最小変更で直して」
- 「既存スタイルに合わせて実装して」

### CODE3
対象：
- 高難度の推論
- 複雑な設計判断
- 品質重視の修正
- 長文仕様読解を伴う実装案
- 微妙な責務分離の判断が必要なタスク

担当：
- Coder3

例：
- 「この設計の責務分離が妥当か見て」
- 「複数仕様を踏まえて安全な修正案を作って」

## ルーティング優先順位

入力のルーティングは次の順で決める。

### 1. 明示コマンド
最優先。

例：
- `/chat`
- `/plan`
- `/analyze`
- `/ops`
- `/research`
- `/code`
- `/code1`
- `/code2`
- `/code3`

明示指定がある場合、原則それに従う。  
ただし危険操作や権限制約がある場合は安全側で制御する。

### 2. ルール辞書
明確なキーワード、構文、依頼形式から判断する。

例：
- 「調べて」→ `RESEARCH`
- 「原因」→ `ANALYZE`
- 「実装して」→ `CODE`
- 「実行して」→ `OPS`

### 3. 分類器
辞書で決まらない場合、LLM ベース分類器で判定する。

分類器は次を返すことを目指す。
- category
- confidence
- reason
- ambiguity flags

### 4. 安全側フォールバック
最後まで曖昧なら `CHAT` に戻す。

これは誤実行防止のため。  
雑に `OPS` や `CODE` に流さないこと。

## ルーティング判断の細則

### 原則1: 実行語が入っていても、即 `OPS` にしない
例：
- 「この不具合を直して」
- 「テストが落ちる理由を見て」

これらはまず `ANALYZE` または `CODE` である可能性が高い。  
いきなり Worker 実行へ流さない。

### 原則2: 曖昧な設計相談は `CHAT` または `PLAN`
実装に見えても、まず整理が必要なら `PLAN` を優先する。

### 原則3: 複雑修正は Coder を先に通す
即実行より、`plan` / `patch` を生成してから Worker へ渡す。

### 原則4: 破壊的操作は Worker でも慎重に扱う
削除、依存変更、設定変更、CI 変更は高リスク。  
`OPS` であっても確認前提とする。

### 原則5: 調査と実装を混ぜない
「調査してから直す」は  
`ANALYZE` → `CODE` → `OPS`  
の順が基本。最初から全部混ぜない。

---

## 推奨ルーティングパターン

### 相談・雑談
入力  
→ `CHAT`

### 仕様整理
入力  
→ `PLAN`

### 原因調査
入力  
→ `ANALYZE`

### 外部調査
入力  
→ `RESEARCH`

### 実装案作成
入力  
→ `CODE1` または `CODE3`

### 安定実装
入力  
→ `CODE2`

### patch 適用・コマンド実行
入力  
→ `OPS`

---

## 複合タスクの扱い

複合タスクは一発で全部実行しない。  
段階分解する。

例：
「このバグの原因を調べて、必要なら直してテストも通して」

推奨分解：
1. `ANALYZE`
2. `CODE`
3. `OPS`

複合タスクを一つのカテゴリへ無理に押し込まないこと。

## Coder 系の使い分け

### Coder1
向いている：
- 初期案
- 発散
- 下書き
- 複数案比較
- 仕様のたたき台

### Coder2
向いている：
- 実装
- テスト作成
- 差分生成
- 手堅い修正
- 既存コードへの追従

### Coder3
向いている：
- 高品質な設計判断
- 複雑仕様読解
- 長文文脈下での修正方針
- 難しい責務分離
- 品質優先のコーディング

使い分けで迷う場合：
- 発散したい → Coder1
- 安定実装したい → Coder2
- 難問を丁寧に解きたい → Coder3

---

## Chat が抱え込んではいけないもの

Chat は入口である。  
そのため次を抱え込んではいけない。

- 複雑な設計判断の本体
- 大規模なコード生成
- patch 適用
- コマンド実行
- 実行結果の改ざん的要約

Chat は交通整理と返却に集中する。

---

## Worker に渡す前の条件

Worker に渡す前に、できるだけ次を満たす。

- 目的が明確
- 対象ファイルまたは対象操作が明確
- `plan` または `patch` がある
- 高リスク操作かどうかが分かっている
- 実行後に何を確認するかが分かっている

曖昧なまま Worker に投げない。

---

## ログに残すべき判断

ルーティング時は少なくとも次を残せる形にする。

- user_input summary
- selected category
- selected role
- confidence
- route reason
- fallback の有無
- job_id
- session_id

これにより、なぜその役割へ流したかを後から追えるようにする。

---

## 禁止事項

- なんとなく高性能そうだから CODE3 に寄せる
- 実行語だけで即 `OPS` にする
- 曖昧入力を強引に `CODE` に流す
- Coder に直接実行させる
- Chat に設計と実行を両方抱えさせる
- 調査・設計・実行を無分解で一括処理する

---

## 迷ったときのデフォルト

迷った場合は次の順で考える。

1. まず `CHAT`
2. 次に `PLAN` または `ANALYZE`
3. 実装が必要なら `CODE`
4. 実行段階で `OPS`

「まず考える、あとで動く」を優先する。
