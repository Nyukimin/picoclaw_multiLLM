## 1. 実装開始前の必須確認事項（CRITICAL）

### 1.1 go.mod モジュールパスの検証（最優先）

**絶対にやってはいけないこと**
- `go.mod` のモジュールパスを確認せずに実装を開始する
- fork 元や過去のモジュールパスをそのまま使う
- import を推測で書き始める

**必ず実行すること**

```bash
# 1. リポジトリ URL を確認
git remote -v

# 2. go.mod の module 行を確認
grep "^module " go.mod

# 3. ビルドが通ることを確認
go build ./...
````

**正しいモジュールパスの原則**

* `git remote -v` のリポジトリと `go.mod` の module が整合していること
* 不一致があれば、実装前に修正してから進めること

**これが最優先である理由**

* モジュールパスが誤ると import が全体的に壊れる
* 後から修正すると全ファイルへ波及する
* ビルド・テスト・LSP のすべてが不安定になる
* 小さな確認を飛ばしたせいで巨大な手戻りになる

### 1.2 実装開始前チェックリスト

以下を満たしてから実装を開始すること。

* [ ] `git remote -v` を確認した
* [ ] `go.mod` の module を確認した
* [ ] 不一致があれば先に修正した
* [ ] `go build ./...` が通ることを確認した
* [ ] 正本仕様 `docs/01_正本仕様/実装仕様.md` を確認した
* [ ] 対象責務が Chat / Worker / Coder のどれかを明確にした

## 2. プロジェクト概要

### 2.1 プロジェクト名

**RenCrow** (`picoclaw_multiLLM`)

### 2.2 目的

RenCrow は、LINE / Slack などからの指示を受け、複数の LLM を適切にルーティングして処理する超軽量 AI アシスタントです。

主な目標は以下です。

* 低スペック環境でも動作すること
* Chat / Worker / Coder の責務分離を守ること
* ルーティング・実行・記録を追跡可能にすること
* 自動化と安全性を両立すること

### 2.3 主要機能

* マルチ LLM ルーティング
* Worker による即時実行
* セーフガード付きの patch 適用
* Ollama 常駐管理とヘルスチェック
* セッション管理と日次カットオーバー
* 構造化ログとトレーサビリティ

### 2.4 使用技術

* 使用言語: Go 1.23
* 主対象プラットフォーム: Linux
* 開発環境: Linux / macOS
* ローカル LLM: Ollama
* 外部 LLM:

  * DeepSeek API（Coder1）
  * OpenAI API（Coder2）
  * Anthropic Claude API（Coder3）

## 4. 実行環境と制約

### 4.1 プラットフォーム

* 主要ターゲット: Linux
* 開発環境: Linux / macOS
* Windows は標準対象外とし、必要時のみ個別確認する

### 4.2 メモリ制約（最重要）

* 目標: Go プロセス単体でメモリ使用量 10MB 未満を目指す
* セッション・ログ・一時データを無制限に保持しない
* 大きな配列、マップ、キャッシュを安易に常駐させない
* 必要なものだけを短時間保持し、外部へ書き出す

### 4.3 Ollama 制約

* Ollama がローカル常駐していることを前提とする
* Chat / Worker モデルは `keep_alive: -1` を前提とする
* ヘルスチェックを起動時および実行前に行う
* MaxContext 8192 を超える前提で設計しない
* モデル要件と制約はコードと仕様の双方で確認する

### 4.4 API キー管理

API キーは必ず環境変数または安全なシークレット管理経由で扱う。

* `ANTHROPIC_API_KEY`
* `DEEPSEEK_API_KEY`
* `OPENAI_API_KEY`

**禁止**

* 設定ファイルへの平文保存
* サンプルコードへの直書き
* テストコードへの固定埋め込み

## 5. Go 実装ルール

### 5.1 エラーハンドリング

* すべてのエラーを適切に扱う
* `err != nil` を省略しない
* 文脈付きで包む
  例: `fmt.Errorf("load config: %w", err)`
* panic は原則避ける
* recover を使う場合も、隠蔽ではなく記録と復旧のために使う

### 5.2 並行処理

* goroutine は必要最小限にする
* 無制限起動を避ける
* `context.Context` によるキャンセル可能性を確保する
* channel close と待ち合わせの整合を守る
* `sync.Mutex` / `RWMutex` は unlock 漏れを防ぐ

### 5.3 命名規則

* パッケージ名: 小文字、単数形を基本とする
* 公開関数: PascalCase
* 非公開関数: camelCase
* 定数: PascalCase または UPPER_SNAKE_CASE
* 略語は必要最小限にし、意味が伝わることを優先する

### 5.4 実装スタイル

* 小さく読める関数を優先する
* 早すぎる抽象化を避ける
* その場しのぎの分岐追加でごまかさない
* 既存コードの責務と整合する場所へ書く
* 一見便利でも責務をまたぐ共通化は慎重に扱う

## 6. 新規ファイル作成と命名

### 6.1 基本方針

AI は実装中に必要と判断した場合、新規ファイルを作成してよい。
ただし、新規作成は常に正しいとは限らないため、まず既存ファイルへの追加・修正で済むかを確認すること。

新規作成前に必ず確認すること。

* 同等責務の既存ファイルがないか
* 既存ディレクトリ構造に自然に収まるか
* 新規ファイル化によって責務分離が改善されるか
* 名前を付ける理由を説明できるか

### 6.2 命名ルール

新規ファイル名は、既存の命名規則と責務に合わせて AI が決定してよい。
ただし、以下を守ること。

* 役割と責務が分かる名前にする
* 近傍ディレクトリの命名規則に合わせる
* Go / test / docs の文脈に合う名前にする
* 曖昧な思いつき名を使わない
* バージョン番号をファイル名に埋め込まない
* 一時的な作業ファイル名を恒久ファイルにしない

### 6.3 使ってはいけない名前の例

以下のような曖昧名は禁止。

* `new`
* `final`
* `tmp`
* `temp`
* `copy`
* `test2`
* `draft`
* `misc`

### 6.4 優先順位

新規ファイル名を決める際は、次の順で判断する。

1. 同ディレクトリの既存命名に合わせる
2. 同責務の近傍パッケージ命名に合わせる
3. Go の一般的慣習に合わせる
4. それでも迷う場合は、責務が最も明確に伝わる名前を採用する

### 6.5 作成時の注意

* package 名とファイル責務を一致させる
* テストが必要なら対応するテストファイルも検討する
* import パスと責務境界を壊さない
* 既存責務と重複するファイルを乱立させない

## 7. 責務の分離（Chat / Worker / Coder）

### 7.1 三役割モデル

| 役割     | 責務        | 主な内容                         |
| ------ | --------- | ---------------------------- |
| Chat   | 意思決定・対話管理 | ユーザー対話、ルーティング、結果返却           |
| Worker | 実行・道具係    | ファイル編集、コマンド実行、テスト実行、patch 適用 |
| Coder  | 設計・実装案作成  | 分析、plan 生成、patch 生成、実装案作成    |

### 7.2 不変ルール

1. **Coder は破壊的操作を直接実行しない**

   * Coder は `plan` と `patch` を生成する
   * 実適用は Worker が担当する

2. **Worker は実行主体である**

   * ファイル書き込み
   * コマンド実行
   * テスト実行
   * patch 適用
   * 実行結果記録

3. **Chat は交通整理と返却に集中する**

   * 実行主体にならない
   * 複雑な実装本体を抱え込まない
   * 調査・設計・実行を一人でまとめて処理しない

### 7.3 実装時の判断基準

変更を書く前に必ず判断すること。

* これは Chat の責務か
* これは Worker の責務か
* これは Coder の責務か
* この変更は責務境界を越えていないか

責務の違う層へロジックを混ぜないこと。

ルーティングの詳細は `routing-policy.md` を正本とする。
このファイルでは実装上必要な最小限のみ扱う。

### 8.1 ルーティングカテゴリ

* `CHAT`
* `PLAN`
* `ANALYZE`
* `OPS`
* `RESEARCH`
* `CODE`
* `CODE1`
* `CODE2`
* `CODE3`

### 8.2 優先順位

1. 明示コマンド
2. ルール辞書
3. 分類器
4. 安全側フォールバック

### 8.3 安全側フォールバック

曖昧な入力や危険な入力は、安易に `OPS` や `CODE` に流さず、まず `CHAT` または `PLAN` / `ANALYZE` 側で整理する。

### 8.4 CODE 系の使い分け

* `CODE1`: 仕様整理、初期案、発散
* `CODE2`: 手堅い実装、差分生成、テスト追加
* `CODE3`: 高難度推論、複雑な仕様読解、品質重視

## 9. セーフガードと高リスク操作

### 9.1 高リスク操作

以下は高リスク操作として扱う。

* 依存関係の追加・更新
* ファイル削除
* CI / build / deploy 設定変更
* 大規模横断修正
* セーフガード無効化
* シークレット取り扱い変更

### 9.2 ルール

* 高リスク操作は独断で進めない
* 実行前に内容を要約できること
* 実行後に結果と影響範囲を記録すること
* 可能ならロールバック可能性を確保すること

### 9.3 必須セーフガード

以下は常時有効を前提とする。

* Git auto-commit
* 保護ファイルパターン
* 実行前サマリ
* 実行ログ記録

## 10. 禁止事項

### 10.1 実装レベル

* 仕様を読まずに実装する
* コードの現状挙動を確認せずに修正する
* Coder が破壊的操作を直接実行する
* 実行語だけで即 `OPS` に流す
* 既存の問題あるパターンを踏襲して拡張する
* 同じアプローチで 3 回以上だらだら試す
* 問題のある箇所だけ直して、同原則の他箇所を放置する
* 一時ファイルを恒久ファイルとして残す
* `LINT` や `go test` を通さずに完了扱いする

### 10.2 運用レベル

* API キーを平文保存する
* MaxContext 制約を無視する
* `keep_alive: -1` 前提を崩す
* Worker 実行ログを残さない
* 仕様と実装の不整合を黙って放置する
* archive / 履歴文書を正本扱いする

### 10.3 E2E テスト

* 本番と異なる設定経路を使う
* `.env` / 環境変数 / `config.yaml` の流れを無視する
* テスト専用の別世界を作って安心しない

## 11. セッション管理

### 11.1 基本方針

* `session_id` で追跡する
* セッションごとに必要最小限の状態だけを持つ
* 長期記憶は外部へ逃がす
* 日次カットオーバーで整理する

### 11.2 日次カットオーバー

* 基準時刻: JST 00:00
* 目的:

  * メモリリセット
  * ログ整理
  * セッションの区切り明確化

### 11.3 メモリ管理

* `recent_turns` は最小限
* `short_memory` 中心で運用
* 長期保持が必要な情報は Obsidian など外部へ書き出す

## 14. 開発フロー

### 14.1 基本サイクル

すべての変更は、原則として以下で進める。

1. 仕様確認
2. 現状把握
3. テスト設計
4. 最小実装
5. 検証
6. リファクタ
7. 記録

### 14.2 TDD

新機能追加・バグ修正は TDD を原則とする。

#### Red

* 失敗するテストを書く
* 再現できることを確認する

#### Green

* テストを通す最小実装を行う

#### Refactor

* 読みやすさと責務の整合を改善する
* テストが通り続けることを確認する

### 14.3 問題解決姿勢

* 2 回試して解けなければ、原因分析に戻る
* 同じ手法の微修正を繰り返さない
* 既存の誤った前提を疑う
* 場当たり修正より、責務と根本原因を優先する
